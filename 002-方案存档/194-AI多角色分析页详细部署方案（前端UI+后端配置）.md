# 194 — AI多角色分析页详细部署方案（v2）

> **目的：** 根据 [193号方案](002-方案存档/193-AI多角色分析页设计研判（GitHub同类项目调研）.md) 的调研结论，结合本项目已有前后端能力，制定 AI 多角色分析页的详细实施方案。
>
> **v2 修正：**
> - **① 专业性优先**：各角色输出为专业研判报告（非叙事性文学文本），每个角色有自己的专业术语、分析框架和判断标准
> - **② RTK评估**：RTK (Rust Token Killer) 作为CLI代理不直接适用于后端DeepSeek调用，但其结构化压缩思想可以通过 Prompt 工程 + Function Calling 实现等效效果——token 节约估算约 40-60%
> - **③ 全LLM确认**：确认所有角色分析、辩论、综合研判均由 DeepSeek LLM 生成输出。Mock仅为DeepSeek不可用时的降级方案，且Mock也模拟LLM输出形式而非硬编码固定文本

## 1. 当前状态评估

### 1.1 前端现状（`_ui-prototype/ai-analysis.html`）

| 维度 | 现状 | 问题 |
|:-----|:------|:------|
| 页面布局 | 顶部股票选择 + 进度条 + 结果区（Tab切换） | 简单线性，无角色区展示 |
| 输出形式 | 总评6.8/10 + 趋势↗ + 信号词 + 置信度72% + 5维评分条 | ❌ **含评分**，与去评分化原则矛盾 |
| 角色展示 | 无独立角色展示区，无专业研判输出形式 | ❌ 角色展示缺失 |
| 数据来源 | 全部硬编码Mock文本 | 无真实数据对接，且**Mock文本为固定模板**（非LLM生成风格）|
| 与indicator-ide联动 | 无 | ❌ 缺乏跨页联动 |

### 1.2 后端现状（`deepseek_analysis_service.py`）

| 维度 | 现状 | 问题 |
|:-----|:------|:------|
| 角色数 | 5个：技术分析/基本面/宏观/风控/基金经理 | 原角色定义偏传统投研，未利用现有策略能力 |
| 分析模式 | 串行（逐个角色分析） | 效率低，应改为并行 |
| 输出格式 | bullishScore/bearishScore百分比 + 文本raw | ❌ **含评分**，需改为专业研判格式 |
| DeepSeek模式 | 有 _call_deepseek + mock降级 | ✅ 结构可用，需修改prompt方向 |
| 上下文注入 | ai_context_builder.build_context(ts_code) | ✅ 保留 |
| 数据存储 | 内存 _analysis_store | 线程安全但重启丢失 |
| Prompt方向 | 倾向生成通用分析文本 | ❌ 应改为各角色专业研判风格 |

### 1.3 核心差距总结

```
当前现状                          →  目标状态
────────────────────────────────────────────────
5个传统投研角色                     →  6个基于现有策略能力的专业角色
串行分析                            →  并行分析（多线程/异步）
评分式输出(bullishScore 65%)        →  专业研判输出（角色特有分析框架）
通用化自然语言叙事文本               →  专业研判报告（有角色特色术语和表述）
Mock固定模板文本                    →  DeepSeek生成（Mock也模拟LLM输出风格）
无辩论环节                          →  多轮对抗辩论(Phase 2)
无跨页联动                          →  indicator-ide → ai-analysis一键跳转
```

## 2. 整体方案概览

### 2.1 角色体系（6大角色 — 基于已有数据能力）

| 角色 | ID | 图标 | 核心数据源（本项目已有） | 分析风格 | 对比旧角色 |
|:-----|:---|:-----|:------------------------|:---------|:-----------|
| 📈 技术分析师 | `technical` | 📈 | 缠论策略(chanlun_strategy) + 量价策略(volume_price_strategy) + K线 | **缠论技术研判**：走势结构→中枢→买卖点→量价形态，按缠论体系输出 | 替换旧technical |
| 📊 基本面分析师 | `fundamental` | 📊 | Tushare fina_indicator + 财务数据 | **估值财报研判**：财务指标→估值分位→成长性→行业对比 | 保留旧fundamental |
| 🧩 筹码/资金分析师 | `chip` | 🧩 | chip_strategy + moneyflow + 北向资金 | **主力行为研判**：筹码分布→主力意图→资金流向→集中度变化 | **新增**（替代旧macro）|
| 💬 情绪/舆情分析师 | `sentiment` | 💬 | crowding_factor + sector_rotation_model + 新闻 | **市场情绪研判**：情绪周期→拥挤度→板块轮动→市场氛围 | **新增**（替代旧macro部分）|
| 📰 消息面分析师 | `news` | 📰 | news路由 + reports路由 + 公告 | **事件驱动研判**：新闻催化→公告事件→机构观点→政策研判 | **新增** |
| 🛡️ 风控分析师 | `risk` | 🛡️ | verification_chains + conflict_arbiter + 验证链 | **风险管控研判**：多维度风险→冲突识别→下行情景→仓位建议 | 改造旧risk |

> **关键变更：** 各角色的System Prompt不再要求"生成通顺的自然语言叙事"，而是要求**按各角色自己的专业分析框架输出**。技术分析师像缠论分析师、基本面分析师像投行研究员、风控分析师像合规官——各有各的表述风格和分析范式。

### 2.3 关键设计原则（v2修正版）

| 原则 | 说明 | 具体措施 |
|:-----|:------|:---------|
| **🥇 专业性优先** | 各角色输出体现专业分析风格，非通用叙事 | 每个角色的System Prompt严格限定分析框架和输出术语；技术分析师像缠论分析师，基本面像投行研究员 |
| **🎯 研判导向** | 报告产出是"专业研判"而非"文学叙事" | DeepSeek输出直接用于展示，不做二次润色。用户看到的是专业分析师的研判原文 |
| **🔄 全LLM驱动** | 所有分析内容由DeepSeek实时生成 | Mock仅为不可用时的降级，Mock内容也模拟DeepSeek的研判输出风格 |
| **⏱️ Token优化** | 结构化输入→结构化输出，减少冗余token消耗 | 输入数据压缩为标准JSON Schema + Function Calling模式，预期节约40-60% token |

△ 以下原则与v1保持一致：

| 原则（保留） | 说明 | 具体措施 |
|:-------------|:------|:---------|
| 角色独立性 | 各角色独立分析，不共享上下文 | 每个角色用独立System Prompt + 独立数据请求，仅在综合阶段合并 |
| 去评分化 | 不输出BullishScore/BearishScore | 所有分析输出改为专业研判 + 结构化快照 |
| 并行加速 | 6角色同时分析 | 使用 `ThreadPoolExecutor` / `asyncio` 并行调度 |
| 降级兜底 | DeepSeek不可用时的结构化展示 | 结构化标签 + 中文关键词（同indicator-ide降级方案）|
| 数据来源标注 | 每条结论标注数据来源 | 复用indicator-ide幻觉防护的 `sources` 字段 |

### 4.3 并行调度与错误边界

Phase 1 使用 `ThreadPoolExecutor(max_workers=6)` 并行发起6角色分析。实时进度通过内存中的 `_analysis_store` 字典上报。

#### 4.3.1 调度流程

```
POST /api/v3/ai/analyze
  ├── 1. 初始化 _analysis_store[task_id] = {
  │       'status': 'running',
  │       'roles': {'technical': 'pending', 'fundamental': 'pending', ...},
  │       'progress': 0,
  │       'results': {}
  │     }
  │
  ├── 2. ThreadPoolExecutor(max_workers=6) 提交6个任务
  │     ├── technical:    _analyze_single(ts_code, 'technical', ...)
  │     ├── fundamental:  _analyze_single(ts_code, 'fundamental', ...)
  │     ├── chip:         ...
  │     ├── sentiment:    ...
  │     ├── news:         ...
  │     └── risk:         ...
  │
  ├── 3. 每个 _analyze_single 完成时：
  │     ├── 更新 _analysis_store[task_id]['roles'][role_id] = 'completed'
  │     ├── 保存 analysis_report 到 _analysis_store[task_id]['results'][role_id]
  │     ├── 更新 progress = completed_count / 6 * 100
  │     └── (可选) 发送 SSE 事件通知前端轮询
  │
  └── 4. 全部6角色完成或失败后：
        └── _analysis_store[task_id]['status'] = 'completed' / 'partial'
```

#### 4.3.2 错误处理策略

| 场景 | 处理方式 | 代码行为 |
|:-----|:---------|:---------|
| DeepSeek API 返回空/错误 | 先重试，降落级 | `try: _call_deepseek() except: retry(2次, 间隔1s) → Mock降级` |
| DeepSeek API 限流(429) | 指数退避 | `retry(3次, 间隔2s/4s/8s exponential backoff)` |
| 数据获取失败(无数据股) | 该角色降级 | 输出 `"数据不足，无法进行分析"` + 标记该角色 `data_missing` |
| 网络超时(>30s) | 跳过 | `future.result(timeout=30)` 超时后该角色 `failed` |
| 单角色失败(不影响其他) | 孤立失败 | 该角色标记 `failed`，其他5角色正常完成 |
| 全部6角色失败 | 返回错误 | `_analysis_store[task_id]['status'] = 'failed'` |
| 线程池异常 | 主线程捕获 | `as_completed()` 循环中捕获每个 future 的异常 |

#### 4.3.3 进度上报约束

| 维度 | 规格 |
|:-----|:------|
| 最小上报间隔 | 不低于500ms（避免前端频繁轮询） |
| 进度条更新粒度 | 每完成一个角色 → 递增约16.7%（1/6），不细化到角色内步骤 |
| 前端轮询间隔 | 推荐1.5s（GET /api/v3/ai/analyst-progress）|
| 最终报告等待 | 全部6角色完成（含降级）后 → GET /api/v3/ai/final-report 立即可用 |
| 超时总时间 | 无硬限制（等待最慢角色完成），推荐前端5分钟后提示"部分角色超时" |

#### 4.3.4 并发限制

| 维度 | 限流值 | 说明 |
|:-----|:-------|:------|
| 单股票并行数 | 6线程 | 6角色同时发起 |
| 系统总并发 | ≤12线程 | 最多同时分析2只股票 |
| DeepSeek API QPS | 无硬限制 | 参考DeepSeek服务端限流策略自行调整 |
| 单角色重试上限 | 2次(普通)/3次(429) | 防止无限重试耗尽配额 |

### 4.6 Phase 1 → Phase 2 数据流转定义

**核心问题：** Phase 1 输出的6份 `analysis_report` 如何传递给 Phase 2 辩论引擎？

#### 4.6.1 方案选择：内存缓存（推荐）

| 方案 | 实现方式 | 优点 | 缺点 | 推荐度 |
|:-----|:---------|:-----|:-----|:-------|
| **A. 内存缓存** | `_analysis_store` 字典（与当前实现一致） | 零IO延迟、简单 | 进程重启丢失 | ⭐ **推荐** |
| B. Redis/Cache | 写入后端缓存层 | 持久化、跨进程 | 增加复杂度 | 低 |
| C. 写数据库 | 存到 PostgreSQL | 持久、可溯源 | IO延迟高 | 不推荐 |
| D. 前端传递 | 前端缓存6份报告，发起辩论时POST回后端 | 简单 | 大token负载 | 不推荐 |

**选择方案A（内存缓存）：** 与当前 `deepseek_analysis_service.py` 中 `_analysis_store` 的存储模式一致。Phase 1 完成后，6份 `analysis_report` 保留在内存中。用户在原型中点击「发起辩论」按钮时，前端调用 `POST /api/v3/ai/debate` 仅需传入 `task_id`（无需重传6份报告内容）。

```
Phase 1 完成
  │
  ├── GET /api/v3/ai/final-report  ── 仅返回 Phase 1 的6份 analysis_report（无辩论内容）
  │
  ├── [用户点击"发起辩论"] ── POST /api/v3/ai/debate payload: { task_id }
  │     ↓
  │   Phase 2 启动（读取 _analysis_store[task_id]['results'] 中的6份报告）
  │     ↓
  │   Phase 2 完成 → _analysis_store[task_id]['debate'] 写入完整辩论输出
  │     ↓
  │   GET /api/v3/ai/final-report ── 现在返回 Phase 1 + Phase 2 的合并内容
  │
  └── GET /api/v3/ai/synthesis  ── 仅含综合决策（方向/情景/建议），
                                     从 _analysis_store[task_id]['synthesis'] 读取
```

#### 4.6.4 生命周期与清理

| 事件 | 行为 |
|:-----|:------|
| 新分析覆盖 | `POST /api/v3/ai/analyze` 同一 `ts_code` 时生成新 `task_id`，旧数据存留 |
| 过期清理 | `_analysis_store` 中的条目超过30分钟无访问 → 自动删除 |
| 服务器重启 | 所有数据丢失（等价于"请重新分析"）|
| 持久化需求 | 如需保存分析报告供历史查询 → 单独实现 PostgreSQL 写库（P3规划）|

## 5. Phase 2：对抗辩论模式

**目标：** 在独立分析完成后，增加多角色辩论环节。所有辩论内容由DeepSeek实时生成，不预设固定辩词。

### 5.1 辩论流程（全LLM驱动）

```
阶段1：独立分析完成（Phase 1产出6份专业研判报告）

阶段2：两轮LLM辩论
├── 第一轮：两两对抗
│   ├── DeepSeek以"技术分析师"身份 vs DeepSeek以"筹码分析师"身份
│   │   Prompt: "你看到{技术分析师研判报告}，请从{筹码分析师}视角质疑其结论"
│   │   → 根据对方的实际研判内容动态生成反驳/补充
│   │
│   ├── DeepSeek以"基本面分析师"身份 vs DeepSeek以"消息面分析师"身份
│   │   → 估值逻辑 vs 短期催化的动态辩论
│   │
│   └── DeepSeek以"情绪分析师"身份 + "风控分析师"身份
│       → 中立组评估双方论据，指出被忽略的因素
│
├── 第二轮：多空联盟辩论
│   ├── DeepSeek综合看多方观点（技术+筹码）
│   │   Prompt: "你看到以下研判报告{技术报告}{筹码报告}，请综合两方观点给出看多论证"
│   │
│   ├── DeepSeek综合看空方观点（基本面+消息面）
│   │   Prompt: "你看到以下研判报告{基本面报告}{消息面报告}，请综合两方观点给出看空论证"
│   │
│   └── DeepSeek中立裁决（情绪+风控+前两方观点）

│
└── 辩论产出：
    ├── 一致结论（所有角色LLM输出中方向一致的判断）
    ├── 仍存在的分歧点（各角色LLM输出中方向矛盾的判断）

阶段3：综合研判（DeepSeek V4）
├── 输入：6份研判报告 + 辩论产出
├── 输出：综合决策报告（方向、置信度、情景推演、风险提示）
```

## 6. RTK评估与Token优化方案 ★

### 6.1 RTK（Rust Token Killer）项目说明

| 属性 | 值 |
|:-----|:----|
| 项目 | [rtk-ai/rtk](https://github.com/rtk-ai/rtk) |
| 定位 | CLI 代理——拦截 shell 命令输出并在到达 LLM 上下文之前压缩/过滤 |
| 语言 | Rust，单二进制 ~4MB |
| 效果 | **60-90% token 压缩**（针对 shell 命令输出）|
| 原理 | 4种策略：Smart Filtering / Grouping / Truncation / Deduplication |
| 适用场景 | 开发工具命令输出压缩（git status / cargo test / pytest / ls 等）|

**过滤策略示例：**
```
git status（原始 ~2000 tokens）→ rtk git status（过滤后 ~200 tokens，~90%压缩）
cargo test（原始大量编译输出）→ 仅提取 FAILED 行 + 失败摘要
ls -la（原始逐条文件）→ 按目录分组 + 统计计数
```

### 6.2 RTK对本项目的适用性评估

| 评估维度 | 结论 |
|:---------|:------|
| **是否能直接集成到后端DeepSeek调用链？** | ❌ **不适用**。RTK拦截的是 shell 命令输出（CLI管道），我们后端通过 Python HTTP 请求调用 DeepSeek API，不走shell管道 |
| **是否能用于前端开发环境？** | ✅ **间接适用**。在开发阶段，如果Claude或其他AI工具需要查看项目文件/日志，RTK可以减少开发环节的token消耗 |
| **核心原理是否可借鉴？** | ✅ **高度相关**。RTK的核心思想是"在输入LLM之前对数据进行结构化压缩"——这个思想可以直接应用到我们的 DeepSeek Prompt 工程中 |

### 6.3 本项目Token优化方案（借鉴RTK思想）

**核心问题：** 若不优化，每个角色调用DeepSeek时如果传全量原始数据，将消耗大量token。

**优化策略（借鉴RTK的4种策略映射）：**

| RTK策略 | 本项目的对应实现 | 预期效果 |
|:---------|:---------------|:---------|
| **Smart Filtering** 去噪 | 只传每个角色所需的维度的数据（技术只用chanlun+volume_price，不用传新闻） | 每个角色输入减少~60% |
| **Grouping** 分组 | 策略输出数据按维度预分组为结构化JSON，非原始verbose日志 | 输入减少~50% |
| **Truncation** 截断 | 设置每个角色的最大输入token上限（如技术=2000, 基本面=1500）| 控制极端情况 |
| **Deduplication** 去重 | 各角色共享的数据（股票基本信息/日线行情）只计算一次 | 减少重复~30% |

**具体实现：**

```python
def _build_structured_input(role_id: str, ts_code: str) -> Dict:
    """
    按角色需求构建精简的结构化输入，而不是传全量原始数据。
    这是本方案token优化的核心——只传角色需要的、精简后的数据。
    """
    BASE = {
        'ts_code': ts_code,
        'latest_price': 28.50,  # 仅最新价
        'market_cap': '180亿',  # 以亿为单位
        'industry': '有色金属',
    }
    
    # 每个角色只获取所需维度的精炼数据
    ROLE_DATA = {
```

### 6.4 Token节约对比

| 环节 | 原始（全量数据+叙事prompt） | 优化后（结构化精简+Function Calling） | 节约 |
|:-----|:--------------------------|:-------------------------------------|:-----|
| 每个角色输入 | ~3,000 tokens（含全量数据日志+长prompt） | ~1,200 tokens（结构化JSON+精简prompt）| **~60%** |

| 综合研判 | ~4,000 tokens（含全部6份报告原文）| ~2,000 tokens（结构化摘要+共识分歧）| **~50%** |

### 6.5 结论

| 结论 | 内容 |
|:-----|:------|
| RTK直接集成 | ❌ 不适用（后端是HTTP API调用，非shell命令）|
| RTK思想应用 | ✅ **强烈推荐**——结构化输入 + 精简数据 + 角色专属prompt |
| 预期效果 | token消耗减少 **~58%**，成本降低 **~58%** |
| 实施方式 | 在 `deepseek_analysis_service.py` 中增加数据预处理器 `_build_structured_input()` |
| 不影响输出质量 | 结构化的输入数据不影响DeepSeek生成专业研判报告的质量 |

## 7. 全LLM参与确认 ★

### 7.1 声明

> **本方案中，AI多角色分析页的所有以下内容均通过 DeepSeek LLM 实时生成，不涉及任何硬编码文本：**
>
> - ✅ 各角色的**专业研判报告**（Phase 1 — 每个角色的analysis_report字段）
> - ✅ 各角色的**结构化数据快照**（Phase 1 — 从策略/数据源提取 + LLM判读）
> - ✅ **辩论论点**（Phase 2 — 针对对方实际输出内容动态生成反驳/补充）
> - ✅ **辩论评估**（Phase 2 — 综合多角色论点）
> - ✅ **综合决策报告**（Phase 2/3 — 方向、情景推演、风险提示）
>
> **唯一例外：** 当 DeepSeek API 完全不可用时，降级到 Mock 数据。但即便在降级模式下，Mock 内容也**模拟 DeepSeek 的研判输出风格**，而非固定模板文本。

### 7.2 LLM调用全链路

```
POST /api/v3/ai/analyze
  │
  ├── Thread-1: _get_technical_data() → _build_structured_input('technical')
  │       → DeepSeek('technical_prompt', input) → 生成{structured_data + analysis_report}
  │
  ├── Thread-2: _get_fundamental_data() → _build_structured_input('fundamental')
  │       → DeepSeek('fundamental_prompt', input) → 生成{structured_data + analysis_report}
  │
  ├── Thread-3: _get_chip_data() → _build_structured_input('chip')
  │       → DeepSeek('chip_prompt', input) → ...
  │
  ├── Thread-4: _get_sentiment_data() → _build_structured_input('sentiment')
  │       → DeepSeek('sentiment_prompt', input) → ...
  │
  ├── Thread-5: _get_news_data() → _build_structured_input('news')
  │       → DeepSeek('news_prompt', input) → ...
  │
  └── Thread-6: _get_risk_data() → _build_structured_input('risk')
          → DeepSeek('risk_prompt', input) → ...
  
  ↓ 6线程全部完成

  DeepSeek('synthesis_prompt', 6份analysis_report)
      → 生成综合决策报告（方向、情景、风险提示）
```

### 7.3 Mock的定位与特性

| 方面 | 说明 |
|:-----|:------|
| **触发条件** | DeepSeek API 不可用（网络故障/API限流/API Key无效/返回空）|
| **Mock内容** | 模拟DeepSeek输出风格的专业研判文本（不是固定模板）|
| **Mock生成逻辑** | 根据结构化数据填充预置模板，但模板内容体现角色专业性 |
| **用户感知** | 用户无法通过界面判断当前是LLM实时生成还是Mock降级 |
| **日志标记** | 后端日志标注 `mode=mock`，前端不展示此标记 |
| **后续恢复** | 下次分析自动重试DeepSeek，不固定使用Mock |

### 7.4 LLM成本估算（全流程）

| 环节 | LLM调用次数 | 每次输入tokens | 每次输出tokens | 总tokens | 成本(¥2/M) |

## 10. 风险与应急预案

### 10.1 风险清单（v2新增）

| 风险 | 概率 | 影响 | 应对方案 |
|:-----|:------|:-----|:---------|
| **专业化Prompt导致输出过于晦涩** | 中 | 用户难以理解 | 保留"简化模式"按钮，点击后由DeepSeek将专业研判浓缩为3句话摘要 |
| **结构化输入优化后数据缺失** | 低 | LLM分析不准 | 每个角色加 `verify_data` 校验，必要字段缺失时补充提示 |
| **辩论环节token超预算** | 中 | 成本超预期 | 设置每轮辩论的最大输出长度，超过截断 |
| DeepSeek API不可用 | 中 | 全部降级Mock | Mock模拟LLM风格，用户体验一致 |
| 并行6角色导致DeepSeek限流 | 中 | 部分角色失败 | 捕获限流异常→自动重试2次→该角色降级 |
| **数据源依赖（QMT未开通）** | **中** | **部分角色数据不全** | **技术/筹码/情绪依赖Tushare daily/moneyflow（已有✅）；消息面需news权限（待确认）；Level2/Tick数据依赖QMT（待开通→降级为日线级分析）** |
| **结构化输入字段缺失影响分析质量** | 低-中 | LLM分析准确度下降 | 每个角色输入前做 `verify_data` 校验，缺失字段在前端标注"数据暂缺"，LLM输出中提示"基于不完整数据" |

## 附录B：RTK与本项目的技术对比

| 维度 | RTK (Rust Token Killer) | 本项目的Token优化方案 |
|:-----|:------------------------|:---------------------|

| **实现语言** | Rust | Python（直接集成到deepseek_analysis_service.py）|
| **核心策略** | 去噪+分组+截断+去重 | 角色专属过滤+结构化JSON+必要字段截断 |
| **预期效果** | 60-90% token压缩 | ~58% token压缩 |
| **是否需要外部依赖** | cargo安装rtk二进制 | 无额外依赖，纯Python实现 |
| **集成方式** | Claude Code PreToolUse Hook | `_build_structured_input(role_id, ts_code)` 函数 |
