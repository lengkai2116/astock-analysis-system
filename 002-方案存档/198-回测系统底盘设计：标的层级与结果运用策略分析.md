# 回测系统底盘设计：标的层级与结果运用策略分析

> **⚠️ 核心前提**：本系统最终用户是**非专业人士**（非量化分析师、非投资经理）。本报告的每一项设计评估都以"普通人能否理解并操作"为第一约束。

---

## 问题一：回测标的的层级选择

### 1.1 系统策略架构的特殊性

当前系统不是"一个策略打天下"，而是多层策略组合架构：

```
┌─────────────────────────────────────────────────────────────────┐
│  策略层（5+7个独立策略）                                         │
│  ├─ 5大策略：缠论 / 量价形态 / 筹码 / 多因子 / BOCIASI           │
│  ├─ 7独立策略（#196新增）：情绪周期 / 主力行为 / 趋势通道 /       │
│  │  涨停板 / 风控 / 波浪 / 斐波那契                              │
│  └─ 各策略独立产出信号 + 置信度                                  │
├─────────────────────────────────────────────────────────────────┤
│  融合层                                                          │
│  ├─ UPF（四层加权）：结构30% + 形态25% + 情绪20% + 因子25%       │
│  ├─ ConflictArbiter（四级裁定）：缠论>量价>筹码>因子>情绪         │
│  └─ 输出：融合信号 + 置信度                                       │
├─────────────────────────────────────────────────────────────────┤
│  解读层                                                          │
│  ├─ 七步验证链（#65）：4条交叉验证链 → 三级仲裁模式               │
│  ├─ UIDF（#39）：UPF(60%) + AI(40%) 融合                        │
│  ├─ DeepSeek V4 叙事引擎 -> 情景化推演（A/B/C三场景）            │
│  └─ 输出：中文标签 + 情景描述，**无评分、无强制买卖建议**          │
└─────────────────────────────────────────────────────────────────┘
```

关键矛盾：**回测引擎测试的是 BUY/SELL 信号的交易绩效，但系统的最终输出是现状识别+情景描述，不是交易信号**。

### 1.2 三种回测标的层级选项

#### 选项A：仅回测单一策略（量价单独/缠论单独……）

| 维度 | 评估 |
|------|------|
| ✅ 优势 | 信号纯净，能精确判断每个策略的独立价值；问题定位精准（"量价形态不行"→知道改量价） |
| ❌ 劣势 | 与系统实际使用方式脱节——用户看到的是组合输出，不是单一策略；实践中没有哪个策略会单独使用 |
| 🎯 适用场景 | 选股系统的因子筛选阶段、策略开发阶段的初步验证 |
| 👤 对非专业用户 | **最不可理解的方式**——"我是来看这只股票行不行的，你给我看缠论的夏普比率？" |

#### 选项B：回测组合策略（UPF/ConflictArbiter融合后的信号）

| 维度 | 评估 |
|------|------|
| ✅ 优势 | 模拟真实使用环境，能回答用户真正关心的问题："这套分析体系能不能帮我赚钱" |
| ❌ 劣势 | 问题定位困难——回测亏了，是缠论的问题还是量价的问题？是权重不对还是融合逻辑有问题？ |
| 🎯 适用场景 | 策略上线前最终验证、整体绩效评估 |
| 👤 对非专业用户 | **最直观的方式**——"系统说买就买，说卖就卖，看看这样操作的成绩单" |

#### 选项C：层级化回测（先逐层诊断，再组合验证）【推荐】

```
第一层：单一策略回测（诊断）
├─ 量价形态 → 独立回测结果
├─ 缠论 → 独立回测结果
├─ 筹码 → 独立回测结果
├─ 多因子 → 独立回测结果
└─ BOCIASI → 独立回测结果

第二层：消融回测（定位贡献）
├─ 全策略组合（5个全上）
├─ 去掉量价（其他4个）
├─ 去掉缠论（其他4个）
├─ 去掉筹码（其他4个）
└─ 每去掉一个 vs 全组合的绩效差异 = 该策略的边际贡献

第三层：组合策略回测（最终验证）
└─ UPF融合信号 → 最终绩效
```

| 维度 | 评估 |
|------|------|
| ✅ 优势 | 既能回答"总体好不好"，也能回答"哪个拖后腿" |
| ⚠️ 复杂度 | 回测工作量×3，对非专业用户需要设计清晰的可视化 |
| 👤 对非专业用户 | **需要中间层**：普通人只需要看"总体好不好"和"哪个最弱"，不需要看每个策略的具体指标值 |
| 🎯 推荐理由 | 唯一能同时满足"诊断需求"和"评估需求"的方案 |

### 1.3 推荐的用户视角

对非专业用户不展示"单一策略的夏普比率是多少"，而是展示：

```
┌─────────────────────────────────────────────┐
│  📊 策略健康度概览                            │
├─────────────────────────────────────────────┤
│  🟢 量价形态  表现正常  ← 对组合有正向贡献     │
│  🟡 缠论      表现偏弱  ← 近3个月贡献下降     │
│  🔴 筹码      表现异常  ← 需要关注            │
│  🟢 多因子    表现正常                         │
│  🟢 BOCIASI  表现正常                         │
├─────────────────────────────────────────────┤
│  组合总体：年化+8.2%，最大回撤-11.5%          │
│  💡 建议：尝试暂缓筹码策略权重，观察2个月      │
└─────────────────────────────────────────────┘
```

**核心设计原则**：以"策略健康度"替代"策略指标值"，以"贡献度"替代"夏普比率"，以"关注/正常/异常"替代具体数字。

### 1.4 单策略 vs 组合回测的差异影响分析

| 维度 | 单策略回测 | 组合回测 | 差异影响 |
|------|-----------|---------|---------|
| **交易频率** | 各策略独立信号频率不同（缠论低频、量价中频） | 融合后频率取交集或加权 | 组合后交易频率可能显著下降 |
| **胜率** | 各策略单独胜率差异大 | 融合后胜率趋向均值回归 | 单策略高胜率≠融合后高胜率 |
| **最大回撤** | 单策略回撤可控 | 组合可能因信号冲突导致回撤放大或缩小 | 组合可能降低回撤（分散化）也可能放大（错误时机） |
| **信号冲突成本** | 无（单一信号主导） | 融合层仲裁可能导致错过最佳买卖点 | 这是组合策略最大的隐含成本 |
| **参数敏感性** | 单个策略参数敏感 | 融合后参数敏感性降低（分散效应） | 组合策略通常更稳健 |
| **非专业用户理解难度** | 低级：难以理解"为什么要单独测试这个" | 中级：能理解"这套系统整体表现" | 组合更友好 |
| **调试价值** | 高：明确知道哪个策略有问题 | 低：需要消融分析才能定位 | 单策略对开发者更有用 |

## 问题二：回测结果的运用策略

### 2.1 核心困境

当前系统的输出范式（180号 §7 重设计）与传统回测存在**结构性矛盾**：

```
传统范式：策略信号 → 回测 → 绩效指标 → "好"或"不好" → 调整/部署
            ↑ 交易信号可验证
            ↑ 指标可量化为"好策略的标准"

本系统范式：策略组合 → 现状识别 → DeepSeek叙事 → 用户自行决策
            ↑ 输出的是描述，不是指令
            ↑ "好的现状描述"没有客观量化标准
```

**核心问题**：交易信号可以回测（买入→赚钱=好），但现状描述不能回测（"当前处于中枢震荡"→无法判断说得对不对）。

> **2026-06-23 修订**：该矛盾已通过新增"实操参考"板块解决，详见 §2.6。

### 2.2 回测结果在三层运用中的设计

#### 2.2.1 第一层：策略质量监控（后台自动运行）

系统应**持续监控**各策略的历史信号质量，不依赖用户主动发起回测：

| 监控项 | 触发条件 | 产出 | 用户可见性 |
|--------|---------|------|-----------|
| 月度参数平原校验（#58已有） | 每月1日自动 | 参数稳定性报告 | 仅异常时通知 |
| CSCV过拟合概率（#56已有） | 每季度或策略变更后 | PBO值 | 仅异常时通知 |
| 策略信号漂移检测 | 持续运行 | 信号分布变化告警 | 仅异常时通知 |

**对非专业用户透明**：这些在后台运行，只有在"策略出现异常"时才通知用户，平时不打扰。

#### 2.2.2 第二层：决策支持输出（用户主动发起回测时）

当用户点击"开始回测"时，输出不应该是一堆指标数值，而应该是**结构化解读**：

```
┌─────────────────────────────────────────────────────────────┐
│  📋 回测解读（非专业用户版）                                 │
├─────────────────────────────────────────────────────────────┤
│  🟢 总体评估：良好                                          │
│                                                             │
│  📈 策略表现：                                              │
│  · 如果按系统信号操作，过去3年总收益 +35%                     │
│  · 对比沪深300同期 -2%，跑赢市场                             │
│  · 最糟糕的时候亏了 -12%，发生在2024年1月（市场整体大跌）      │
│                                                             │
│  ⚠️ 需要关注：                                               │
│  · 2025年10月之后信号质量下降（市场状态变为震荡格局）          │
│  · 缠论策略在震荡市中表现偏弱（这是正常现象）                  │
│                                                             │
│  ✅ 策略总体可靠，暂无调整必要                                │
│  💡 如果希望减少震荡期回撤，可考虑开启「风险控制」策略         │
└─────────────────────────────────────────────────────────────┘
```

**核心原则**：不说专业术语（"夏普比率0.85"），只说人话（"跑赢市场"、"最糟糕的时候"）。

#### 2.2.3 第三层：用户决策辅助（回测结果不佳时）

如果回测结果显示策略表现差，用户面临"要不要调整参数"的决策。按用户能力分三档：

| 用户类型 | 能力 | 系统提供的选择 |
|---------|------|--------------|
| 👤 初级用户 | 只看最终结果 | 仅提供"一键优化"按钮，系统自动选择最优参数组合 |
| 👤 中级用户 | 能理解"风险"概念 | 提供2-3种预设模式（保守/均衡/进取），用户选择后系统应用对应参数集 |
| 👤 高级用户 | 能理解参数含义 | 开放参数调整界面，但每个参数旁边显示推荐值和风险提示 |

**推荐：默认仅暴露"一键优化"和"模式选择"，高级参数面板默认折叠。**

### 2.3 生产环境下的参数调整机制

当前系统参数**散落在各策略文件中**，没有统一的参数管理入口：

```
volume_price_strategy.py:  window_5/10/20/30/40, threshold_0.15, 三周期分位...
chanlun_strategy.py:       lookback_period=120, bi_threshold, fenxing_confirm...
chip_strategy.py:          window=60(→120), trend_sensitivity...
bociasi_slowline.py:       lookback_252, threshold_0.02...
state_dependent_factor_weight.py: WEIGHT_MATRIX 10×5 权重矩阵...
optimizer.py:              PRESET_GRIDS 4组参数网格...
```

**重新设计为"参数档案"体系**：

```
┌─────────────────────────────────────────────┐
│  📁 参数档案管理                              │
├─────────────────────────────────────────────┤
│  📌 当前使用：均衡模式（默认推荐）              │
│  ┌─ 缠论回看窗口：120根 ✅ 推荐               │
│  └─ 量价窗口：自适应15-30根 ✅ 推荐            │
├─────────────────────────────────────────────┤
│  📌 可选预设：                                 │
│  ├─ 🟢 保守模式：更低风险，更少交易            │
│  │  · 缠论窗口：180根（更严格确认）             │
│  │  · 量价窗口上限：40根（更耐心）              │
│  │  · UPF权重：结构40%+形态20%                │
│  ├─ 🟡 均衡模式（当前）                        │
│  └─ 🔴 进取模式：更高收益，更高回撤            │
│     · 缠论窗口：60根（更快反应）               │
│     · 量价窗口下限：10根（更灵敏）             │
│     · UPF权重：形态35%+因子30%                │
├─────────────────────────────────────────────┤
│  📌 一键优化                                  │
│  └─ 基于过去12个月数据自动寻找最优参数组合       │
│     ⏳ 预计耗时：约30秒                        │
└─────────────────────────────────────────────┘
```

**参数调整的工程实现路径**：

```
当前状态：参数硬编码 → 无UI入口
↓  Phase 1（最小改动）
从数据库/配置文件读取参数 → 提供预设切换（3种模式）
↓  Phase 2（完整方案）
参数档案管理页面 → 可视化编辑 → 版本回退 → A/B对比
```

### 2.4 用户无法判断结果时的"兜底逻辑"

当用户看到回测结果但无法判断"这策略行不行"时，系统必须提供**自解释能力**：

#### 2.4.1 基准锚定——"比什么"

所有指标必须有**参照系**才能有意义：

| 指标 | 裸数字（用户无法判断） | 带参照（用户能判断） |
|------|---------------------|-------------------|
| 年化收益+5.8% | 高还是低？ | 同期沪深300是-2%，跑赢了 ✓ |
| 最大回撤-12.3% | 算大算小？ | 同期沪深300最大回撤-18%，控制得更好 ✓ |
| 夏普比率0.85 | 好还是差？ | 主动管理型基金的典型范围是0.5-1.0，处于中等偏上 ✓ |

#### 2.4.2 三档判定规则

系统内置一套**硬编码判定规则**（用户无需理解，直接看结论）：

```
综合评级系统（引擎自动计算，用户只看到结果）：

🟢 优（绿色）：可以信赖
  条件：年化收益>0 && 最大回撤<20% && 跑赢基准 && 夏普>0.5
  结论："策略表现良好，建议继续使用当前配置"

🟡 中（黄色）：需要谨慎
  条件：年化收益>0 但 最大回撤>20% 或 夏普<0.5
  结论："策略能赚钱但波动较大，建议使用保守模式降低风险"

🔴 差（红色）：不宜使用
  条件：年化收益<0 或 PBO>60% 或 最大回撤>30%
  结论："当前策略在历史数据上表现不佳，建议暂停使用并尝试一键优化"
```

#### 2.4.3 情景回溯——"什么时候会亏钱"

对非专业用户，**"什么时候会亏"比"平均赚多少"更有决策价值**：

```
📊 策略在不同市场环境下的表现：

牛市（2020-2021）  🟢 +28%  表现优异
熊市（2022）       🟢 -5%   显著跑赢大盘（大盘-21%）
震荡（2023-2024）  🟡 +3%   基本持平，信号质量下降
反弹（2024-2025）  🟢 +18%  有效捕捉反弹

💡 结论：该策略在牛熊市都能控制风险，但在震荡市信号会变弱。
      如果你认为当前市场将继续震荡，建议切换至保守模式。
```

#### 2.4.4 置信度评估——"这个回测结果可信吗"

用户还需要知道**回测结果本身的可信度**，避免对过拟合的结果盲目信任：

```
🔬 可信度评估：

过拟合概率（PBO）：12% ✅（低于20%，可信度高）
参数稳健性：参数平原宽度>30%，绩效波动8% ✅（非参数山峰）
跨周期验证：5个不同市场环境下表现一致 ✅

结论：回测结果可信，策略在多种市场条件下表现稳定。
```

#### 2.4.5 "我该做什么"终极按钮

当用户看完所有信息仍然不确定时，页面底部应有一个**明确的行动指引**：

```
┌─────────────────────────────────────────────┐
│  还是不确定？                                 │
│                                             │
│  点击「一键优化」→ 系统自动帮你找到最优参数      │
│  · 比较500+种参数组合                        │
│  · 选择风险收益比最佳的方案                    │
│  · 应用后自动更新回测结果                      │
│                                             │
│  [🔧 一键优化]                                │
└─────────────────────────────────────────────┘
```

### 2.5 "回测→运营"闭环设计

回测不应是一次性操作，而应形成持续改进闭环：

```
┌──────────┐   发现异常    ┌──────────┐   自动/手动    ┌──────────┐
│ 日常策略 ├─────────────→│ 触发回测 ├──────────────→│ 参数调整  │
│ 运行监控 │              │ 自动评估  │              │          │
└──────────┘              └────┬─────┘              └────┬─────┘
                               │                        │
                               ▼                        ▼
                        ┌──────────────┐         ┌──────────────┐
                        │ 用户确认？    │         │ 更新后的策略  │
                        │ ┌─自动接受    │         │ 回到运营监控  │
                        │ └─用户知情后  │         └──────────────┘
                        │   确认       │
                        └──────────────┘
```

**对非专业用户**：这个闭环的后半段（回测→参数调整）应该是**自动化的**，用户只需要在系统提示"策略表现下降，已自动切换为保守模式"时点"了解"即可。

## 2.6 "实操参考"板块：打通现状描述→回测信号的桥梁

### 2.6.1 缘起：三场景推演的结构化能力

现有 indicator-ide.html 中的情景化推演（A/B/C三场景）是**编码阶段示例内容**，未来正式上线时其内容将由 DeepSeek V4 实时生成或由策略配置动态决定。但重要的是其**结构模式**是通用的：

```
每个场景 = 触发条件集（可量化布尔规则） + 方向判定 + 预期结果

情景A — 放量突破：
  条件: 近3日量能≥均量1.5倍 AND 站稳中枢上沿13.50不破 AND 主力资金转为流入
  方向: 买入
  预期: 打开上行空间

情景B — 缩量回踩：
  条件: 量能萎缩至均量50%以下 AND 回踩中枢下沿12.10不破
  方向: 等待/持有
  预期: 蓄力待突破

情景C — 破位下行：
  条件: 放量跌破中枢下沿12.10 AND 情绪继续恶化 AND 主力资金持续大幅流出
  方向: 卖出/规避
  预期: 中枢下移
```

**只要未来每个场景保持"条件+方向+预期"的三元结构**，无论内容由AI生成还是由规则决定，都能被标准化地翻译为回测信号。这个模式是未来可扩展的，不受具体内容变化的影响。核心在于**条件必须是可量化的布尔表达式**（参考"近3日量能≥均量1.5倍"这种格式），而不是自然语言模糊描述。

### 2.6.2 "实操参考"板块设计

在个股策略分析页的现有"行动建议"下方，增加一个**"实操参考"**板块，内容为系统根据当前情景推演凝练出的明确操作信号：

```
┌─────────────────────────────────────────────────────────────┐
│  💡 行动建议（已有板块，保持现有风格）                        │
│  · 若放量突破 → 可谨慎关注                                   │
│  · 若缩量回踩 → 暂观察                                      │
│  · 若破位下行 → 规避风险                                    │
│  （仅供参考，不构成投资建议）                                 │
├─────────────────────────────────────────────────────────────┤
│  📋 实操参考（新增板块）                                      │
│  ┌─────────────────────────────────────────────────────┐     │
│  │ 系统模拟操作信号（仅供回测验证，不构成操作指引）        │     │
│  │                                                       │     │
│  │  当前策略组合（量价+缠论+筹码+因子+BOCIASI）            │     │
│  │  ┌──────────────────────────────────────┐            │     │
│  │  │ 开仓条件（任一触发即执行买入）：        │            │     │
│  │  │ ① 量能 ≥ 20日均量 × 1.5 AND           │            │     │
│  │  │    收盘 ≥ MA5 AND MA5方向向上          │            │     │
│  │  │ ② 回踩中枢下沿不破 AND 缩量50%+        │            │     │
│  │  │ ③ IF 情景A OR 情景B → BUY            │            │     │
│  │  │ 仓位：默认 30%初始，突破确认后加至60%    │            │     │
│  │  ├──────────────────────────────────────┤            │     │
│  │  │ 平仓条件（任一触发即执行卖出）：          │            │     │
│  │  │ ① MACD日线死叉 AND 量能萎缩            │            │     │
│  │  │ ② 跌破中枢下沿 -3%止损                 │            │     │
│  │  │ ③ IF 情景C → SELL                    │            │     │
│  │  └──────────────────────────────────────┘            │     │
│  │                                                       │     │
│  │  【→ 发送到回测】  [📊 查看历史回测结果]               │     │
│  └─────────────────────────────────────────────────────┘     │
└─────────────────────────────────────────────────────────────┘
```

**关键设计原则**：

| 原则 | 说明 |
|------|------|
| **"实操参考"不是行动建议的重复** | 行动建议是自然语言描述 + 免责声明，面向用户阅读；实操参考是结构化规则，面向回测引擎消费 |
| **明确标明用途** | "仅供回测验证，不构成操作指引"，确保用户理解这是模型评估工具而非交易指令 |
| **信号可编辑** | 用户可以在发送回测前手动调整条件（如修改仓位比例、调整止损线），系统记录用户偏好 |
| **一键发送到回测** | "发送到回测"按钮将当前规则打包传入后端回测引擎，产出回测报告 |
| **内容不受情景推演约束** | 实操参考直接输出结构化规则，情景推演只是用户侧的"为什么解释"。未来无论内容如何变化（3场景变2场景、条件数值调整、完全替换策略），只要实操参考保持"条件+方向"的规则结构，回测引擎的消费逻辑就无需改变 |

### 2.6.3 实操参考如何解决回测信号源问题

之前 §2.1 被标记为"结构性矛盾"的问题，通过实操参考板块得到了直接解决：

```
旧问题：现状描述（"处于中枢震荡"）→ 不可回测
         ↑ 缺少可验证的交易信号

新链路：情景推演（面向用户解释"为什么"）
         ↓
       实操参考（直接输出结构化的开仓/平仓规则）← **这就是回测信号源**
         ↓
       回测引擎直接消费规则 → 回测报告
         ↑ 无需翻译层，因为实操参考本身已经是机器可读的规则
```

实操参考的规则结构（以 JSON 举例）：

```json
{
  "ts_code": "000001.SZ",
  "entry_rules": {
    "logic": "AND",  // 所有条件同时满足才开仓
    "conditions": [
      {"field": "volume_ratio", "op": ">=", "value": 1.5, "label": "量能≥均量1.5倍"},
      {"field": "close_position", "op": "above", "value": "MA5", "label": "收盘站上MA5"},
      {"field": "ma5_direction", "op": "==", "value": "UP", "label": "MA5方向向上"}
    ],
    "position_pct": 0.3,
    "add_position": {"condition": "突破确认", "pct": 0.6}
  },
  "exit_rules": {
    "logic": "OR",   // 任一条件触发即平仓
    "conditions": [
      {"field": "macd_cross", "op": "==", "value": "DEATH_CROSS", "label": "MACD死叉"},
      {"field": "drawdown", "op": ">=", "value": 0.03, "label": "跌破中枢下沿-3%止损"}
    ]
  }
}
```

这个结构直接就是后端回测引擎的输入——不需要任何"翻译"或"适配"。引擎逐日检查 `entry_rules.conditions` 是否全部满足，满足则执行买入；逐日检查 `exit_rules.conditions` 是否任一满足，满足则执行卖出。

### 2.6.4 完整数据流

```
旧问题：现状描述（"处于中枢震荡"）→ 不可回测
         ↑ 缺少可验证的交易信号

新链路：情景推演（面向用户解释"为什么"）
         ↓
       实操参考（直接输出结构化的开仓/平仓规则）← **这就是回测信号源**
         ↓
       回测引擎直接消费规则 → 回测报告
         ↑ 无需翻译层，因为实操参考本身已经是机器可读的规则
```

现在整个数据流成为：

```
个股策略分析页
├─ 卡1~卡4 现状识别（五维现状卡片）  ← 纯描述，不可回测
├─ 卡5 冲突与不确定性                  ← 纯描述，不可回测
├─ 交叉验证条（4链验证）              ← 验证链质量评价
├─ 行动建议（自然语言版）              ← 面向用户阅读
├─ 📋 实操参考（结构化规则版）【新增】 ← **回测信号源**
│   └─ "发送到回测"按钮 → 后端适配器 → 回测引擎
└─ AI仲裁与建议输出                    ← 面向用户阅读

回测系统页（backtest.html）
├─ "从策略分析导入"按钮【新增】
│   从当前查看的股票拉取实操参考规则
├─ 回测配置面板（参数/预设/基准）
├─ 结果面板（指标+图表+诊断报告）
└─ "应用参数调整"按钮【新增】
    将回测优化后的参数写回策略引擎
```

这意味着**回测不再需要一个独立的数据输入过程**——用户在做个股策略分析时自然生成操作规则，一键发送即可回测。回测结果又可以反馈到策略参数优化，形成完整闭环。

### 3.4 一句话总结

> **回测系统对于非专业用户，应该是一个"自动贩卖机"而非"实验室"——投入股票代码，拿到可读的策略健康报告和明确的行动建议，而非一堆需要专业知识解读的指标数字。**

- 回测标的 = 组合策略为主（用户感受），消融回测为辅（诊断定位），单一策略仅在后台用于分析
- 回测信号源 = indicator-ide "实操参考"板块输出的结构化开仓/平仓规则 → ScenarioToSignalAdapter 翻译 → 回测引擎
- 结果运用 = 指标→文字解读，参数调整→预设模式和一键优化，无法判断→三档判定+情景回溯+置信度评估三重兜底
