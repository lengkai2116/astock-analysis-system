## 问题一：红蓝分界线与白线不一致

### 浏览器实测数据

**条件：** idx=299（日期=？, closePrice=16.33）

| 测量项 | 值 | 说明 |
|:-------|:----|:------|
| 白色当前价线 Y（chip-chart 内） | **216.05px** | 通过 DOM `getBoundingClientRect()` 十字光标对齐 ✅ |
| 红蓝理论分界 Y（chip-chart 内） | **155.26px** | 由 chip-chart 自身价格映射公式计算 |
| **偏差** | **60.79px** | **红蓝分界比白线高 60.79px** ❌ |
| 白线上方红色条（应为蓝色） | **28 条** | 错误标记为盈利 |
| 白线下方蓝色条（应为红色） | **0 条** | 全部蓝色在白线上方，未错误扩散 |

### 根因分析

#### 两套独立的坐标系

176 号方案修复了白线的 Y 坐标（改为 DOM 对齐十字光标），但**红蓝颜色的判定逻辑仍然使用 chip-chart 自身的价格映射公式**，二者使用的坐标系不同：

| 坐标系 | 用途 | closePrice=16.33 对应的 Y | 来源 |
|:-------|:-----|:--------------------------|:------|
| **坐标系 A：ECharts grid / 十字光标** | 白色线定位 | **216.05px** | `chart.convertToPixel()` + `getBoundingClientRect()` |
| **坐标系 B：chip-chart 自身价格映射** | 红蓝判定 | **155.26px** | `(1 - (closePrice - lo) / range) * 325` |

**差距来源：** 两个坐标系的 `[lo, hi]` 价格区间不同：

```
ECharts grid Y轴 范围:  12.00 ~ 20.00 (8元)  → closePrice=16.33 在 54% 位置
chip-chart 60日范围:   14.47 ~ 18.05 (3.58元) → closePrice=16.33 在 52% 位置
```

虽然百分比接近（54% vs 52%），但**容器高度 325px × 百分比差 = 实际错位可达数十像素**。

#### 当前判定逻辑

`renderChipChart()` 第 1075 行：

```javascript
var isProfitable = price <= closePrice;
```

其中 `price` 由 chip-chart 自身范围 `lo + ((i + 0.5) / bins) * range` 计算，而非由十字光标 Y 坐标反推。

#### 修正思路

**方向：** 红蓝颜色判定不应基于独立的 `price <= closePrice` 公式，而应基于该条线与白色线的**视觉位置关系**：

```
// 当前（错误）
var isProfitable = price <= closePrice;
// 线Y = 216.05时 → 红蓝边界 = 155.26，错位60.79px

// 修正后（应该）
// 每条线的 Y 与 whiteY 比较
// 线Y >= whiteY → 低于或等于收盘价 → 获利（红色）
// 线Y <  whiteY → 高于收盘价 → 套牢（蓝色）
```

**注意：** 由于筹码线使用自身坐标系计算 `lineY = (1 - (price - lo) / range) * containerH`，而白线使用 DOM 坐标对齐，两者仍不在同一坐标系。因此实际修正需要：

1. **方案 A（推荐）：** 将 price 通过 ECharts 的 `convertToPixel` 计算其实际像素 Y，与 crosshairY 比较
2. **方案 B（更可靠）：** 保持线自身 Y 坐标计算逻辑不变，但计算 `isProfitable` 时改用 `lineY >= whiteY`（线的Y与白色线Y比较——但前提是白色线Y要用与线相同的坐标系）
3. **方案 C（推荐）：** 在 `renderChipChart` 接受 `crosshairY` 后，每条线的颜色判定改为与白线位置对齐——使用与白线相同的 DOM 坐标换算

> **建议方案 B：** 最简单可靠。白线在赋值前先用芯片自身坐标计算出 `ownWhiteY` 位置，然后 `isProfitable = lineY >= ownWhiteY`。这样红蓝分界与白色线在同一条水平线上。

### 影响范围

| 项目 | 说明 |
|:-----|:------|
| 修改点 | `renderChipChart()` 中约 1 行颜色判定逻辑 |
| 工作量 | **极小（< 3 行）** |
| 风险 | 低。纯视觉修正，不涉及数据计算 |

## 问题二：筹码形态随日期变化不明显

### 浏览器实测数据

对 idx=[100, 400, 700] 三个不同日期的筹码分布进行采样：

| 日期 | closePrice | Y轴价格范围 | bins | 形态特征 |
|:----|:-----------|:------------|:-----|:---------|
| 100 → 9/27 | 13.40 | 9.50~14.17 (4.66元) | 466 | sin曲线峰形，45%处最高 |
| 400 → 9/11 | 17.25 | 15.69~19.20 (3.51元) | 351 | sin曲线峰形，45%处最高 |
| 700 → 8/27 | 16.50 | 14.97~19.00 (4.04元) | 404 | sin曲线峰形，45%处最高 |

**核心问题：** 三个日期的筹码分布**相对形状完全一致**——都是 sin 波峰形、峰值在 45% 位置、对称分布。差异仅在于：
- 价格区间 `[lo, hi]` 随日期变化（60日 lookback）
- Math.random() 产生 ±20% 随机波动

### 根因分析

模拟筹码数据的生成逻辑（`renderChipChart()` 第 1040-1052 行）是一个**与日期无关的固定公式**：

```javascript
// 这些参数在每个日期调用时完全相同
var center = 0.45;                    // 峰位固定
var weight = Math.sin(Math.min(p, 1-p) * Math.PI * 1.8);  // sin 曲线固定
var distFactor = 1 - Math.abs(p - center) * 1.6;           // 距离因子固定
// 唯一变化的是：
weight = weight * (0.6 + 0.4 * Math.random());             // ±20% 随机噪声
```

**本质：这不是一个以日期为参数的函数，而是一个静态公式。** 与真实筹码分布的核心差异：

| 特征 | 当前模拟数据 | 真实筹码分布数据 |
|:-----|:-------------|:----------------|
| 数据来源 | `sin + random()` 固定公式 | 每日逐笔成交的 **OHLCV 数据** |
| 峰形 | **单一 bell curve**，45% 高峰 | 多峰、偏态、价格集群 |
| 随日期变化 | 仅 `[lo, hi]` 范围改变 + 随机噪声 | **成交量加权分布**，每日完全不同 |
| 筹码峰位置 | 固定在 **(hi+lo)/2** 附近 | 在**成交密集区**（支撑/阻力位）|
| 真实感 | 用于原型布局验证 | 反映真实市场参与成本 |

### 真实 API 接入后的预期

后端 `GET /api/v3/chip/distribution/{ts_code}?date=` 返回的筹码数据（由 `ChipDistributionEstimator` 计算）使用：

```
每个价位筹码量 = Σ(当日成交量 × 价格权重) × 衰减系数
```

其中：
- **150 个价格区间**（非当前 350-450 动态区间）
- **0.5% 日衰减**（历史交易按日指数衰减）
- **OHLCV 估算**（每根 K 线的成交量分配到对应价格区间）
- **筹码峰检测**（自动识别成交密集区）

**预期效果：**

| 场景 | 形态变化 |
|:-----|:---------|
| 涨停日 | 大量筹码堆积在涨停价附近 |
| 放量下跌日 | 上方出现密集套牢峰（蓝色高峰）|
| 横盘震荡 | 筹码逐步向中间集中，呈单峰形态 |
| 突破关键位 | 突破前筹码峰形成支撑/阻力，突破后形态剧变 |

**结论：对接真实 API 后，筹码形态将随日期显著变化**，当前显示无明显变化是因为模拟数据的生成逻辑过于简单，而非原型编码问题。

### 红蓝分界修正方案（待确认）

修正 `renderChipChart()` 中的颜色判定逻辑，改为固定使用与白线相同的 Y 坐标基准：

```javascript
// 当前（错误 — 独立坐标系）
var isProfitable = price <= closePrice;

// 修正（与白线同一水平）
// lineY 是当前线在 chip-chart 中的 Y 坐标
// whiteY 是白线在 chip-chart 中的 Y 坐标（已通过与十字光标 DOM 对齐修正）
// 线在白色线下方或重合 → 获利（红）；线在白色线上方 → 亏损（蓝）
var isProfitable = lineY >= whiteY;
```

**关键是：** 在 `whiteY` 通过 DOM 对齐十字光标之前，先用自身坐标系计算一个**基准值**，然后红蓝判定参考这个基准值——确保红蓝分界线与白色线（以及十字光标横线）在视觉上处于同一水平。

具体做法：在计算白线位置之前，先计算芯片自身坐标系下的 `ownWhiteY`，然后白线赋值为通过 DOM 对齐的 `whiteDOMY`，而颜色判定使用 `lineY >= ownWhiteY`。

或者更彻底：颜色判定也使用 DOM 坐标系：

```javascript
// 将每条线的 price 通过 convertToPixel 换算为实际像素
// 与 crosshairY 比较
```

**已确认，修正已实施 ✅**

修正内容：
1. `renderChipChart()` 中 whiteY 计算提前到筹码线循环之前
2. `var isProfitable = price <= closePrice` → `var isProfitable = lineY >= whiteY`
3. 消除重复的 whiteY 渲染代码，保留白线 HTML 渲染
