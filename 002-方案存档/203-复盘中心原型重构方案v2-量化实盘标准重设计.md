# 203 — 复盘中心原型重构方案v2 — 量化实盘标准重设计

## 一、背景与问题陈述

复盘中心（playback.html）已于2026-06-24完成首版全量重写（~148→~1100行），实现了三视图（池概览+详情弹窗+统计）、三阶段（pending/holding/completed）和18个mock单元。但在审查中发现了以下核心问题：

### 问题1：策略标注与真实策略组合脱节

indicator-ide 是个股策略分析页，其设计体系是**多策略维度同时运行**——缠论走势分析、量价形态分析、筹码与资金、情绪环境、多因子分析、主力密码六个维度同时分析一只股票，输出各维度信号后经过交叉验证和仲裁得到综合结论。

但当前的 `addToReviewPool()` 函数仅将 `strategy_label: '综合策略'` 以及一组硬编码的 mock 信号值写入复盘池。这导致所有"加入复盘池"的条目都丢失了**真实的策略组合信息**，具体表现为：

- 不知各维度是否开启（哪些checkbox勾选了）
- 不知各维度的独立分析输出（方向、置信度、信号）
- 不知交叉验证链的结果（主力行为验证/结构位置验证/外部环境验证/共振验证）
- 不知仲裁结论和情景推演

### 问题2：缺乏量化实盘操作标准

当前系统仅模拟"买入→持有→卖出"过程，缺少量化实盘的核心机制：

| 缺失项 | 行业标准 |
|:-------|:---------|
| 资金分配模型 | 总资金→每笔分配的明确规则（固定比例/Kelly公式/槽位制） |
| 仓位计算 | 基于置信度、ATR波动率、总资金上限的风险调整 |
| 止盈止损 | 分级止盈（Tier1 50%@1R + Tier2 trailing）、ATR基准止损 |
| 组合约束 | 单只上限、单行业上限、最大持仓数、回撤熔断 |
| 完整退出原因 | 止损/一级止盈/二级止盈/移动止盈/超时/熔断/手动 |

### 问题3：缺少资金总览与日盈亏视图

顶部没有展现账户整体资金状态（总资产、持仓市值、现金余额），也没有按入池日期分组的日盈亏汇总图表来判断"整体盈多亏少"。

### 问题4：布局与样式问题

- 2列卡片布局太宽（所有卡片都用 `1fr 1fr`），3列更美观
- 详情弹窗背景透明度过高（`rgba(255,255,255,0.06)` + 遮罩 `rgba(0,0,0,0.45)`），内容难以阅读

### 2.2 量化实盘核心标准（综合提炼）

#### 2.2.1 资金分配

| 方法 | 说明 | 适用场景 |
|:-----|:------|:---------|
| **槽位制** | 每策略固定预分配资金（如10万×N槽） | 多策略并行（SilverQuant模式） |
| **Kelly公式** | `f* = (p*b - q)/b`，实际用0.25x-0.5x分数Kelly | 胜率可估算的单一方向 |
| **固定比例风险模型** | 单笔风险=总资产×0.25%-1.5% | 最通用，行业标配 |
| **ATR波动率调整** | 仓位反比于ATR：高波动小仓位 | 结合其他方法使用 |

**本方案选用**：固定比例风险模型（主）+ 置信度调整（辅）+ ATR波动率上限校验

#### 2.2.2 仓位计算公式

```
riskAmount = min(
    equity × baseRiskPct,         // 基础风险：0.5% equity
    equity × confidenceK,         // 置信度调整：0.25%~1.5%区间
    cashBalance                   // 可用现金上限
)
shares = floor(riskAmount / stopDistance)
positionValue = shares × entryPrice
// 最终校验：positionValue ≤ equity × 10%（单只上限）
```

其中 `confidenceK = 0.0025 + (confidence - 0.5) × 0.03125`（confidence 0.55→0.41%, 0.95→1.66%）

#### 2.2.3 止损模型（三线体系）

| 线位 | 计算方式 | 作用 |
|:-----|:---------|:-----|
| **硬止损** | `entryPrice - max(2×ATR, entryPrice×0.05)` | 底线，不可破 |
| **策略风控线** | 从indicator-ide的practical_ref.risk_line获取 | 策略自身的建议止损 |
| **移动止盈** | 价格突破TP1后→止损移至成本价，后续跟踪最高价-8% | 保护利润 |

#### 2.2.4 止盈模型（分级退出）

```
Tier 1（50%仓位）: entryPrice + 1R     // R = entryPrice - slPrice
Tier 2（剩余50%）: entryPrice + 2.5R   // 配合移动止盈
移动止盈轨迹:  一旦价格触及TP1，止损移至成本价+0.5R，只上不下的跟踪
```

#### 2.2.5 组合约束

| 约束 | 值 | 触发行为 |
|:-----|:---|:---------|
| 单只个股上限 | ≤总资产10% | 超出时按比例缩减仓位 |
| 单行业上限 | ≤总资产30% | 不再接受同行业新的入场 |
| 最大并行持仓 | 10只 | 超出时新条目保持在pending |
| 日回撤熔断 | 2% | 当日触发→全部强制平仓，冻结24h |
| 单次超时 | 60天 | 超时自动退出 |

#### 2.2.6 退出原因枚举

| 枚举值 | 中文名 | 触发条件 |
|:-------|:-------|:---------|
| `stop_loss_hit` | 止损触发 | 价格跌破止损线 |
| `take_profit_tier1` | 一级止盈 | 价格触及Tier1，卖出50% |
| `take_profit_tier2` | 二级止盈 | 剩余仓位触及Tier2 |
| `trailing_stop` | 移动止盈 | 从最高价回撤8%触发 |
| `time_stop` | 超时退出 | 持有超过60个交易日 |
| `circuit_breaker` | 风控熔断 | 日回撤超过2% |
| `signal_reversal` | 信号反转 | 仲裁结论方向逆转 |
| `manual` | 手动平仓 | 用户主动操作 |

### 2.3  A股交易成本标准

虚拟实盘必须计入真实交易成本，否则收益率会被系统性地高估（无成本交易比真实情况每年多出约5%-15%的净收益）。

#### 2.3.1 A股交易费用明细（2026年现行标准）

| 费用项目 | 费率 | 收取方向 | 收取规则 |
|:--------|:-----|:---------|:---------|
| **印花税** | 0.05%（万分之5） | 仅卖出时收取 | 2023年8月由0.1%降至0.05% |
| **券商佣金** | 0.015%-0.025%（万1.5-万2.5） | 买卖双向收取 | 实际费率协商确定，部分含最低5元 |
| **过户费** | 0.001%（万0.1） | 买卖双向收取 | 沪深两市均收取 |
| **证管费** | 0.002%（万0.2） | 含在佣金中 | 一般不单独列出 |
| **经手费** | 0.00341%（万0.341） | 含在佣金中 | 交易所收取 |

**本方案采用的上限标准（保守估计）：**

| 项目 | 费率 | 说明 |
|:-----|:-----|:------|
| 佣金（单边） | 0.025% | 按万2.5计算 |
| 过户费（单边） | 0.001% | 万0.1 |
| 印花税（仅卖出） | 0.05% | 万5 |
| **单笔买入成本** | **0.026%** | 佣金+过户费 |
| **单笔卖出成本** | **0.076%** | 佣金+过户费+印花税 |
| **单次交易（买入+卖出）** | **0.102% ≈ 万10** | 完整双向 |

#### 2.3.2 模拟中的费用计算逻辑

```javascript
function calculateTradeCost(price, shares, isBuy) {
  const amount = price * shares;                            // 成交金额
  const commission = Math.max(amount * 0.00025, 1);         // 佣金万2.5，最低1元
  const transferFee = amount * 0.00001;                     // 过户费万0.1
  const stampTax = isBuy ? 0 : amount * 0.0005;             // 印花税仅卖出
  return { amount, commission, transferFee, stampTax, total: amount + commission + transferFee + stampTax };
}
// 买入：支付 = 成交金额 + 佣金 + 过户费
// 卖出：回收 = 成交金额 - 佣金 - 过户费 - 印花税
// P&L计算：卖出回收 - 买入支付
```

#### 2.3.3 费用对收益率的影响示例

| 持仓时长 | 单次交易费用占比 | 年化影响（假设换手率） |
|:---------|:----------------|:--------------------|
| 短线（T+5） | 0.102%×2=0.204% | 约0.204%×48周=9.8% |
| 中线（T+20） | 0.102%×2=0.204% | 约0.102%×12月=2.45% |
| 长线（T+60） | 0.102%×2=0.204% | 约0.102%×4季=0.82% |

结论：**短线交易时费用不可忽略**，本方案所有P&L计算均需扣除交易成本后再呈现。

### 3.3 策略条件的来源与复盘中心执行的关系

> **核心原则：复盘中心是策略的执行者，不是策略的定义者。**
> 策略层面的条件（买卖点、持仓建议、止损位、止盈目标等）由个股策略分析页（indicator-ide）输出，复盘中心仅按规则虚拟执行，不做策略层面的决策。

#### 3.3.1 indicator-ide 已有策略输出（当前页面已存在）

经核实，indicator-ide 当前已完整输出以下策略层面数据，只需在 `addToReviewPool()` 中正确抓取即可：

| 策略输出 | 页面位置 | 数据结构 |
|:---------|:---------|:---------|
| 各维度分析结果 | `.dim-card[data-dim="chanlun/volume-price/chip/..."]` | direction + confidence + metrics + narrative |
| 交叉验证链 | `.cross-verify-bar .verify-item` | 4条验证链（status/brief/score） |
| 情景推演A/B/C | `.scenario-card` | 标题+概率+条件+展望 |
| 行动建议 | `.advice-body` | 建议文本 |
| **开仓条件** | `#entryConditions` | 三组条件（量能+价格+情景匹配） |
| **平仓条件** | `#exitConditions` | 三组条件（MACD死叉+止损+情景匹配） |
| **仓位建议** | entryConditions文本末尾 | "初始30%，突破确认后加至60%" |
| 数据来源 | `#arbSourcesBody` | 策略模块+知识库引用 |

**关键字段说明（开仓条件示例）：**

```
① 量能 ≥ 20日均量 × 1.5 AND 收盘站上MA5 AND MA5方向向上
② 回踩中枢下沿不破 AND 缩量至均量50%以下
③ IF 情景A(放量突破) OR 情景B(缩量回踩) → BUY
仓位：初始30%，突破确认后加至60%
```

**平仓条件示例：**

```
① MACD日线死叉 AND 量能萎缩
② 跌破中枢下沿 -3% 止损
③ IF 情景C(破位下行) → SELL
```

#### 3.3.2 数据流：indicator-ide → 复盘中心

```
indicator-ide 策略分析完成
     │
     ├── 各维度方向/置信度/信号 → dimension_outputs[] (快照记录)
     ├── 交叉验证结果 → cross_verify[] (快照记录)
     ├── DeepSeek仲裁输出 → arbitration (快照记录)
     ├── 开仓条件(entry_conditions) → 复盘中心执行依据
     ├── 平仓条件(exit_conditions) → 复盘中心执行依据
     ├── 建议止损位(risk_line) → 复盘中心虚拟止损基准
     ├── 建议目标位(target_price) → 复盘中心虚拟止盈基准
     └── 仓位建议(position_suggestion) → 复盘中心仓位分配参考
                     │
                     ▼  addToReviewPool() 抓取全部
                复盘中心(playback)
                     │
                     ├── 虚拟执行：按开仓条件检测入场时机
                     ├── 虚拟执行：按平仓条件检测出场时机
                     ├── 虚拟执行：按止损/止盈位管理风控
                     └── 复盘分析：交易完成后评定策略有效性
```

**实例说明（以西藏矿业为例）：**

| 步骤 | 角色 | 操作 |
|:-----|:-----|:------|
| 1 | indicator-ide | 2026-06-20 对西藏矿业完成6维分析，输出情景推演和实操参考 |
| 2 | 用户 | 点击"加入复盘池"，系统抓取全量策略快照 |
| 3 | 复盘中心 | 收到条目→设置为pending（待实盘），等待开仓条件触发 |
| 4 | 复盘中心 | 每日按真实A股数据检查开仓条件①：量能≥20日均量×1.5 AND 收盘站上MA5 |
| 5 | 复盘中心 | 条件满足→执行模拟买入，从账户扣减资金（含手续费），转为holding |
| 6 | 复盘中心 | 每日按真实价格更新浮盈，检查平仓条件 |
| 7 | 复盘中心 | 条件②（-3%止损）触发→模拟卖出，资金回笼（含手续费），转为completed |
| 8 | 复盘中心 | 计算P&L（扣除买卖双向费用），更新账户资金，记录退出原因 |

#### 3.3.3 复盘中心不负责的策略事项

| 事项 | 归属 | 原因 |
|:-----|:-----|:------|
| 买卖点判定 | indicator-ide的策略分析模块 | 各维度独立产生的方向信号，是策略分析的结果 |
| 仓位比例 | indicator-ide的实操参考 | 策略层面的风险判断，各策略自成体系 |
| 止损/止盈价 | indicator-ide的实操参考 | 维度信号点位的参考，来源于分析逻辑 |
| 情景推演 | indicator-ide的DeepSeek仲裁 | AI综合判断各维度的可能发展方向 |

  const grouped = {};

### 4.7.1 组合策略维度归因模型

> **核心问题**：当一只策略同时运行多维度（如缠论+量价+筹码三维），某笔交易的完整盈亏+10%应如何合理分配给各个维度？

**v2.2 方案：置信度加权归因（Confidence-Weighted Attribution）**

```
权重_i      = confidence_i / Σ(all_confidence)
归因P&L_i  = 单元P&L × 权重_i
```

示例：缠论0.85、量价0.70、筹码0.55 → 权重40.5%/33.3%/26.2%
+10% → 分别归因 +4.05%/+3.33%/+2.62%
总归因 = 原始P&L，无重复无遗漏

**优势**：比全量归因（一个单元+10%赋给每个维度，重复计入）更合理

**局限性**（界面中需透明提示用户）：
1. 置信度高 ≠ 实际贡献高。组合策略各维度信号通过仲裁融合后再决策，独立贡献本质不可观测
2. 维度间可能存在交互效应，加权无法捕捉
3. 需要更完整的信号级记录才能精确归因

**远期方向——扩展数据模型**：

在 `dimension_outputs` 中增加独立回测和决策权重字段：
```js
{
  dim: 'chanlun',
  direction: 'bullish',
  confidence: 0.85,
  standalone_backtest_pnl: '+3.2%',  // 单维回测收益
  decision_weight: 0.35               // 实际决策权重
}
```

同时在后端新增**维度级单维回测入口**，让用户查看仅缠论作为单一策略的历史表现，不依赖组合归因。

### 4.9 复盘报告生成框架（新增）

> **设计哲学变更：** 报告不是给系统自身的修订指令（系统不自修正），而是为开发平台提供可定位的诊断数据。报告的核心目标是：**定位问题出在策略组合的哪一层，以及问题的具体表现是什么。** 开发平台的大模型基于这些诊断数据判断是否需要修订。

#### 4.9.1 策略组合分层诊断模型

系统的策略运行是一个五层流水线，报告分析必须映射到这五层才能精准定位问题：

```
【输入层】个股基础数据（OHLCV + 资金流向 + 量价数据）
       ↓
  Layer 1 --- 多维度现状识别（并行）
       缠论走势分析 | 量价形态分析 | 筹码与资金 | 情绪环境 | 多因子分析 | 主力密码
       ↓
  Layer 2 --- 交叉验证链（4条）
       主力行为验证 | 结构位置验证 | 外部环境验证 | 共振验证
       ↓
  Layer 3 --- AI仲裁（DeepSeek V4 Layer 2）
       仲裁方向 + 维度一致性 + 情景推演A/B/C
       ↓
  Layer 4 --- 执行条件层（practical_ref）
       入场条件 | 出场条件 | 止损位设定 | 目标位设定
       ↓
  Layer 5 --- 资金与风控执行层（capital_allocation）
       仓位大小 | 止损距离 | 止盈层级 | 手续费影响
       ↓
【输出层】虚拟实盘执行结果（P&L + max_drawdown + exit_reason）
```

**核心原则：** 每份报告的结论定位到具体的 Layer，不跨层推测，不输出系统修订指令。

#### 4.9.5 防AI幻觉设计

1. **分层诊断限制**：每个维度的诊断限制在其所属层的数据范围内。Layer 1的问题不推测到Layer 3的影响，避免无依据的关联。

2. **只诊断不开方**：报告输出"缠论维度方向判断正确，量价维度方向判断部分正确"，而不是"建议调整量价形态参数"。系统修订由开发平台大模型基于诊断数据自行决策。报告给出的是定位数据（Where/What），不是解决方案（How）。

3. **数据完整性标注**：每份报告开头标记 `data_completeness`（0-1）和 `data_gaps[]`，缺失关键数据的维度自动降级。

4. **诊断置信度**：每个维度的诊断附带 `diagnosis_confidence`，低置信度（<0.6）仅展示为"观察项"。

5. **结构化枚举优先**：评估字段（correct/incorrect/偏早/偏晚/合理）为预定义枚举值，AI输出被限制在结构化框架内，自由文本仅限 summary 和 assessment 段落。

## 六、全自动虚拟执行规则（策略条件驱动）

### 6.1 核心原则

> **所有阶段转化由策略条件自动驱动，提前平仓为唯一手动出口。**
> 
> 复盘中心按 indicator-ide 输出的开仓/平仓条件每日自动检测。
> 当条件满足时自动执行入场/出场。持仓中卡片保留"提前平仓"按钮作为极端情况下的手动出口（不生成复盘报告）。

### 6.2 全自动状态机

```
PENDING
  │  每日自动执行：
  │    1. 读取该日真实A股数据（OHLCV + volume + MA）
  │    2. 对照 entry_conditions 逐条检查条件是否满足
  │    3. 全部条件满足 → 自动转为 HOLDING（含仓位分配+手续费扣减）
  │    4. 任一条件不满足 → 保持 PENDING，下一日继续检查
  │    5. 超过30日仍不满足 → 标记"conditions_unmet"（废弃态）
  ▼
HOLDING
  │  每日自动执行：
  │    1. 读取该日真实价格
  │    2. 更新浮盈/浮亏（基于真实 close）
  │    3. 检查硬止损（low <= stop_loss_price）→ stop_loss_hit
  │    4. 检查止盈（high >= take_profit_price）→ take_profit
  │    5. 检查 exit_conditions → 条件满足 → signal_reversal
  │    6. 持有超60天 → time_stop
  │    7. 账户回撤超限 → circuit_breaker
  ▼
COMPLETED（自动完成）
  │    1. 计算最终P&L（扣除买卖双向手续费）
  │    2. 更新账户资金（cash += 卖出回收资金）
  │    3. 记录退出原因、退出日期
  │    4. 异步生成DeepSeek复盘分析（如需）
```

### 6.3 仓位分配规则（自动触发）

```javascript
function allocatePosition(unit, account) {
  // 1. 计算风险金额
  const baseRisk = account.cash_balance * 0.005;        // 基础风险：0.5%
  const confidenceAdj = 0.0025 + (unit.confidence - 0.5) * 0.03125;
  const kellyRisk = account.cash_balance * confidenceAdj;
  
  // 2. 波动率调整（使用真实ATR）
  const atr = unit.atr || unit.base_price * 0.02;       // 真实ATR或估算值
  const stopDist = Math.abs(unit.practical_ref.risk_line - unit.base_price);
  const volCap = stopDist < atr * 1.5
    ? Math.min(baseRisk, kellyRisk) * 0.5               // 止损太近 → 减半
    : Math.min(baseRisk, kellyRisk);
  
  // 3. 单只上限校验
  const singleCap = account.initial_capital * 0.10;
  const finalRisk = Math.min(volCap, account.cash_balance, singleCap);
  
  // 4. 计算股数（含手续费预留）
  const feeRate = 0.00102;                               // 买入手续费率 ~万10
  const shares = Math.floor(finalRisk / (unit.entry_price * (1 + feeRate)));
  const positionValue = shares * unit.entry_price;
  const totalCost = positionValue * (1 + feeRate);       // 含手续费的买入总成本
  
  // 5. 更新账户
  account.cash_balance -= totalCost;
  account.position_value += positionValue;
  account.position_count++;
  account.total_fees += totalCost - positionValue;
  
  // 6. 写入unit
  unit.capital_allocation = {
    allocated_capital: totalCost,
    shares: shares,
    position_value: positionValue,
    stop_loss_price: unit.practical_ref.risk_line,
    take_profit_levels: [
      { price: +(unit.entry_price + stopDist * 2).toFixed(2), tier: 1 },
      { price: +(unit.entry_price + stopDist * 3).toFixed(2), tier: 2 }
    ]
  };
  unit.stage = 'holding';
  unit.entry_price = unit.entry_price || unit.base_price;
  unit.entry_date = dailyData.trade_date;
  unit.events.push({
    date: dailyData.trade_date,
    type: 'entry',
    desc: `开仓条件满足 → 自动买入 ${shares}股 @ ¥${unit.entry_price}（含手续费 ¥${(totalCost - positionValue).toFixed(2)}）`,
    severity: 'info'
  });
}
```

### 6.4 每日自动推进引擎（实盘数据驱动）

```javascript
function autoAdvance() {
  // 从 _lastSimDate 推进到 today，逐日处理
  let currentDate = _lastSimDate;
  const today = getToday();
  
  while (currentDate < today) {
    currentDate = addDays(currentDate, 1);
    const dailyData = getMarketData(currentDate);  // 从缓存读取真实日K线
    
    // A. 处理所有 PENDING 单元 → 检查开仓条件
    _reviewPool.filter(u => u.stage === 'pending').forEach(u => {
      const conditions = parseEntryConditions(u.practical_ref.entry_conditions);
      let allMet = true;
      for (const cond of conditions) {
        if (\!cond.evaluate(dailyData)) { allMet = false; break; }
      }
      if (allMet) {
        allocatePosition(u, _accountState);
      } else {
        // 检查是否超时废弃
        const daysSinceAdded = daysBetween(u.meta.snapshot_date, currentDate);
        if (daysSinceAdded > 30) {
          u.stage = 'completed';
          u.exit_reason = 'conditions_unmet';
          u.events.push({ date: currentDate, type: 'exit', desc: '30日未触发条件，标记废弃', severity: 'info' });
        }
      }
    });
    
    // B. 处理所有 HOLDING 单元 → 检查出场条件
    _reviewPool.filter(u => u.stage === 'holding').forEach(u => {
      // 更新当前价和浮盈
      u.current_price = dailyData.close;
      u.unrealized_pnl_pct = +((dailyData.close / u.entry_price - 1) * 100).toFixed(2);
      u.holding_days++;
      
      // 检查止损（用当日最低价）
      if (dailyData.low <= u.capital_allocation.stop_loss_price) {
        closePosition(u, _accountState, u.capital_allocation.stop_loss_price, 'stop_loss_hit');
        return;
      }
      
      // 检查止盈（用当日最高价）
      for (const tp of u.capital_allocation.take_profit_levels) {
        if (dailyData.high >= tp.price) {
          closePosition(u, _accountState, tp.price, tp.tier === 1 ? 'take_profit_tier1' : 'take_profit_tier2');
          return;
        }
      }
      
      // 检查超时
      if (u.holding_days >= 60) {
        closePosition(u, _accountState, dailyData.close, 'time_stop');
        return;
      }
    });
    
    // C. 更新账户状态
    updateAccountState(dailyData);
  }
  
  _lastSimDate = today;
  savePoolState();
}

// 页面加载时自动回补
function init() {
  loadPoolState();         // 加载 localStorage
  loadMarketCache();       // 加载行情缓存
  autoAdvance();           // 自动推进到今日
  render();                // 渲染所有视图
}
```

| 触发 | 页面顶部"一键生成报告"按钮点击后执行 |

### 8.4 跨页面通知机制

通过 localStorage 事件桥接通知到其他页面：

```javascript
// 写方（playback页面）
localStorage.setItem('review_notification', JSON.stringify({
  type: 'report_complete',
  count: 8,
  timestamp: Date.now()
}));

// 读方（任何页面，通过 storage 事件监听）
window.addEventListener('storage', function(e) {
  if (e.key === 'review_notification') {
    const data = JSON.parse(e.newValue);
    if (data.type === 'report_complete') {
      showNotification(data.count + ' 份报告已生成并存档到报告中心');
    }
  }
});

// 同页面轮询兜底
setInterval(function(){
  const raw = localStorage.getItem('review_notification');
  if (raw) {
    const data = JSON.parse(raw);
    if (data.type === 'report_complete' && !data.acked) {
      showCompleteNotification(data.count);
      data.acked = true;
      localStorage.setItem('review_notification', JSON.stringify(data));
    }
  }
}, 3000);
```

### 9.2 接收方合并逻辑

```javascript
function readNewEntries() {
  // 1. 优先读v2
  const raw = localStorage.getItem('review_pool_entries_v2');
  if (raw) {
    const entries = JSON.parse(raw);
    localStorage.removeItem('review_pool_entries_v2');
    return entries;
  }
  // 2. fallback：读v1（迁移兼容）
  const rawV1 = localStorage.getItem('review_pool_entries');
  if (rawV1) {
    const entries = normalizeV1ToV2(JSON.parse(rawV1));
    localStorage.removeItem('review_pool_entries');
    return entries;
  }
  return [];
}
```
