---

## 一、战略层：从"脚本集合"到"策略系统"的转型

### 1.1 当前系统状态

```
代码丰富度: ★★★★☆    (策略代码覆盖了绝大部分功能)
架构一致性: ★★☆☆☆    (实际数据流与文档描述偏差大)
路径清洁度: ★☆☆☆☆    (两条芯片信号路径并存，无人调用的框架代码)
可迭代性:   ★★☆☆☆    (信号输出格式混用，难以做批量回测验证)
```

### 1.2 核心矛盾

策略体系（132文档）描述的架构偏理想化，实际运行路径由 `SignalComputationService` 直通。两条芯片信号路径并存：
- **薄路径**（实际在跑的）：`SignalComputationService._compute_chip_signal()` → `ChipScorer.score()`（简化评分器，3维度固定权重）
- **厚路径**（代码完整但无人调用）：`ChipDistributionStrategy.analyze()` → `TradingPhaseDetector` + `ChipDistributionSignalGenerator` + `ConfirmLayer`

### 1.3 转型目标

从一个"前端展示切片"转型为**可自我验证的策略系统生命周期**：

```
策略设计 → 代码实现 → 信号输出 → 赢率证据 → 虚拟验证 → 用户实盘 
    ↑                                                    ↓
    └────────────── 策略自我复盘与迭代 ──────────────────┘
```

## 二、需求逐项分析与确认

### 需求1：三层递进筛选

| 层 | 输入 | 功能 | 频率 | 输出 |
|----|------|------|------|------|
| **L1 风险剔除** | 全市场~5000只 | 基本面 + 规则过滤 | 月度/季度重筛，每日增量 | 安全股票池~2000-3000只 |
| **L2 筹码策略** | L1 输出 | 主力行为识别 | 每日运行 | 主力关注股票~50-100只 |
| **L3 多策略验证** | L2 输出 + 自选股 | 缠论/量化/因子 + AI | 按需运行 | 操作建议 + 赢率证据 |

**约定了三层之间的关系**：L1→L2→L3 是递进关系，不是并列关系。L2 和 L3 同时也要支持**单只股票的独立深入分析**。

### 需求2：单只股票分析支持

L2 和 L3 不仅支持"三层流水线"中的中间层，还要支持用户从自选股、看盘等场景直接进入的策略分析：

```
场景A（三层流水线）: 全量股票 → L1 → L2 → L3 → 输出摘要版(结论简述)
  每只附 suggestion/direction + entry_window + position_pct + holding
  适合列表快速浏览, 用户点击某只进入场景B深入研究

场景B（单股分析）  : 用户自选股/L1选股/L2结果 → L2分析 + L3分析 → 输出完整版(结论+分析链)
  在上述结论基础上, 追加 evidence[] 每项含指标值+判断依据
  + signal_source_detail(触发信号类型/阶段判定依据/各维度指标值)
  适合深入研究: 用户能看到"为什么得出这个结论"
  
  ★ 场景B添加入口: [是否进行虚拟实盘验证] (用户可选开关)
  若用户勾选, 系统记录该股当前策略建议(方向/入场/止损/目标)
  后续按 T+5/T+10/T+20 回调实际股价走势, 输出验证结果到
  账户管理的"虚拟验证复盘"分区

场景C（全市场扫描）: 全量股票 → L1快速过 → [用户选择] → 跳转场景B
```

### 需求3：虚拟验证体系（双轨制）

策略不参与实盘操作，但所有策略输出都必须**可验证**。验证体系分两轨并行：

#### 轨A：后台自动验证（全部策略输出）

每当用户通过方案一或方案二生成策略信号，系统**在后台自动记录并验证**所有信号：

```
后台日志（用户无感知）:
  SignalRecord: {
    ts_code, signal_date, signal_type(方向),
    entry_zone, risk_line, target_zone,
    confidence, backtest_evidence_initial      ← 历史赢率证据
    created_at, status: 'pending'|'completed'
  }
```

信号发出后系统自动回调：
```
T+5:  拉取 signal_date → T+5 实际走势，更新 win_rate_5d / avg_return_5d / sharpe_5d
T+10: 同上，更新 win_rate_10d / avg_return_10d
T+20: 同上，更新 win_rate_20d / avg_return_20d / max_drawdown_20d
status → 'completed'
```

每条 SignalRecord 的完整生命周期：
```
[信号生成] → status=pending → [T+5更新] → ... → [T+20更新] → status=completed
                                                              ↓
                                                      进入周度汇总
```

**验证结果用途**：不进前端展示（不在信号卡片中），只用于后台策略自我复盘和迭代。

#### 轨B：用户可选虚拟实盘验证（单股分析）

在场景B（单只股票分析）页面添加用户可选开关 **"[是否进行虚拟实盘验证]"**。

用户勾选后：
```
1. 记录该股当前策略建议快照:
   {ts_code, date, strategy_name, suggestion, entry_zone, risk_line, target_zone, confidence}
2. 创建 VirtualPosition 记录:
   {ts_code, entry_price(取entry_zone中值), virtual_capital, status='tracking'}
3. 按 T+5/T+10/T+20 回调实际股价走势
4. 验证结果输出到账户管理 → "虚拟验证复盘" 分区
   (与"用户实际操作复盘"分区隔离)
```

#### 两轨对比

| 维度 | 轨A·后台自动 | 轨B·用户可选 |
|------|-------------|-------------|
| 触发方式 | 任何信号生成自动触发 | 用户在单股分析页面手动勾选 |
| 数据量 | 全部信号（可能很大） | 仅用户选择的股票 |
| 结果输出 | 周度汇总报告(不入前端) | 账户管理 → 虚拟验证复盘分区 |
| 用途 | 策略自我复盘和迭代 | 用户可对比自己实际操作的偏差 |

### 需求4：复盘体系（双通道 — 实盘 + 虚拟）

复盘功能分为两个独立分区，统一放在账户管理页面的"复盘"Tab下：

#### 分区A：用户实际操作复盘（125文档对接）

用户在账户管理录入的实盘交易，与系统当时的策略信号做**匹配度分析**：

```
实盘买入 → 当日系统信号是什么？是否在建议的 entry_zone 内？
实盘卖出 → 卖出价 vs target_zone / risk_line？是否过早/过晚？
复盘输出 → 七维评分（大盘/板块/个股/策略/资金/心态/归因）
```

#### 分区B：选定股票虚拟验证复盘

用户在单股分析页勾选"虚拟实盘验证"后，验证结果在此分区展示：

```
虚拟验证复盘列表:
  股票  | 验证日期 | 策略结论 | 实际走势 | 结果判定 | 偏差分析
  茅台  | 06-01   | BUY 185  | T+5:+3.2%| ✅ 正确  | 目标达成92%
  宁德  | 06-01   | HOLD    | T+5:-2.1%| ⚠ 偏差  | 应触发止损...
  茅台  | 05-25   | SELL 192 | T+5:+1.5%| ❌ 过早  | 卖出信号过强...
```

每条验证记录的详细内容：
```
VirtualVerification: {
  id, ts_code, signal_record_id,
  strategy_snapshot: {方向, entry_zone, risk_line, target_zone, confidence},
  checkpoints: [
    {date: T+5, price, vs_entry: +3.2%, vs_target: 92%, vs_risk: +8.1%},
    {date: T+10, price, ...},
    {date: T+20, price, ...}
  ],
  final_judgement: 'CORRECT'|'WRONG'|'PARTIAL',
  deviation_analysis: '过早止盈:卖出后仍上涨+3.1%',
  created_at
}
```
