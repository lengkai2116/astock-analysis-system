## 2. QuantDinger项目全景

### 2.1 核心定位

QuantDinger 是一个**自托管的本地优先 AI 量化交易平台**，口号是"用自然语言描述交易想法 → AI 自动生成 Python 策略 → 回测 → 实盘执行"，致力于成为 TradingView 和 QuantConnect 的自托管替代方案。

### 2.2 技术架构

```
┌─────────────────────────────────────┐
│         Docker Compose              │
│                                     │
│  ┌───────────────────────────────┐  │
│  │  frontend (Nginx + Vue SPA)  │  │  → :8888
│  └──────────────┬────────────────┘  │
│                 │ /api/* proxy       │
│  ┌──────────────▼────────────────┐  │
│  │  backend (Flask + Gunicorn)   │  │  → :5000
│  └──────────────┬────────────────┘  │
│  ┌──────────────▼────────────────┐  │
│  │  PostgreSQL 16 / Redis 7      │  │
│  └───────────────────────────────┘  │
│  外部集成: LLM APIs · 交易所 · MCP  │
└─────────────────────────────────────┘
```

### 2.3 核心亮点

1. **7-Agent 多智能体AI系统** — 三阶段协作分析：并行分析(5 Agent) → 结构化辩论(2 Agent) → 决策输出(1 Agent)
2. **Vibe Coding 策略开发** — 自然语言描述想法 → AI 自动生成可执行的 Python 策略代码
3. **双策略运行时** — `IndicatorStrategy`（向量化快速验证）+ `ScriptStrategy`（事件驱动生产级）
4. **多市场支持** — 加密(CCXT) + 美股(IBKR/Alpaca) + 外汇(MT5) + A股(AkShare)
5. **MCP Server** — 对外提供 Agent Gateway，供 Cursor/Claude Code 调用
6. **本地 RAG 记忆系统** — 经验检索注入智能体 Prompt，评分公式 `score = w_sim·sim + w_recency·recency + w_returns·returns_score`
7. **内建变现系统** — 会员订阅 + 指标市场 + USDT 支付（本项目不适用）

### 5.3 核心差距分析：多Agent AI 系统

**QuantDinger 的 7-Agent 架构：**

```
阶段1 (并行): 市场分析师 · 基本面 · 新闻 · 情绪 · 风险
         ↓ 汇集各自分析结果
阶段2 (辩论): 多头研究员 vs 空头研究员
         ↓ 正反双方论证
阶段3 (决策): 交易员Agent → BUY/SELL/HOLD + 置信度%
```

**本项目现状：** 单Agent — 调用一次 DeepSeek API，传入行情+指标数据，得到分析结果。缺少：
1. ✅ 多视角并行分析（基本面/技术面/情绪/新闻/风险）
2. ❌ 结构化辩论机制（看多 vs 看空结构化论证）
3. ❌ 独立决策Agent（汇总所有输入后做最终判断）
4. ❌ 记忆系统（历史经验检索注入）
5. ❌ 分析结果的链式积累（每次分析结果不跨Session复用）

### 5.4 核心差距分析：Vibe Coding 策略生成

**QuantDinger 的 Vibe Coding 流程：**
1. 用户在策略编辑器输入自然语言（如"我要MACD金叉+RSI过滤的BTC 15分钟策略"）
2. LLM 生成 Python 策略代码（IndicatorStrategy 或 ScriptStrategy 格式）
3. 代码直接可视化到 K-line 图表
4. 一键回测 → 一键实盘

**本项目现状：**
1. ❌ 无自然语言→代码转换通道
2. ❌ 无受控代码执行环境
3. ✅ 已有代码编辑器（待完整接入）
4. ✅ 已有 K-line 图表展示能力

#### 6.1.1 多Agent AI分析系统 —— 300-Breakthrough-1

**目标：** 实现三阶段多Agent分析流水线，赶上QuantDinger的AI能力

**设计概要：**

```
POST /api/v3/ai/multi-agent-analysis
         ↓
┌─────────────────────────────────────────────┐
│  Phase 1 — 并行分析 (5 Agents)              │
│  ┌──────────┐ ┌──────────┐ ┌─────────────┐  │
│  │ 技术面   │ │ 基本面   │ │ 情绪面      │  │
│  │ Analyst  │ │ Analyst  │ │ Analyst     │  │
│  └──────────┘ └──────────┘ └─────────────┘  │
│  ┌──────────┐ ┌──────────┐                  │
│  │ 新闻     │ │ 风险     │                  │
│  │ Analyst  │ │ Analyst  │                  │
│  └──────────┘ └──────────┘                  │
│         ↓ 汇集Agent分析结果                   │
├─────────────────────────────────────────────┤
│  Phase 2 — 结构化辩论                        │
│  ┌──────────────┐  ┌──────────────┐          │
│  │ 看多研究员   │  │ 看空研究员   │          │
│  └──────────────┘  └──────────────┘          │
│         ↓ 正反论证                            │
├─────────────────────────────────────────────┤
│  Phase 3 — 最终决策                          │
│  ┌──────────────────────┐                   │
│  │ 决策Agent            │                   │
│  │ → 买入/卖出/持有     │                   │
│  │ → 置信度% + 推理链    │                   │
│  └──────────────────────┘                   │
└─────────────────────────────────────────────┘
         ↓
返回结构化分析报告 + 决策建议
```

**关键实现细节：**
- 5个Phase1 Agent可并行调用（本项目已有 DeepSeek API 基础设施）
- 每个Agent使用不同的System Prompt + 不同的数据切片
- Phase2辩论：同一LLM用不同上下文分别扮演多空角色
- Phase3决策：汇总所有前序输出，做最终判断
- 利用本项目已有的 `deepseek_analysis_service.py` 和 `ai_context_builder.py` 改造

**估计工作量：** 3-5 天（后端Agent编排）+ 1-2 天（前端展示改造）
**依赖：** 已有 `deepseek_analysis_service.py` / `ai_context_builder.py`

#### 6.1.2 Vibe Coding 策略生成 —— 300-Breakthrough-2

**目标：** 支持自然语言描述 → AI自动生成可运行的策略代码

**设计概要：**

```
用户输入："5日均线上穿10日均线，且RSI在30-70之间时买入..."
         ↓
┌─────────────────────────────────┐
│  VibeCoding Service             │
│  1. NL → Strategy Prompt组装    │
│  2. LLM生成Python策略代码        │
│  3. 语法校验 + 沙箱预跑         │
│  4. 返回策略代码 + 信号可视图    │
└─────────────────────────────────┘
         ↓
┌─────────────────────────────────┐
│  前端展示                       │
│  1. Monaco Editor展示代码       │
│  2. K线图叠加信号标记           │
│  3. 一键保存为策略模板           │
│  4. 一键回测                    │
└─────────────────────────────────┘
```

**关键实现细节：**
- 复用本项目的 `indicator_contract.py` 和 `indicator_sandbox.py` 做代码校验
- 生成代码格式：现有 `BaseStrategy` 格式即可（兼容策略流水线）
- 策略代码保存到 `StrategyConfig` 模型
- 前端集成 Monaco Editor（需新增依赖）

**估计工作量：** 2-4 天（后端服务）+ 2-3 天（前端集成）
**依赖：** 已有 `strategy_template_service.py` / `BaseStrategy` 框架

### 6.2 P1 优先级（建议1-2月内启动）

#### 6.2.1 双策略运行时抽象

**目标：** 借鉴 QuantDinger 的 IndicatorStrategy + ScriptStrategy 双模式

**建议方案：**
- IndicatorStrategy 模式：向量化信号，`df["buy"]` / `df["sell"]` 列模式
- ScriptStrategy 模式：事件驱动，`on_bar()` / `on_init()` 回调模式
- 两者均映射到现有 `BaseStrategy` 基类
- 保留当前流水线的兼容性

**工作量估计：** 2-3 天

#### 6.2.2 RAG 交易记忆系统

**目标：** 基于历史分析结果改善AI建议质量

**建议方案：**
- 利用现有 `UserMemory` 模型 + `ai_context_builder.py`
- 增加评分维度：分析日期、后续表现、置信度
- 每次AI分析前检索相关历史建议
- 实现 QuantDinger 的 `score = w_sim·sim + w_recency·recency + w_returns·returns_score`

**工作量估计：** 2-3 天

#### 6.2.3 策略执行沙箱

**目标：** 安全执行用户/AI生成的策略代码

**建议方案：**
- Python `exec()` + `multiprocessing` 子进程隔离（借鉴QuantDinger）
- 受控上下文：只暴露 `data`, `indicators`, `buy`, `sell` 等有限API
- 超时保护（timeout 30s）+ 资源限制
- 可复用现有 `indicator_sandbox.py`

**工作量估计：** 2 天
