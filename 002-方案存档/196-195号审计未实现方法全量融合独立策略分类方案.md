# 195号审计未实现方法全量融合/独立策略分类方案

## 概述

基于 [195-181号方案方法实施审计报告.md](002-方案存档/195-181号方案方法实施审计报告.md) 中识别的全部未实现方法（含 ❌ 和 ⚠️），结合 LLM Wiki 知识库（787个概念文件）中各方法的领域归属和关联性，进行系统性分类：

1. **🔗 融入现有策略** — 增强现有策略的现状识别深度，不产生冲突
2. **🧩 形成新的独立策略** — 自成体系，供用户通过策略模版页面 + 选股系统 vibe coding 模块选择

---

## 决策原则

| 原则 | 说明 |
|------|------|
| 逻辑同源性 | 方法与现有策略源于相同理论体系（如缠论方法→缠论策略），优先融合 |
| 知识库关联性 | LLM Wiki 概念文件中的 shared tags 和 cross-references 决定方法归属 |
| 输出范式兼容 | 融合后不改变现有策略的输出结构（现状叙述 → 非评分） |
| 独立门槛 | 新策略必须有完整的分析框架（输入→分析→输出），用户能独立理解和使用 |
| 零冲突 | 融入的方法只增加判断维度，不修改现有方法的判断逻辑或阈值 |

### 4.1 融合类零冲突规则

1. **只增不减**：所有新方法只作为新检测项追加，不修改现有检测方法的阈值或逻辑
2. **命名空间隔离**：新方法使用 `_is_xxx()` / `_detect_xxx()` 命名前缀，不与现有方法冲突
3. **可选输出**：新方法的输出作为 `detail` 字段中的扩展属性，不改变现有输出结构的必选字段
4. **配置开关**：新检测方法默认开启，但可通过策略配置参数关闭（新增 `enabled_extended_detectors` 列表）

### 4.2 独立策略零冲突规则

1. **独立管道**：新策略在 `SignalComputationService.compute_for_stock()` 中新增独立 `try/except` 块，单策略失败不影响其他策略
2. **UPF热插拔**：新策略通过 UPFEngine 新增 layer 接入，各 layer 权重总和归一化(1.0)
3. **独立输出字段**：新策略输出统一使用 `{strategy_name}_result` 键名，不与现有策略重叠
