
### 1.2 核心问题象限

按影响面×修复成本划分：

| 象限 | 问题 | 优先级 |
|:----:|------|:------:|
| **高影响×低成本** | P1 交易语义/免责缺失 → 文案替换<br>P2 硬编码色值 → 引用CSS变量<br>P2 按钮emoji → 独立图标 | 🔴 立即 |
| **高影响×中等成本** | P2 API 401/500错误 → 排查后端认证<br>P2 getWatchlistData缺失 → 补充函数<br>P2 /api/factor-combinations 500 → 后端修复 | 🟡 第1期 |

### 1.4 当前前端能做什么（后端能力 vs 前端展示缺口）

| 后端能力 | 前端现状 | 缺口 |
|---------|:--------:|:----:|
| 100+ Alpha因子计算 | 因子管理页面已有 | ⚠️ 因子详情展示不足（IC/IR等分析） |
| 缠论分析（分型/笔/中枢） | 指标IDE已有，挂载为策略 | ⚠️ K线图上的缠论可视化标记未完整展示 |
| 筹码分布分析 | 无独立页面 | ❌ **完全缺失** |
| AI多角色分析 | AI分析页面已有 | ✅ 基础可用 |
| 策略回测V2引擎 | 回测页面已有 | ⚠️ 结果展示深度不足（缺少权益曲线/回撤曲线等） |
| 实时数据推送 (SocketIO) | 自选监控页面已有 | ✅ 基础可用 |
| 行情数据同步/管理 | 无独立入口 | ❌ **完全缺失**（诊断用） |
| 策略模板系统 | 策略模板页面已有 | ✅ 基础可用 |
| 报告生成 | 报告中心页面已有 | ✅ 基础可用 |
| 复盘机制 | 无独立页面 | ❌ **完全缺失** |
| 系统诊断/健康检查 | 无独立入口 | ❌ **完全缺失** |

**结论**：后端能力富余，前端展示存在3个明显缺口。但不应盲目新建页面，需评估哪些"真的需要"、哪些"可以暂缓"。

```
┌─ Layer 4: 业务页面  ──────────────────────────────────────────┐
│  dashboard / indicator-ide / screener / watchlist / ...       │
│  每个页面 = 1 view + 若干业务子组件                            │
└───────────────────────────────────────────────────────────────┘
                             依赖
┌─ Layer 3: 业务组件  ──────────────────────────────────────────┐
│  KLineChart / SignalPanel / FactorSelector                    │
│  StrategySignalPanel / ReviewReport / ...                     │
└───────────────────────────────────────────────────────────────┘
                             依赖
┌─ Layer 2: 通用组件  ──────────────────────────────────────────┐
│  DisclaimerFooter / EmptyState / DataSourceStatusBar          │
│  ErrorBoundary / ShortcutsHelp / LoadingSpinner               │
│  (AntDV 标准组件不封装，直接使用)                             │
└───────────────────────────────────────────────────────────────┘
                             依赖
┌─ Layer 1: 基础层  ────────────────────────────────────────────┐
│  global.less (Token系统) / Pinia Stores / Router / Services  │
│  Axios / i18n / Moment                                        │
└───────────────────────────────────────────────────────────────┘
```

**核心原则**：
- Layer 1 不动（配置/路由/状态/API服务不重构）
- Layer 2 补充缺失组件（免责声明/空状态等），新增少量通用组件
- Layer 3 优化现有业务组件（颜色Token化/交互增强）
- Layer 4 分期创建新页面，不做大规模重写

本方案在162号检查报告的客观结论基础上，结合后端全量能力映射分析，制定了 **"先止血 → 再规范 → 后补强 → 最后优化"** 的四期渐进路线。

| 阶段 | 核心交付 | 投入 |
|:----:|---------|:----:|
| Phase 0 | P1问题修复（交易语义+免责声明） | 2小时 |

| Phase 3 | 浅色主题/复盘中心/VueUse集成/性能优化 | 5天 |

**核心原则**：不重构现有框架，在既定技术栈内做定向增强。前两期侧重"对错"（规范性问题），后两期侧重"好坏"（体验性问题）。
