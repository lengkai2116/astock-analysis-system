## P1级问题可行性评估

### 2. 前端响应拦截规则与后端返回结构不兼容

#### 问题描述

- **位置**: `frontend/vue-project/src/utils/request.js:26-29`
- **问题**: 前端期望 `res.code === 1/200`，后端返回 `{ success: true/false, data, message }`
- **影响**: 绝大多数后端成功响应被前端误判为失败

#### 代码现状分析

```javascript
// request.js (前端)
service.interceptors.response.use(
  response => {
    const res = response.data
    if (res.code !== 1 && res.code !== 200) {  // 检查 code 字段
      Message.error(res.msg || '请求失败')
      return Promise.reject(new Error(res.msg || '请求失败'))
    }
    return res
  }
)
```

```python
# 后端响应风格 (大多数接口)
return jsonify({
    'success': True,    # 使用 success 而非 code
    'data': data,
    'message': 'success'
})
```

#### 可行性评估

| 评估维度 | 评分 | 说明 |
|---------|------|------|
| 修复难易度 | ⭐⭐⭐⭐ (4/5) | 需要统一前后端契约 |
| 修复时长 | ⭐⭐⭐⭐ (4/5) | 2-3小时可完成 |
| 测试难度 | ⭐⭐⭐ (3/5) | 需要全面测试接口 |
| 影响范围 | ⭐⭐⭐⭐⭐ (5/5) | 影响所有 API 接口 |
| 总体可行性 | ⭐⭐⭐⭐ (4/5) | **中等难度但必须修复** |

#### 具体修复方案

##### 方案对比

| 方案 | 优点 | 缺点 | 推荐度 |
|------|------|------|--------|
| **方案A：前端兼容后端** | 改动集中，风险小 | 后端风格不够规范 | ⭐⭐⭐⭐⭐ |
| 方案B：后端统一返回 code | API更规范 | 需要改所有接口 | ⭐⭐⭐ |

##### 推荐方案：修改前端（方案A）

```javascript
// request.js
service.interceptors.response.use(
  response => {
    const res = response.data
    
    // 兼容两种格式
    if (res.code !== undefined) {  // 如果有 code 字段
      if (res.code !== 1 && res.code !== 200) {
        Message.error(res.msg || '请求失败')
        return Promise.reject(new Error(res.msg || '请求失败'))
      }
      return res
    } else if (res.success !== undefined) {  // 如果有 success 字段
      if (!res.success) {
        Message.error(res.message || res.msg || '请求失败')
        return Promise.reject(new Error(res.message || res.msg || '请求失败'))
      }
      return res
    } else {
      // 直接返回原始数据
      return res
    }
  }
)
```

##### 修复预估工作量

- **编码时间**: 30分钟
- **测试时间**: 1.5小时（测试主要页面）
- **代码审查**: 15分钟
- **总计**: ~2小时
