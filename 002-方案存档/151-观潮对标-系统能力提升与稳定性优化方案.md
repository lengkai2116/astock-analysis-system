## 2. 核心借鉴：计算架构升级

### 2.1 Web Worker 计算池

**观潮做法**：`AdvancedIndicatorWorkerPool` 通过 Web Worker 池并行计算技术指标，主线程仅负责渲染。包含 `ChartWorkerBridge` (通信桥) 和 `ComputeScheduler` (任务调度器)。

**现状**：指标计算在 Python 后端服务中串行执行（依赖 `services/signal_computation_service.py`），前端无并行计算能力。

**改进方案**：在浏览器端引入轻量级 Web Worker 计算池，将高频实时计算移至前端。

#### 架构设计

```
┌───────────────┐      ┌───────────────────────┐
│  UI Thread    │      │  Worker Pool          │
│               │      │                       │
│  KLineChart   │─────▶│  Worker 1 (缠论)       │
│  Resampling   │      │  Worker 2 (量价)       │
│  Calc Request │      │  Worker 3 (因子)       │
│               │      │  Worker 4 (形态)       │
│  Result ←─────│──────│  ComputeScheduler     │
│  Render       │      │  (任务队列+优先级)     │
└───────────────┘      └───────────────────────┘
```

**Worker 池实现**：

```javascript
// services/workerPool.js — Web Worker 池
class WorkerPool {
  constructor(workerCount = navigator.hardwareConcurrency || 4) {
    this.workers = []
    this.taskQueue = []
    this.pendingTasks = new Map()
    this.taskId = 0

    for (let i = 0; i < workerCount; i++) {
      const worker = new Worker(
        new URL('./indicatorWorker.js', import.meta.url),
        { type: 'module' }
      )
      worker.onmessage = (e) => this._handleResult(e)
      this.workers.push({ worker, busy: false })
    }
  }

  // 提交计算任务
  submit(type, data, priority = 'normal') {
    // 优先级: high > normal > low
    return new Promise((resolve, reject) => {
      const task = {
        id: ++this.taskId,
        type,          // 'chanlun' | 'volume_price' | 'chip' | 'factor'
        data,          // { symbol, period, klines, params }
        priority,
        resolve,
        reject,
      }
      this.taskQueue.push(task)
      this._schedule()
    })
  }

  // 调度: 按优先级排序 + 分配空闲 Worker
  _schedule() {
    // 观潮的 ComputeScheduler 逻辑
    this.taskQueue.sort((a, b) => {
      const prio = { high: 0, normal: 1, low: 2 }
      return prio[a.priority] - prio[b.priority]
    })

    for (const entry of this.workers) {
      if (!entry.busy && this.taskQueue.length > 0) {
        const task = this.taskQueue.shift()
        entry.busy = true
        this.pendingTasks.set(task.id, task)
        entry.worker.postMessage({
          id: task.id,
          type: task.type,
          data: task.data,
        })
      }
    }
  }

  _handleResult(e) {
    const { id, result, error } = e.data
    const task = this.pendingTasks.get(id)
    if (task) {
      if (error) task.reject(error)
      else task.resolve(result)
      this.pendingTasks.delete(id)
    }
    // 释放 Worker
    const entry = this.workers.find(w => w.idle === false)
    if (entry) entry.busy = false
    this._schedule()
  }
}
```

**适配现有体系的注意点**：
- 当前我们的指标计算在 Python 后端，Worker 池解决的是前端 K 线图表侧边展示的轻量计算（如简单移动平均、RSI、MACD 等实时计算）
- 后端仍负责大规模策略回测和因子计算
- Worker 脚本中使用 `comlink` 或原生 `postMessage` 通信

**实施路径**：

| 步骤 | 内容 | 工作量 |
|------|------|--------|
| 1 | 搭建 Worker 池基础设施 (WorkerPool 类) | 1天 |
| 2 | 迁移缠论指标计算到 Worker (indicatorWorker.js) | 2天 |
| 3 | 迁移量价指标计算到 Worker | 2天 |
| 4 | 实现 ComputeScheduler 优先级调度 | 1天 |
| 5 | 集成到 KLineChart 组件 | 1天 |

### 2.2 指标计算引擎重构

**观潮做法**：`ChartIndicatorManager` + `IndicatorEngine` + `AdvancedIndicatorWorkerPool` 三层架构。

**现状**：`backend/app/indicators/` 目录下包含基础指标定义，但缺少统一管理引擎。

**改进方案**（前端侧）：

```javascript
// 前端统一指标计算引擎 (配合 Worker 池)
class IndicatorEngine {
  constructor(workerPool) {
    this.pool = workerPool
  }

  // 预置指标定义 (观潮 ISP 23 函数的子集)
  static PRESETS = {
    ma: (data, period) => SMA(data, period),
    ema: (data, period) => EMA(data, period),
    macd: (data, fast, slow, signal) => MACD(data, fast, slow, signal),
    rsi: (data, period) => RSI(data, period),
    boll: (data, period, std) => BOLL(data, period, std),
    kdj: (data, n, k, d) => KDJ(data, n, k, d),
    // ... 观潮 ISP 23 函数
  }

  async calculate(symbol, indicators, params) {
    // 提交到 Worker 池并行计算
    const tasks = indicators.map(ind =>
      this.pool.submit(ind.type, { symbol, indicator: ind, ...params })
    )
    return Promise.all(tasks)
  }
}
```

### 2.3 数据服务层统一

**观潮做法**：`DataService` 作为统一数据获取层，前端所有数据请求经过 DataService 路由。

**现状**：前端 `services/` 下多个独立 service 文件（chartService/screenerService/aiAnalysisService 等），无统一数据层。

**改进方案**：引入统一 `DataService` 封装层，不替代现有 service 文件，而是在上层增加统一接口。

```javascript
// services/dataService.js — 统一数据服务层 (参考观潮 DataService)
export class DataService {
  constructor() {
    this.cache = new Map()           // 内存缓存
    this.pending = new Map()         // 请求去重
  }

  // 统一请求入口
  async fetch(endpoint, params = {}) {
    const cacheKey = `${endpoint}?${JSON.stringify(params)}`
    // 请求去重 (同一端点+参数同时请求时复用)
    if (this.pending.has(cacheKey)) {
      return this.pending.get(cacheKey)
    }
    // 缓存检查 (5s 缓存)
    if (this.cache.has(cacheKey)) {
      const { data, ts } = this.cache.get(cacheKey)
      if (Date.now() - ts < 5000) return data
    }
    const promise = axios.get(endpoint, { params })
    this.pending.set(cacheKey, promise)
    const response = await promise
    this.pending.delete(cacheKey)
    this.cache.set(cacheKey, { data: response.data, ts: Date.now() })
    return response.data
  }

  // 批量请求 (观潮 klines/batch API 类似)
  async batch(requests) {
    return Promise.all(requests.map(r => this.fetch(r.endpoint, r.params)))
  }
}
```

### 3.2 股票搜索服务

**观潮做法**：服务端拼音搜索 `vendor-pinyin-*.js` 集成。

**现状**：前端本地搜索，无拼音支持，搜索性能受限于本地数据量。

**改进方案**：

```python
# backend/app/services/stock_search_service.py
import pypinyin

class StockSearchService:
    def __init__(self):
        self.stock_cache = {}   # 全量股票数据缓存
        self.pinyin_index = {}  # 拼音前缀索引
        self._build_index()

    def _build_index(self):
        """构建拼音前缀索引 (Aho-Corasick 自动机)"""
        for stock in self._load_all_stocks():
            code = stock['code']
            name = stock['name']
            # 首字母拼音索引
            initials = ''.join([
                p[0][0] for p in pypinyin.pinyin(name, style=pypinyin.FIRST_LETTER)
            ])
            # 全拼索引
            full_pinyin = ''.join([
                p[0] for p in pypinyin.pinyin(name, style=pypinyin.TONE2)
            ])
            self.pinyin_index[initials] = self.pinyin_index.get(initials, []) + [code]
            self.pinyin_index[full_pinyin] = self.pinyin_index.get(full_pinyin, []) + [code]

    def search(self, query: str, limit: int = 20):
        """搜索: 股票代码精确匹配 / 名称模糊 / 拼音前缀"""
        if query.isdigit():  # 代码精确匹配
            return self._exact_code(query)
        # 拼音前缀匹配
        results = self._pinyin_prefix(query)
        # 名称模糊匹配
        results += self._name_fuzzy(query)
        return results[:limit]
```

## 4. 数据层增强

### 4.1 数据源三态管理

**观潮做法**：数据源状态分三级——`normal`（正常）→ `degraded`（降级）→ `fallback`（回退），用户可直观了解数据质量。

**现状**：本项目使用 Tushare Pro + AKShare 双数据源，但无状态管理。数据断流时用户无感知。

**改进方案**：

```python
# backend/app/data/data_source_manager.py — 数据源状态管理
from enum import Enum
import time

class DataSourceStatus(Enum):
    NORMAL = 'normal'      # 数据正常
    DEGRADED = 'degraded'  # 降级 (延迟/部分字段缺失)
    FALLBACK = 'fallback'  # 回退 (主数据源不可用，切后备)
    UNAVAILABLE = 'unavailable'  # 不可用

class DataSourceHealth:
    """数据源健康状态"""
    def __init__(self, name, priority=0):
        self.name = name
        self.priority = priority
        self.status = DataSourceStatus.NORMAL
        self.last_check = time.time()
        self.failures = 0
        self.consecutive_failures = 0
        self.avg_latency_ms = 0
        self.last_error = None

class DataSourceManager:
    """数据源管理器 — 自动检测 + 状态切换 + 降级策略"""
    def __init__(self):
        self.sources = {}  # name -> DataSourceHealth
        self.active_source = None  # 当前活跃数据源
        self.degraded_threshold = 5000   # 5s 延迟触发降级
        self.fallback_threshold = 3      # 3次连续失败触发回退

    def check_health(self):
        """周期健康检查"""
        for name, health in self.sources.items():
            latency = self._ping(name)
            health.avg_latency_ms = latency
            if latency > self.degraded_threshold:
                health.status = DataSourceStatus.DEGRADED
            if health.status == DataSourceStatus.DEGRADED:
                # 尝试自动恢复
                pass

    def get_data(self, endpoint, params):
        """数据获取 — 自动降级与回退"""
        source = self._get_active_source()
        try:
            result = source.fetch(endpoint, params)
            # 成功 → 恢复状态
            self._record_success(source.name)
            return result
        except Exception as e:
            self._record_failure(source.name, e)
            if self._should_fallback(source.name):
                # 自动切换到备用数据源
                return self._fallback(endpoint, params)
            raise

    def get_status_snapshot(self):
        """返回状态快照 (供前端状态组件使用)"""
        return {
            'active': self.active_source,
            'sources': {
                name: {
                    'status': health.status.value,
                    'latency': health.avg_latency_ms,
                    'failures': health.failures,
                }
                for name, health in self.sources.items()
            }
        }
```

**前端联动**：配合 `150-观潮对标-前端UI设计优化借鉴方案.md` 第 4.1 节的 `DataSourceStatus` 状态指示组件，后端新增 `GET /api/v3/data-source/status` 端点供前端轮询。该端点的返回格式需与 150 号方案约定的 DataSourceStatus 三态定义对齐。

### 4.2 数据级联缓存

**观潮做法**：IndexedDB (浏览器端持久化大容量数据) + localStorage (小配置项)。

**现状**：仅 localStorage 用于少量配置项，无前端二级缓存。

**改进方案**：引入 IndexedDB 作为前端数据缓存层。

```javascript
// services/cacheService.js — IndexedDB 缓存服务
const DB_NAME = 'a-stock-cache'
const DB_VERSION = 1
const STORES = {
  klines: 'klines',       // K线数据 (大容量)
  indicators: 'indicators', // 指标计算缓存
  watchlist: 'watchlist',  // 自选股
  drawings: 'drawings',    // 画图数据
  preferences: 'preferences', // 用户偏好
}

class IndexedDBCache {
  async open() {
    return new Promise((resolve, reject) => {
      const request = indexedDB.open(DB_NAME, DB_VERSION)
      request.onupgradeneeded = (e) => {
        const db = e.target.result
        // 建表 + 索引
        if (!db.objectStoreNames.contains(STORES.klines)) {
          const store = db.createObjectStore(STORES.klines, { keyPath: 'cacheKey' })
          store.createIndex('symbol', 'symbol', { unique: false })
          store.createIndex('expires', 'expires', { unique: false })
        }
        // ... 其他 Store
      }
      request.onsuccess = () => resolve(request.result)
      request.onerror = () => reject(request.error)
    })
  }

  async set(store, cacheKey, data, ttlSeconds = 300) {
    const db = await this.open()
    const tx = db.transaction(store, 'readwrite')
    tx.objectStore(store).put({
      cacheKey,
      data,
      expires: Date.now() + ttlSeconds * 1000,
    })
  }

  async get(store, cacheKey) {
    const db = await this.open()
    const tx = db.transaction(store, 'readonly')
    const result = await tx.objectStore(store).get(cacheKey)
    if (!result) return null
    if (result.expires < Date.now()) {
      this.delete(store, cacheKey)  // 惰性过期
      return null
    }
    return result.data
  }
}
```

### 4.4 数据版本迁移

**观潮做法**：`SchemaMigration` 类处理数据版本升级。

**现状**：无数据版本管理，用户数据格式变更时需要手动清理。

**改进方案**：引入轻量级 Schema Migration 系统。

```javascript
// services/schemaMigration.js — 数据版本迁移
const MIGRATIONS = {
  1: (db) => {
    // v1 → v2: 添加 expires 索引
    const store = db.createObjectStore('klines', { keyPath: 'cacheKey' })
    store.createIndex('expires', 'expires', { unique: false })
  },
  2: (db) => {
    // v2 → v3: 添加画图表
    db.createObjectStore('drawings', { keyPath: 'id' })
  },
}

class SchemaMigration {
  static async run() {
    const version = localStorage.getItem('schema_version') || 0
    for (let v = version + 1; v <= MIGRATION_TARGET; v++) {
      if (MIGRATIONS[v]) {
        await MIGRATIONS[v](await openDB())
        localStorage.setItem('schema_version', v)
      }
    }
  }
}
```

### 5.1 WebSocket 连接管理增强

**观潮参考**：WebSocket 重连机制 + 连接状态指示。

**现状**：`socketService.js` 基础 WebSocket 实现，重连能力有限。

**改进方案**：

> **与 153 号方案的集成**：本 WebSocket 增强为 `153-观潮对标-AI能力深度分析与升级方案.md` §4.3 AiSignalBus 的信号推送提供底层连接保障。升级后的 WebSocket 实例应作为 AiSignalBus 的后端推送通道共享。

```javascript
// services/socketService.js — 增强版 WebSocket 管理
class EnhancedSocketService {
  constructor() {
    this.ws = null
    this.reconnectAttempts = 0
    this.maxReconnect = 10
    this.reconnectDelay = 1000  // 初始 1s, 指数退避
    this.heartbeatInterval = null
    this.listeners = new Map()
    this.status = 'disconnected'  // connected | disconnected | reconnecting
  }

  connect(url) {
    this.ws = new WebSocket(url)
    this.ws.onopen = () => {
      this.status = 'connected'
      this.reconnectAttempts = 0
      this._startHeartbeat()
    }
    this.ws.onclose = () => {
      this.status = 'disconnected'
      this._scheduleReconnect()
    }
    this.ws.onerror = (err) => {
      // 网络错误不立即断开，等待 onclose
    }
  }

  _scheduleReconnect() {
    if (this.reconnectAttempts >= this.maxReconnect) return
    const delay = Math.min(
      30000,
      this.reconnectDelay * Math.pow(1.5, this.reconnectAttempts)
    )
    this.status = 'reconnecting'
    setTimeout(() => {
      this.reconnectAttempts++
      this.connect(this.url)
    }, delay)
  }

  _startHeartbeat() {
    // 30s 心跳
    this.heartbeatInterval = setInterval(() => {
      if (this.ws.readyState === WebSocket.OPEN) {
        this.ws.send(JSON.stringify({ type: 'ping' }))
      }
    }, 30000)
  }
}
```

### 5.3 请求去重与防抖

**观潮做法**：从 `DataService` 实现推断有请求去重机制。

**现状**：无统一请求去重，K 线缩放时可能重复请求。

**改进方案**：

```javascript
// utils/requestDedupe.js — 请求去重
const pendingRequests = new Map()

export async function dedupedRequest(url, params, options = {}) {
  const key = `${url}|${JSON.stringify(params)}`
  const { dedupe = true, cache = false, cacheTTL = 5000 } = options

  // 去重: 相同请求未完成时复用
  if (dedupe && pendingRequests.has(key)) {
    return pendingRequests.get(key)
  }

  const promise = axios.get(url, { params })
  if (dedupe) {
    pendingRequests.set(key, promise)
    promise.finally(() => pendingRequests.delete(key))
  }

  return promise.then(r => r.data)
}
```

### 5.4 定时任务监控

**现状**：`backend/app/scheduler.py` 实现基础定时任务，无健康监控。

**改进方案**：增加任务健康检查 + 自动重启 + 执行统计。

```python
# backend/app/monitors/task_health_monitor.py
class TaskHealthMonitor:
    """定时任务健康监控 (借鉴观潮的健康模式)"""
    def __init__(self):
        self.tasks = {}
        self.alert_threshold = 3  # 连续3次失败警报

    def record_execution(self, task_name, success, duration_ms, error=None):
        if task_name not in self.tasks:
            self.tasks[task_name] = {
                'total': 0, 'success': 0, 'failures': 0,
                'consecutive_failures': 0,
                'avg_duration_ms': 0,
                'last_execution': None,
                'last_error': None,
            }
        t = self.tasks[task_name]
        t['total'] += 1
        t['last_execution'] = datetime.now()

        if success:
            t['success'] += 1
            t['consecutive_failures'] = 0
        else:
            t['failures'] += 1
            t['consecutive_failures'] += 1
            t['last_error'] = error

        # 加权移动平均执行时长
        t['avg_duration_ms'] = t['avg_duration_ms'] * 0.7 + duration_ms * 0.3

        # 连续失败告警
        if t['consecutive_failures'] >= self.alert_threshold:
            self._send_alert(task_name, t)
```
