# 系统数据管道整体核查与能力评估报告

## 一、概述

本报告对 A股股票分析决策支持系统的**数据获取、存储、流转、使用**四大环节进行全面核查，评估系统整体能力与稳定性，识别风险点并提出改进建议。

### 核查范围

| 维度 | 涵盖内容 |
|------|---------|
| 数据源 | Tushare Pro / AkShare / QMT / 用户输入 |
| 存储层 | PostgreSQL / DuckDB / Redis / localStorage / Vuex |
| 流转层 | 后端路由(14个文件/128+端点) / 前端服务(6个文件) / Nginx代理 |
| 消费层 | 前端9个页面 / 策略引擎 / SignalGenerator / Screener |
| 部署层 | Docker Compose (5个容器) / 健康检查 / 环境变量 |

### 3.3 数据一致性问题

| 问题 | 影响 | 说明 |
|------|------|------|
| **双写一致性** | ⚠ 中 | DataManager 同时写入 DuckDB + PostgreSQL，非事务性写入，异常时可能不一致 |
| **缓存穿透** | ⚠ 低 | 首次请求无缓存时直接查Tushare，在高并发下可能重复请求 |
| **日线聚合** | ⚠ 低 | 周线/月线优先从本地日线聚合，与Tushare直接获取的周线可能存在差异 |
| **DuckDB锁** | ⚠ 中 | 单连接模式，多个worker可能争锁 |

### 6.3 现有风险

| # | 风险 | 级别 | 影响 | 建议 |
|---|------|------|------|------|
| 1 | **Tushare Token 明文存储** | 🟠 中 | Token泄露可能被滥用 | 用Docker Secrets或环境变量注入 |
| 2 | **DuckDB 并发锁** | 🟠 中 | 多进程写入时可能锁住 | 使用读写分离或连接池 |
| 3 | **前端 Docker 未启动** | 🟠 中 | 生产环境不可用 | 构建并启动 frontend 容器 |
| 4 | **AI 分析 Mock 未实现** | 🟡 低 | 页面功能不可用 | 接入 DeepSeek/LM Studio |
| 5 | **无缓存失效策略** | 🟡 低 | DuckDB/Tushare数据可能过期 | 添加 TTL + 定时刷新 |
| 6 | **部分路由无异常装饰器** | 🟡 低 | 异常时返回500无信息 | 补充 `@handle_exceptions` |
| 7 | **CORS 全开放** | 🟡 低 | 生产环境安全风险 | 限制来源域名 |
| 8 | **策略模板/报告 DB 空** | 🟢 低 | 页面功能空白 | 初始化默认数据 |
| 9 | **日线聚合算法差异** | 🟢 低 | 周线/月线与Tushare可能偏差 | 聚合方式后期校准 |
| 10 | **全量筛选性能** | 🟢 低 | 5000只股票约30秒+ | 添加分批处理/数据预加载 |

### 6.4 错误处理链

```
[前端]
  Axios拦截器 → 统一错误提示 (message.error)
  → 组件内 catch (console.error)
  → 兜底：显示空状态/空数组

[后端]
  @handle_exceptions 装饰器 (15/128路由已覆盖)
  → 异常类型分类 (ValueError/KeyError/TypeError/DatabaseError)
  → JSON错误响应 {success: false, error, error_type}
  → 日志记录 (traceback)

[剩余未覆盖]
  screener.py 部分路由 ❌ (未使用装饰器)
  realtime.py 部分路由 ❌ (未使用装饰器)
  qmt.py 全部路由 ❌ (未使用装饰器)
```
