## 二、策略框架在IDE中的变化评估

### 2.1 变化评估矩阵

| 维度 | 原IDE（改造前） | 现IDE（Phase 3后） | 变化程度 | 影响评估 |
|------|---------------|------------------|---------|---------|
| **分析对象** | 单只股票技术面 | 单只股票（仍不变） | 无变化 | ⚠️ 框架面向多股票，IDE仍是单股 |
| **K线图表** | 无 | 有（klinecharts + B/S标记 + 指标叠加） | 🔴 大幅增强 | 对第三层单股验证有帮助 |
| **代码编辑** | 无 | 有（CodeMirror 5 + Python模板 + 运行） | 🔴 大幅增强 | 对第三层自定义策略有帮助 |
| **策略信号** | 仅有信号列表 | 自定义+系统信号分组 | 🟡 增强 | 仍缺层间关系展示 |
| **指标选择** | 仅有策略开关 | 指标芯片选择器 | 🟡 增强 | 仍缺融合权重配置 |
| **框架相关UI** | 无 | 无 | 无变化 | ❌ 核心缺失 |

### 2.2 核心缺失分析

| 缺失项 | 后端已有 | 前端缺失 | 影响 |
|--------|---------|---------|------|
| **三层筛选可视化** | `MultiLayerStockScreener` | 无入口、无结果展示 | 用户无法看到选股过程 |
| **信号融合配置** | `SignalFusion`（chip 0.4/chanlun 0.3/factor 0.3） | 权重硬编码 | 无法定制策略偏好 |
| **层间数据流** | 各层独立可调用 | 无 | 无法查看各层结果 |
| **批量选股结果** | 后端可批量处理 | IDE只展示单股 | 无法查看全市场评分 |
| **参数优化入口** | `GridSearchOptimizer` | 无 | 无法调优策略参数 |
| **框架执行报告** | 各框架模块有返回结构 | 无 | 无法直观查看输出 |

### 2.3 结论

**策略框架在修改后的IDE中变化显著，但IDE布局未跟随变化**：

- Phase 3 的增强（K线图表、代码编辑器、B/S标记）主要服务于**单股票的技术分析**，对应框架的**第三层（策略验证层）**
- 但框架的**第一层（风险剔除）**和**第二层（主力识别）** 在IDE中完全没有表现
- **信号融合**的权重配置、**参数优化**的入口、**批量选股**的结果展示全部缺失

## 三、建议的布局方案

### 3.1 设计原则

1. **两模式并存**：单股分析模式（现有）和框架选股模式（新增），互不干扰
2. **不破坏现有布局**：框架选股作为独立视图，不影响现有的图表+编辑器布局
3. **重用现有组件**：框架选股模式下选中某只股票后，可点击进入单股分析查看详情
4. **循序渐进**：先实现最核心的三层可视化和信号融合配置

### 3.2 整体架构

```
指标IDE → [标签页切换]
  ├── 📊 单股分析（当前布局）
  │   └── 代码编辑器 | K线图表+指标 | 策略信号
  │
  └── 🎯 框架选股（新增）
      └── 筛选流程 | 选股结果 | 信号融合配置
```

### 3.3 框架选股模式（新增）——三栏布局

```
┌──────────────────┬──────────────────────────────┬──────────────────┐
│  筛选流程面板     │  选股结果面板                 │  信号融合配置     │
│  (Pipeline)      │  (Results)                  │  (Fusion Config)  │
│                  │                              │                  │
│  🌐 全市场 (5000) │  ┌────────────────────────┐ │  ┌──────────────┐│
│      ↓           │  │ 排名 │ 代码 │ 名称 │ 评分 │ │  │  策略权重    ││
│  🟢 L1: 风险剔除  │  │ 1  │ 600519│ 茅台 │ 92 │ │  │  筹码: [===] ││
│      ↓           │  │ 2  │ 000858│ 五粮液│ 88 │ │  │  缠论: [==-] ││
│  🟢 L2: 主力识别  │  │ 3  │ 002415│ 海康 │ 85 │ │  │  因子: [==-] ││
│      ↓           │  │ ...                     │ │  ├──────────────┤│
```

### 3.4 各面板详细设计

#### 3.4.1 筛选流程面板（左，280px）

| 元素 | 设计 | 说明 |
|------|------|------|
| **层节点** | 垂直流程图，每层一个卡片 | 显示当前层名称、筛选条件摘要 |
| **数量标签** | 每层右侧显示通过股票数 | 从初始~5000逐步减少到~20 |
| **层级状态** | 颜色区分（🟢通过/🔵运行中/⏳待执行） | 执行时动画展示数据流动 |
| **执行按钮** | 面板顶部 `▶ 执行筛选` | 触发后端 `MultiLayerStockScreener` |
| **参数设置** | 齿轮图标，弹出抽屉 | 配置各层参数阈值 |

**交互流程**：
1. 点击"执行筛选" → 后端逐层调用筛选器
2. 每层完成后，数量标签实时更新
3. 全部完成后，"选股结果"面板更新
4. 点击某只股票 → 跳转"单股分析"模式，自动加载该股票的K线+信号

#### 3.4.2 选股结果面板（中，flex）

| 元素 | 设计 | 说明 |
|------|------|------|

| **排序** | 默认按综合评分降序 | 可切换按ASR/集中度/阶段排序 |
| **行点击** | 选中后在右侧显示详情 | 双击跳转单股分析 |
| **分页** | 每页20条 | 支持快速翻页 |
| **导出** | 工具栏"导出CSV" | 下载选股结果 |

#### 3.4.3 信号融合配置面板（右，300px）

| 元素 | 设计 | 说明 |
|------|------|------|
| **权重滑块** | 筹码/缠论/因子 三组 Slider | 联动确保总和=1.0，实时显示百分比 |
| **阶段加分** | 复选框 + 加分值输入 | 建仓期+2、洗盘末期+1 等 |
| **评分预览** | 选中某股票时显示评分明细 | ASR评分×0.3 + 集中度×0.2 + ... |
| **保存配置** | "保存为默认"按钮 | 持久化到后端 |

### 3.5 关键交互——模式切换

```
📊 单股分析  |  🎯 框架选股

[标签页切换，URL参数?mode=single|framework]
```

| 状态 | 切换处理 |
|------|---------|
| 当前股票代码 | 切换模式时保留，框架模式选中股票后回传给单股模式 |

| 图表缩放/位置 | 单股模式独有状态，切换到框架时释放 |

| 冲突点 | 解决方式 |
|--------|---------|
| 两模式共用 `currentTsCode` | 统一使用，框架选中股票后赋值给 `currentTsCode` |
| CodeEditor 的 stockCode prop | 跟随 `currentTsCode` 自动更新 |
| 信号面板数据源 | 框架模式使用 `SignalFusion` 结果，单股模式使用 `/api/v2/strategy/outputs` |
| 布局CSS冲突 | 框架模式用独立的 `v-show` 容器，CSS scope scoped |

## 五、用户自选股集成策略

### 5.1 现状分析

当前系统的自选股功能存在三个断点，导致单股分析模块（indicator-ide）无法接入用户的个人自选股：

| 断点 | 当前状态 | 影响 |
|------|---------|------|
| **路由参数未读取** | 自选页 `goToChart()` 传递 `{ query: { symbol } }`，但 indicator-ide 的 `mounted()` 未读取 `$route.query.symbol`，硬编码默认 `600519.SH` | 从自选页点"分析"永远跳转到茅台 |
| **无自选股快捷入口** | indicator-ide 顶部搜索框仅支持全市场模糊搜索，无"从自选股选择"按钮 | 用户无法在分析时快速切换到自己关注的股票 |
| **自选股无持久化** | 自选页 `watchlist` 数据仅存于前端内存数组，页面刷新即丢失 | 跨页面无法共享自选股状态 |

### 5.2 设计原则

1. **最小改动用最大效果**：优先打通断点，不做过度设计
2. **增量新增，不破坏现有逻辑**：所有修改向后兼容
3. **分阶段推进**：关键断点立即修复，完整自选体系后续独立实施

### 5.3 第一阶段：断点修复（本方案中实施）

#### 5.3.1 修复1：`mounted()` 读取路由参数

**文件**：`src/views/indicator-ide/index.vue`

```diff
- currentTsCode: '600519.SH',
+ currentTsCode: this.$route.query.symbol || '600519.SH',
```

改造后：从 watchlist 页点击"分析" → 路由参数 `symbol=000001.SZ` → indicator-ide 默认加载该股票 → 正常。

#### 5.3.2 修复2：单股分析顶部添加"从自选股选择"

在 toolbar 搜索框旁增加一个下拉选择器：

```
┌───────────────────────────────────────────────────┐
│  ⭐ 自选  ▾  │  [搜索股票...]  │  日线 周线 月线  │
│  ─────────── │                                   │
│  ✓ 贵州茅台  │                                   │
│    宁德时代  │                                   │
│    招商银行  │                                   │
│    ...       │                                   │
└───────────────────────────────────────────────────┘
```

**数据来源**：使用 `localStorage` 作为临时共享存储。
- 自选页（watchlist）在增删自选股时同步写入 `localStorage.setItem('user_watchlist', JSON.stringify(list))`
- indicator-ide 在 `mounted()` 时读取 `localStorage.getItem('user_watchlist')`

**设计要点**：
- 下拉项显示：股票名称 + 代码
- 选中后：`currentTsCode` 切换，自动重新加载K线和信号
- 选中项后自动关联到右侧策略信号面板

#### 5.3.3 修复3：watchlist 页写入 localStorage

**文件**：`src/views/watchlist/index.vue`

- 自选数组有变更时（新增/删除/初始化），同步调用 `syncWatchlistToStorage()`
- 改造极小，仅增加一个工具方法 + 在 watch 和 methods 中调用

```javascript
syncWatchlistToStorage() {
  try {
    const list = this.watchlist.map(s => ({ symbol: s.symbol, name: s.name }))
    localStorage.setItem('user_watchlist', JSON.stringify(list))
  } catch (e) { /* ignore */ }
}
```

### 5.4 第二阶段：独立实施（本方案不涵盖，后续任务）

| 功能 | 说明 | 预估工时 |
|------|------|---------|
| **自选股后端 API** | 新增 watchlist 表 + CRUD 路由 | 2d |
| **Vuex 状态共享** | 将自选股提升为全局 store 模块 | 1d |
| **批量导入/导出** | 支持文本批量粘贴、CSV 导入 | 1d |
| **自选分组** | 按板块/策略分组自选股 | 2d |
| **同步机制** | localStorage ↔ 后端 ↔ Socket.IO 实时 | 1d |

### 5.5 局限性说明

第一阶段方案使用 `localStorage` 作为临时共享方案，存在以下局限：

| 局限 | 影响 | 后续方案 |
|------|------|---------|
| 仅存储基础信息（symbol + name） | 下拉列表无涨跌幅/价格等实时数据 | 第二阶段接入后端 API 后可完备 |
| 同一浏览器不同 tab 间不会自动同步 | 需手动刷新页面才能看到更新 | 第二阶段使用 Vuex + WebSocket |
| 浏览器清理缓存后会丢失 | 重新添加即可 | 第二阶段使用数据库持久化 |

## 七、风险与备选方案

| 风险 | 影响 | 应对 |
|------|------|------|
| 后端筛选器API尚未暴露为HTTP接口 | 前端无法调用 | 先创建 `screenerService.js` 定义接口契约，后端再补充路由 |
| 框架模式下信号融合需要真实数据 | 展示可能为空 | 使用Mock数据先验证UI流程，再加真实数据 |
| 两模式切换导致状态混乱 | 使用体验差 | 严格管理状态生命周期：模式切换时 `beforeRouteLeave` 保存关键状态 |
| 布局过于拥挤 | 用户不适 | 框架模式面板宽度可调（splitter），右侧面板可折叠 |
| localStorage 自选股不同步 | 数据不一致 | 明确告知此为临时方案，最终依赖后端持久化 |
