# 股票现状识别与输出的最优方式定义

**基于 LLM Wiki 知识库全体系深度梳理，先定义最优标准，再对照现有能力**

---

## 前言：方法论说明

本文档不讨论数据可得性、不讨论现有代码能力、不讨论实施难度。**只讨论一个问题**：

> 基于当前知识库中已建立的认知体系，一个股票"现状"的最优识别方式和输出方式应该是什么？

之后再来回答"我们距离这个最优标准有多远"。

---

## 一、知识库中关于"股票现状"的完整架构

### 1.1 核心发现：知识库已给出完整的五层抽象架构

从原始数据到最终现状输出，知识库定义了一条**不可逆的五层抽象链**（`五层抽象架构（K线分析）.md`）：

```
观测层 → 结构层 → 度量层 → 情绪层 → 信号层
```

每一层是对下一层的抽象与结构化。**任何一层都不能跳过**。

| 层级 | 输入 | 处理 | 输出 | 对应知识域 |
|:----|:-----|:-----|:-----|:----------|
| ① 观测层 | 原始K线(OHLCV) | 包含处理→标准序列 | 无包含的K线序列 | 缠论基础 |
| ② 结构层 | 标准K线序列 | 分形→笔→线段→中枢→走势类型 | 走势结构图(中枢区间、买卖点位置) | 缠论 |
| ③ 度量层 | 标准K线+走势结构 | 实体/影线比、量价配合度、筹码穿透力 | 量化度量指标 | 量价分析 |
| ④ 情绪层 | 量化度量+走势结构 | K线形态→情绪状态映射 | 情绪指数/情绪周期阶段 | 情绪周期 |
| ⑤ 信号层 | 情绪指数+量化度量+走势结构 | 多因子融合→动态权重决策 | 最终交易信号 | 多因子决策 |

**核心约束**：信号层（⑤）不可以直接从观测层（①）跳到。跳过中间层的策略必然存在"识别盲区"——看到的不是结构化的现状，而是原始数据的表象。

### 1.2 一套股票现状 = 一个五维特征向量

在五层抽象的基础上，知识库进一步定义了"现状"是什么样的——通过 `五维统一分类框架.md`：

```
股票现状(t) = {
    ① 形态本体:  "启明星" | "多方炮" | ... （形态类型）
    ② 位置特征:  "中枢下沿附近" | "筹码主峰上方" | ... （结构内位置）
    ③ 量价关系:  "放量阳线确认" | "缩量下跌衰竭" | ... （量验证）
    ④ 走势结构:  "背驰段" | "中枢内" | "第三类买点附近" | ... （结构定位）
    ⑤ 情绪周期:  "冰点期" | "高潮期" | ... （情绪状态）
}
```

**关键设计**：这五个维度不只是"五个标签"。它们是五个相互独立的信息通道，任何单一通道的缺陷都会导致现状识别不完整。

### 1.3 三层信号结构的角色分工

知识库 `K线形态：跨策略融合的统一桥梁.md` 进一步定义了各策略域的角色分工：

```
确认层（慢线） ← 多因子选股  → 基本面/趋势背景，为现状提供"大方向"
执行层（快线） ← K线形态+量价  → 精确定位当前状态的具体类型和强度
过滤层（情绪+筹码） ← 情绪周期+筹码分布 → 信号可信度校验
```

这三层必须配合作业。**单一策略域独立运行为什么不够？** 因为知识库中有明确的约束：

- 缠论："包含处理后的K线与原始K线**混用**可能导致信号混乱"（`跨策略融合桥梁.md:102`）
- 量价："形态是'果'而非'因'，**交易者效应可能导致形态失效**"（同上:103）
- "同一根K线在**不同时间维度上**的形态和结构意义完全不同"（同上:104）

### 1.4 融合的冲突调和机制

多策略协同必然产生信号矛盾。知识库定义了四级裁定机制：

```
① 结构优先：缠论结构关键节点（如第三类买点），结构权威性高于情绪判断
② 位置优先：重要支撑/压力位（如筹码主峰），位置权威性高于形态本身
③ 量价验证：任何信号最终需通过极端量价关系确认
④ 嵌套处理：大级别情绪周期中，小级别看多信号有效性降低
```

## 二、最优现状识别 = 五层架构的完整落地

### 2.1 最优方式的定义

**最优现状识别** = 对一只股票在任何时刻 t，通过五层抽象架构，生成一个五维特征向量，且每个维度的输出都经过了该维度专有知识域的校验。

不是"缠论给一个结果+量价给一个结果+多因子给一个结果然后拼在一起"。**而是**：

```
① 观测层统一输入：原始K线 → 缠论包含处理 → 标准K线序列（作为所有上层分析的共同基础）

② 结构层（缠论域）：
   标准K线序列 → 分形→笔→线段→中枢→走势类型
   产出：{走势方向, 中枢位置, 中枢类型, 背驰状态, 买卖点触发}
   校验：多级别联立（日线+60分钟中枢对比）

③ 度量层（量价域）：
   标准K线序列 + 走势结构 → 
       波段阶段判定 → 成交量状态分析 → 量价关系分析 → 增强形态检测
   产出：{波段阶段, 价格分位, 量价形态(VP-1~VP-9), 增强形态集, 主力阶段推断, 多空动量}

④ 情绪层（情绪域）：
   量价度量 + 走势结构 → 
       快线(MA20强势股占比/换手率/涨跌停比/RSI) +
       慢线(ERP/股债位置差/融资余额) → 
       四象限情绪判断
   产出：{情绪周期阶段(冰点/复苏/高潮/退潮), 快慢线共振状态}

⑤ 信号层（多因子决策域）：
   情绪指数 + 量化度量 + 走势结构 →
       形态量化中枢 = α×缠论中枢 + β×量价形态评分 + γ×情绪状态因子 + δ×筹码集中度因子
   产出：{综合得分, 动态权重, 仓位建议}
```

### 2.3 最优融合方式：形态量化中枢

知识库给出了五维现状的融合公式（`xing-tai-liang-hua-zhong-shu.md`）：

```
形态量化中枢 = α × 缠论中枢（结构性权重）
              + β × 量价形态评分
              + γ × 情绪状态因子
              + δ × 筹码集中度因子
```

| 参数 | 含义 | 确定方法 |
|:----|:-----|:--------|
| α | 缠论结构权重 | 中枢内部低（假突破多）、离开段高（趋势延续） |
| β | 量价形态权重 | 与成交量验证程度正相关 |
| γ | 情绪状态权重 | 冰点/高潮期高（极端情绪下信号更可靠） |
| δ | 筹码集中度权重 | 集中度高时增权（主力控盘信号更可靠） |

权重确定方法：

| 方法 | 适用场景 | 优缺点 |
|:----|:---------|:------|
| 经验法 | 初期框架建设 | 快速但主观 |
| 优化法（网格搜索/贝叶斯） | 回测阶段 | 需大量历史数据 |
| 自适应法 | 运行中动态调整 | 实现后效果最佳 |
| 机器学习法（GBDT） | 后期精细化 | 需防控过拟合 |

### 2.4 补充维度：知识库中已建立的实用分析框架

除五维核心框架外，知识库还定义了多个**可直接工程化的分析工具**，它们为五维中的特定维度提供精确的量化实现方法。

#### 2.4.1 波浪理论：阶段定位工具

`langli-mai-dian.md` 定义了六种买点的风险收益特征，为"位置特征"维度提供补充框架：

| 买点位置 | 性质 | 优势 | 劣势 |
|:--------|:----|:-----|:-----|
| C浪末端 | 左侧抄底 | 利润空间最大 | 可靠性低，可能接飞刀 |
| 1浪启动 | 趋势反转初期 | 可靠性尚可，上涨空间大 | 可能只是单浪反弹 |
| 2浪末端 | 大趋势中的逆势回调 | 即将进入最强3浪 | 存在跌破1浪低点风险 |
| 3浪启动 | 顺势最强浪 | 成功率最高 | 有时无回调买点 |
| 4浪末端 | 顺势回调买入 | 上升趋势明确 | 3浪天量可能导致5浪衰竭 |
| 5浪启动 | 末升段，最凶猛 | 爆发力最强 | 量价异常=顶部信号，长线不宜加仓 |

波浪与量价结合的阶段判断规则：
- 3浪成交量通常最大（量价同步确认）
- 3浪和5浪常见量价背离（5浪新高但量低于3浪）
- 2浪和4浪回调缩量（健康的调整特征）

#### 2.4.2 量托/价托：量价维度的精确信号

`liangtuo-liang-ya.md` 和 `jiatuo-yujia-ya.md` 定义了均量线和均线的三角形交叉信号：

**量托（做多信号）**：
- MA5量 > MA10量 金叉 → MA20量走平 → MA5、MA10上穿MA20形成封闭三角形
- 量托领先于价托（量在价先）

**量压（做空信号）**：
- MA5量 < MA10量 死叉 → MA20量走平 → MA5、MA10下穿MA20形成封闭三角形

**价托（做多信号）**：
- MA5上穿MA10金叉 + 价格站上MA20 + MA20由跌转平 + MA5/MA10上穿MA20形成三角形
- 低位形成=入场信号
- MACD底背离 + 价托 = 高效买点

**价压（做空信号）**：
- 高位形成=出场信号
- MACD顶背离 + 价压 = 高效卖点

#### 2.4.3 多方炮/空方炮：K线组合的强度分级

`duofang-pao-kongfang-pao.md` 定义了四种强度等级，**强度取决于位置**：

**多方炮四级**：

#### 2.4.5 123法则：趋势反转的系统性确认

1. 价格突破趋势线（原趋势结束）
2. 价格不再创新低/新高（趋势可能反转或横盘）
3. 反向推动浪形成（价格大概率沿新方向运行）

#### 2.4.6 背离确认套装：多重验证机制

`beili-xingtai.md` 定义了背离信号的完整确认体系：

- **层级效应**：长周期上升趋势顶背离 > 短周期；长周期下降趋势底背离 > 短周期
- **叠加效应**：当市场周期超过观测周期时，出现二次甚至三次背离。**第二次背离通常产生最优买卖点**
- **确认工具包**：趋势线/通道 + 10日均线（股票"生命线"） + K线反转形态 + 价格反转形态（头肩、双顶底） + 成交量确认 + 多指标互验

### 3.3 从数据到叙述的链路

最优的"现状输出"文本是如何产生的？

```
五维特征向量 → 形态量化中枢(融合评分) → 结构化信号 → to_narrative()
```

**不是**：原始数据 → LLM AI → 文本（跳过了中间所有结构化层）

**而是**：原始数据 → 五层抽象 → 五维特征向量 → 形态量化中枢评分 → 冲突调和 → \(\{\)确定性结构化输出\(\}\) → 可选：LLM AI润色（增强可读性但不改变结构化结论）

这一步非常关键：**AI的角色是润色叙事（提升可读性），不是做判断（信号来自结构化分析）**。知识库 `关键交易日识别框架` 中对此有硬约束："PBO < 5% 才认为信号有效"——如果AI直接做判断跳过了统计验证，就无法满足这个约束。

### 3.4 补充说明：策略输出应按"维度"重组而非按"策略来源"组织

**当前问题**：系统的前端展示（5个现状卡）是按"哪个策略产生的"组织的（缠论卡、量价卡、多因子卡、筹码卡、BOCIASI卡）。用户要了解"走势结构"需要跨卡拼凑。

**最优输出方式**：按"输出维度"重组，每个维度从多个策略获取信息：


## 五、从最优标准倒推实施路径

### 5.1 前提性问题：是否需要"最优"？

知识库 `guo-du-rong-he-feng-xian.md` 给出了一个重要警告：

> "少即是多：优先使用逻辑简单的融合框架，避免复杂组合。业务逻辑优先：有金融逻辑的融合规则比统计显著的规则更可靠。"

这意味着**"最优"不是"最复杂"**。从我们的实际情况出发，最优路径是：

1. **确保每条独立通道都准确**（单一策略的现状识别先过关）
2. **逐步增加融合维度**（一次加一个，每个都经过CSCV验证）
3. **业务逻辑驱动融合**（不是数据驱动）

### 5.2 分阶段的"最优"目标

#### 阶段一：独立通道达标（使每条策略能自主输出准确的现状描述）

**量价策略**（基础最好的）：
- 独立输出完整的五维中维度①③的现状描述 ✅ 已接近
- 仅需扩展50种形态至50%（P2）+ 对接换手率（已有`get_best_volume_series`）
- 可直接产出 `narrative` 文本（`_build_evidence()` 已经有完整中文链）

**缠论策略**（结构最核心的）：
- 独立输出完整的五维中维度②④的现状描述 ⚠️ 缺背驰MACD
- **首要校准**：`DivergenceDetector` 加入MACD面积比较
- 次要增强：线段特征序列算法（`SegmentAnalyzer.build` 目前简化了）

**情绪层**（当前完全缺失的）：
- 这是整个系统最深的缺口
- 但知识库告诉我们：情绪择时可以很轻量（BOCIASI的4+4指标即可起步）
- 快线四指标（MA20强势股占比/换手率/涨跌停比/RSI）均可用现有数据计算
- 慢线四指标（ERP/股债位置差/股债收益差/融资余额）需额外数据源

**多因子层**（当前最虚的）：
- `_compute_factor_signal` 的4个内联因子需要替换为 FactorRegistry
- 从500+因子中筛选5-15个A股有效因子子集（参考 `a-gu-wu-da-you-xiao-yin-zi.md`：市场/税收/市盈率/规模/速动）

按 `guanjian-jiaoyiri-shibe.md` 的框架：
1. CSCV计算过拟合概率（PBO < 5% → 有效）
2. Fama-MacBeth检验因子显著性
3. 参数平原分析（验证稳健性而非最优性）

## 六、核心结论

### 6.1 什么是"有效的现状识别"？

基于知识库体系，"有效的现状识别"必须满足三个条件：

1. **五层抽象链完整**：不能跳层。跳层=识别盲区。
2. **五维输出完备**：每个维度都有对应知识域的校验，缺一维=现状不完整。
3. **确定性优于模糊性**：结构化分析产生结构化输出，AI的作用是润色叙事而不是替代判断。

### 6.2 当前距离最优的差距

| 差距类型 | 项数 | 说明 |
|:--------|:----|:-----|
| 数据输入层面 | 3项 | 情绪指标、筹码分布、多周期K线 |
| 算法精度层面 | 3项 | 缠论背驰MACD、线段特征序列、多因子接入FactorRegistry |
| 架构缺失层面 | 2项 | **情绪层**全缺、**融合中枢**全缺 |
| 校验层面 | 2项 | CSCV过拟合检验、参数平原分析 |

### 6.3 最有性价比的第一步

> **先让量价策略独立输出完整的现状描述（它已经接近），缠论补齐背驰MACD（它的堵点最明确），然后先做二维融合（缠论+量价的结构化因子）。**

这样做的好处：
- 不引入新数据依赖，全部可基于现有OHLCV实现
- 从两个最成熟的通道开始融合，风险最低
- 先验证了融合框架（CSCV检验），再决定是否引入情绪/筹码维度

### 7.5 因子现状识别设计：每个因子能描述什么现状？

#### 7.5.1 核心问题

系统有500+因子的 FactorRegistry，但当前仅5个因子参与计。即使接入所有因子，用户也看不懂0.72这个数字代表了什么。**解决方案**：每个因子新增 `describe_status()` 方法，把数值翻译为结构化的现状描述。

#### 7.5.2 BaseFactor 扩展设计

```python
class BaseFactor(ABC):
    """因子基类（扩展）"""
    
    # 新增：语义维度归属（区别于技术分类category）
    semantic_dimension: str = ""  # "量价关系" / "走势结构" / "情绪环境" / "筹码与资金" / "综合风险感知"
    
    @abstractmethod
    def describe_status(self, value: float, context: Optional[Dict] = None) -> Dict:
        """
        将因子当前值翻译为"现状描述"
        
        参数:
            value: 因子当前计算值（原始值，非归一化）
            context: 可选上下文（如近期历史值序列）
            
        返回:
            {
                "dimension": "量价关系",       # 语义维度归属
                "direction": "向上",            # 方向：向上/向下/走平
                "state": "加速",                # 状态：加速/减速/持续/拐点
                "magnitude": "强劲",            # 强度：强劲/温和/微弱
                "description": "20日价格向上加速，动量强劲",   # 完整中文描述
                "key_signal": "价格加速上行"     # 核心信号（一句话）
            }
        """
        pass
```

#### 7.5.3 现有5个因子的现状识别能力

**因子1：ROC(20) — 短期价格动量**
- **语义维度**：量价关系
- **能识别的现状**：短期价格运动方向/速度/强度


### 7.6 DeepSeek V4 叙事引擎

#### 7.6.1 为什么不用固定模板？

前期使用固定模板（`f"当前为{stage}，{pattern}确认"`）是可行的起点，但随着：
- 新策略/因子的不断加入（FactorRegistry 500+）
- 用户自建策略的引入（输出格式未知）
- 已有策略的升级（输出字段可能变化）

固定模板会持续需要维护。更健壮的方式：**结构化数据 → DeepSeek V4 → 自然语言**。

#### 7.6.2 两层叙事架构

**第一层：卡内叙事（结构化数据 → 每卡一段话）**

```
输入: 卡N的sources + summary 结构化数据 (JSON)
处理: DeepSeek V4 + 维度特定prompt
输出: 50-100字的自然语言描述
```

```python
# 卡1的prompt示例（每个维度有独立的prompt模板）
CARD_NARRATIVE_PROMPT = {
    "走势结构": """
你是一个股票走势结构分析助手。以下是一只股票的走势结构分析的结构化数据。
请用一段50-80字的中文描述该股票的当前走势结构状态。
只描述数据中明确的事实，不要添加任何预测或主观判断。
如果数据中存在矛盾，明确指出。
结构化数据: {structured_data}
"""
}
```

**第二层：跨维度仲裁（卡1-5结构化数据 → AI分析建议）**

```
输入: 
  - 卡1-5的全部结构化数据
  - LLM Wiki中相关概念页（如当前情绪阶段的行为建议）
处理: DeepSeek V4 + 仲裁prompt
输出:
  - 跨维度一致性评估
  - 核心矛盾点
  - 情景推演（if X then Y）
  - 基于推演的策略建议
```

```python
ARBITRATION_PROMPT = """
你是一个多维度股票现状分析助手。以下是该股票5个维度的完整分析数据。
请执行以下任务：

2. **核心矛盾**：提炼当前分析中最值得关注的矛盾点（如有）。

3. **不确定因素**：列出当前分析中数据不完整或判断可信度较低的方面。

4. **情景推演**：基于当前现状，推演2-3个可能的发展情景。
   每个情景包括：触发条件、影响的维度、当前趋势延续/变化的可能性。

5. **建议**：基于以上分析，给出当前状态下的策略思路。
   注意：建议应基于现状描述的自然延伸，而非黑箱结论。
   建议应说明"为什么"而非仅说"做什么"。

维度数据: {cards_data}

LLM Wiki知识参考:
{wiki_pages}
"""
```

#### 7.6.3 降级方案

DeepSeek V4 不可用时使用简化的拼接逻辑：

```python
def narrative_fallback(card_data: Dict) -> str:
    """DeepSeek不可用时的卡叙事降级方案"""
    parts = []
    for source_name, source in card_data.get('sources', {}).items():
        for key, value in source.items():
            if isinstance(value, str) and len(value) > 3:
                parts.append(value)
    return '，'.join(parts[:6]) if parts else "数据暂不可用"
```

#### 7.6.4 架构优势

| 问题 | 模板方案 | DeepSeek方案 |
|:-----|:---------|:------------|
| 新策略增加 | 需新增模板 | 自动处理（结构化数据格式一致） |
| 新因子增加 | 需新增因子描述模板 | 自动处理（describe_status输出一致） |
| 字段变更 | 模板与实际输出不一致 | 自动适配（prompt读取当前结构化数据） |
| 自然语言质量 | 机器感重 | AI级语言组织 |
| 多维度综合分析 | 无法实现（无综合视角） | 天然支持（卡1-5+LLM Wiki） |
| **稳定性** | 完全可控 | 依赖DeepSeek服务 |

### 7.7 用户自建策略范式

#### 7.7.1 核心约束

用户自建策略（自组合因子、自定义规则策略）要纳入五维输出框架，必须遵循以下约束：

1. **输出格式统一**：遵循 `describe_status()` 接口范式
2. **维度映射**：必须声明属于哪个语义输出维度（走势结构/量价形态/筹码资金/情绪环境/综合）
3. **自然语言描述**：输出包含中文描述（非仅为数值）
4. **不输出评分**：与系统策略一致，不输出评分/置信度

#### 7.7.2 自组合因子

用户从 FactorRegistry 选了 N 个因子配权重后：

```
┌─ 自定义组合"短线动量" ──────────────────────────┐
│                                                   │
│ 【该组合识别的现状】                              │
│  因子构成: ROC(10)+RSI(6)+MOM(20)+VOL_RATIO(5)   │
│                                                   │
│  ROC(10)  → 短期动量加速，强度温和                 │
│  RSI(6)   → 中性偏强，未过热                      │
│  MOM(20)  → 中期动量保持正向                      │
│  VOL_RATIO → 量能活跃，配合良好                    │
│                                                   │
│  → 组合整体指向: 短期+中期动量向上，量价配合      │
│  → 归属维度: 量价关系(贡献) + 走势结构(补充)      │
│  → 与系统对比: 方向一致，但短线敏感度更高         │
│                                                   │
└───────────────────────────────────────────────────┘
```

**后端接口设计**：
```python
class CustomFactorCombination:
    """用户自组合因子"""
    
    def describe_status(self, ts_code: str) -> Dict:
        """
        返回该组合的整体现状描述
        """
        factor_statuses = []
        for factor_config in self.factors:
            value = calculator.calculate_single_factor(...)
            factor = registry.get_factor(factor_config['name'])
            status = factor.describe_status(value, context=...)
            factor_statuses.append(status)
        
        # 聚合为组合整体判断
        directions = [s['direction'] for s in factor_statuses]
        return {
            "组合名称": self.name,
            "因子状态": factor_statuses,
            "组合指向": majority_vote(directions),
            "归属维度": "量价关系（为主）",
            "与系统对比": "方向一致（详见卡5）"
        }
```

#### 7.7.3 自定义规则策略

用户编写了自定义规则（如"MACD底背离+均线回踩MA20不破"）：

```

│                                                   │
│ 【该策略识别的现状】                              │
│  状态: MACD零轴上方死叉，DIF未破零轴              │

│    ✅ 价格回踩MA20未破                            │
│    ✅ DIF拐头向上                                │
│    ⏳ 成交量放大确认（尚未满足）                  │
│                                                   │
│  → 策略判断: 上升趋势回调中，等待量能确认        │
│  → 归属维度: 走势结构(趋势延续/回调)             │
│  → 与系统对比: 方向一致但比系统更激进            │
│    （系统等待第三类买点确认，该策略更早介入）     │
│                                                   │
└───────────────────────────────────────────────────┘
```

**维度映射接口**：
```python
class CustomStrategy:
    """用户自定义策略"""
    
    # 用户必须声明的属性
    name: str = "MACD回踩"
    mapped_dimension: str = "走势结构"       # 属于哪个输出维度
    mapped_subject: str = "趋势延续性判断"   # 在这个维度下的具体主题
    
    def describe_status(self, ts_code: str) -> Dict:
        """
        返回该策略的现状描述
        """
        result = self.compute(ts_code)
        return {
            "策略名称": self.name,
            "归属维度": self.mapped_dimension,
            "判断": result.judgment,         # 看多/看空/中性
            "条件清单": result.conditions,    # 满足/不满足的条件
            "描述": result.narrative,        # 一段中文描述
            "与系统对比": "方向一致/矛盾/不可比"
        }
```

#### 7.8.3 DeepSeek仲裁的心理保证

卡1-5的结构化数据 → DeepSeek + LLM Wiki → 情景化建议，这个链路确保了：

1. **每个建议都有出处** — 可追溯到卡1-5中的具体数据
2. **不确定性透明** — 用户知道哪些判断可靠、哪些有缺口
3. **多情景而非单结论** — 用户自己判断哪个情景更可能发生
4. **可验证** — 后续虚拟实盘可检验各情景的预测准确性
5. **可扩展** — 新策略/因子的输出自动纳入分析，无需调整prompt结构

### 7.10 DeepSeek 幻觉防控体系

#### 7.10.1 问题的本质

引入 DeepSeek V4 做卡内叙事和跨维度仲裁，核心风险是**AI可能生成结构化数据中不存在的信息**。具体表现形式：

| 幻觉类型 | 举例 | 危害程度 |
|:---------|:-----|:---------|
| **数据虚构** | 源数据中枢类型="延伸"，AI输出"中枢即将突破" | 高（误导决策） |
| **推理越界** | 源数据中无波动率信息，AI输出"波动率偏低" | 高 |
| **概念误用** | 把"底分型"说成是"底部确认信号"（分型≠底部） | 中 |
| **遗漏关键信息** | 需描述背驰但AI没提 | 中（信息不全） |
| **推测包装** | "中枢延伸中"被写成"目前处于蓄力阶段" | 低（不符合事实优先原则） |

**核心对策**：不依赖AI的自我约束，而是在每个环节设置结构性护栏。

#### 7.10.2 三层护栏体系

```
┌──────────────────────────────────────────────────────────────────┐
│  护栏1：输入约束（prompt层面限制AI发挥空间）                       │
│  → 每层有角色约束prompt                                           │
│  → 结构化数据中嵌入 mandatory 标记（哪些字段必须描述）              │
│  → 明确声明"可描述范围"（严禁超出输入数据）                        │
├──────────────────────────────────────────────────────────────────┤
│  护栏2：知识库锚定（用LLM Wiki约束概念表述的正确性）                 │
│  → 根据当前状态按需检索3-5个相关wiki概念页                          │
│  → AI对任何概念的描述必须基于wiki定义                              │
│  → 仲裁结论强制标注：[数据: 维度/字段] 和 [知识: wiki页面标题]      │
├──────────────────────────────────────────────────────────────────┤
│  护栏3：输出验证（生成后自动化验真）                                │
│  → Layer 1：关键词快速验证（毫秒级，mandatory字段是否都在文本中）    │
│  → Layer 2：claim标注验证（每个数据引用是否指向真实字段）            │
│  → 不通过 → 自动重生成（带违规说明）或降级为fallback拼接            │
└──────────────────────────────────────────────────────────────────┘
```

#### 7.10.3 护栏1：输入约束

**Layer 1（卡内叙事）的角色 prompt**：

```
你是一个数据格式化助手，不是分析师。

你的唯一任务：将输入的结构化数据转化为通顺的自然语言。

约束规则（硬性，必须遵守）：
1. 每一句话必须能对应到输入数据中的至少一个字段
2. 禁止添加输入数据中不存在的信息
3. 禁止使用"可能"、"预计"、"大概率"、"暗示"、"预示"等推测性词汇
4. 禁止添加任何市场评论、操作建议或预测
5. 如果输入中某个字段值为空或null，不要编造，跳过即可

输出要求：
- 每句话控制在15-30字
- 使用简洁的中文陈述句
- 保持客观事实性语气

示例：
❌ "中枢继续延伸，预示后市可能向上突破" —— "预示"是推测
✅ "中枢区间[15.20,16.80]延伸中，已出现2个中枢" —— 数据中的事实
```

**结构化数据中嵌入字段标记**：

```json
{
  "dimension": "走势结构",
  "fields": {
    "中枢":       {"值": "延伸", "来源": "缠论", "mandatory": true},
    "线段":       {"值": "向上", "来源": "缠论", "mandatory": true},
    "背驰":       {"值": "底背驰确认", "来源": "缠论", "mandatory": true},
    "买卖点":     {"值": ["第三类买点"], "来源": "缠论", "mandatory": true},
    "笔段冲突":   {"值": "一致", "来源": "缠论", "mandatory": false},
    "量价波段":   {"值": "多头上涨", "来源": "量价", "mandatory": true},
```

`mandatory: true` 的字段，AI**必须**在输出文本中描述。后续验证也会检查这些字段是否都已出现。

**Layer 2（仲裁推演）的角色 prompt**：

```
你是一个基于知识库的股票分析助手。

你的工作方式：
1. 接收5个维度的结构化数据（卡1-5）
2. 接收LLM Wiki中与当前状态相关的知识页面
3. 仅基于以上信息进行分析

推理规则：
- 每个分析结论必须注明支持的数据来源
  格式：[数据: 维度名称/字段名称]
  示例："走势结构偏多[数据:走势结构/背驰=底背驰]+量价配合[数据:量价形态/量结构=配合]"
- 当引用知识库概念时，必须注明wiki页面
  格式：[知识: wiki页面标题]
  示例："当前符合复苏期特征[知识: BOCIASI快慢线四象限]"
- 如果数据不足以支持某个推理，明确写"数据不足"
- 场景推演必须说明触发条件

禁止：
- 凭空编造wiki中不存在的概念
- 使用"普遍认为"、"众所周知"等无法追溯来源的说法
- 把推测性内容包装成确定性事实
```

#### 7.10.4 护栏2：知识库锚定

**根据当前状态按需检索**，不每次都全量搜索wiki：

```python
def get_wiki_context(cards_data: dict) -> list:
    """根据当前卡片中的关键状态标记，检索相关wiki知识页"""
    search_queries = set()
    
    # 卡1：有背驰 → 检索背驰定义
    if cards_data.get('structure', {}).get('背驰', {}).get('是否确认'):
        search_queries.add("背驰 MACD面积比较 均线相交面积法")
    
    # 卡1：有买卖点 → 检索买卖点行为
    if cards_data.get('structure', {}).get('买卖点', {}).get('买点'):
        search_queries.add("三类买卖点 第三类买点 确认规则")
    
    # 卡4：有情绪阶段 → 检索该阶段的特征和行为建议
    phase = cards_data.get('sentiment', {}).get('情绪阶段')
    if phase:
        search_queries.add(f"BOCIASI {phase} 情绪周期阶段特征")
    
    # 卡5：有冲突 → 检索冲突裁定规则
    if cards_data.get('conflict', {}).get('conflicts'):
        search_queries.add("冲突裁定 四级裁定机制 结构优先")
    
    # 最多检索3条，控制成本和时间
    wiki_results = []
    for q in list(search_queries)[:3]:
        result = wiki_search(q, top_k=1)
        if result:
            wiki_results.append(result[0])
    
    return wiki_results
```

**关键设计**：知识库的作用不是"给AI更多的背景知识"，而是"给AI一个说正确话的参考标准"。当AI需要描述"底背驰"时，不是凭训练数据中的记忆，而是基于wiki中"底背驰=新低+MACD柱面积缩小+笔振幅缩小"的精确定义来组织语言。

#### 7.10.5 护栏3：输出验证

**Layer 1 快速验证**（毫秒级，不调大模型）：

```python
def verify_narrative(text: str, fields: dict) -> dict:
    """
    验证叙事是否覆盖了所有 mandatory 字段
    返回: {"passed": bool, "missing": [field_names]}
    """
    missing = []
    for field_name, field_info in fields.items():
        if field_info.get('mandatory'):
            value = field_info['值']
            value_str = str(value) if not isinstance(value, list) else ' '.join(value)
            if len(value_str) > 2 and value_str[:4] not in text:
                missing.append(field_name)
    
    return {
        "passed": len(missing) == 0,
        "missing": missing
    }
```

**Layer 2 验证**（规则引擎，不调大模型）：

```python
def verify_arbitration(text: str, source_data: dict, wiki_pages: list) -> dict:
    """
    验证仲裁输出中的标注指向真实字段
    """
    violations = []
    import re
    
    # 1. 检查是否有[数据: xxx]标注
    data_citations = re.findall(r'\[数据:\s*([^\]]+)\]', text)
    if not data_citations and len(text) > 20:
        violations.append("仲裁分析未标注数据来源")
    
    # 2. 检查每个数据标注是否指向真实存在的字段
    for citation in data_citations:
        parts = citation.split('/')
        if len(parts) == 2:
            dim, field = parts[0].strip(), parts[1].strip()
            if dim not in source_data:
                violations.append(f"数据标注指向不存在的维度: {dim}")
    
    # 3. 检查wiki引用是否指向真实页面
    wiki_citations = re.findall(r'\[知识:\s*([^\]]+)\]', text)
    wiki_titles = [p.get('title', '') for p in wiki_pages]
    for citation in wiki_citations:
        if not any(citation in title for title in wiki_titles):
            violations.append(f"wiki引用指向未检索的页面: {citation}")
    
    return {
        "passed": len(violations) == 0,
        "violations": violations
    }
```

**兜底：验证不通过后的自修复流程**：

```python
def safe_generate(cards_data: dict, layer: str) -> str:
    """
    带三层防护的生成函数
    """
    wiki_context = get_wiki_context(cards_data) if layer == "arbitration" else []
    
    for attempt in range(2):
        narrative = deepseek_client.chat(
            system=PROMPTS[layer],
            user=json.dumps(cards_data, ensure_ascii=False)
        )
        
        if layer == "narrative":
            result = verify_narrative(narrative, cards_data.get('fields', {}))
        else:
            result = verify_arbitration(narrative, cards_data, wiki_context)
        
        if result['passed']:
            return narrative
        
        if attempt == 0:
            narrative = deepseek_client.chat(
                system=PROMPTS[layer] + f"\n\n上次输出存在以下问题，请修正：{result['violations']}",
                user=json.dumps(cards_data, ensure_ascii=False)
            )
    
    # 重试均失败 → 降级为fallback拼接（宁可生硬，不冒幻觉风险）
    return narrative_fallback(cards_data)
```

**关键设计**：输出验证不是"可选优化"，而是**必选环节**。没有验证通过的文本不允许返回给前端。2次尝试仍失败则降级为 §7.6 的 fallback 拼接—宁可语言生硬，不冒幻觉风险。

#### 7.10.7 各层幻觉防护的投入产出

| 防护层 | 额外成本 | 额外延迟 | 防护效果 |
|:------|:---------|:---------|:--------|
| 护栏1：约束prompt | 0 | 0 | 中（取决于AI遵从度） |
| 护栏1：字段mandatory标记 | 0 | 0 | 高（明确告诉AI必须说什么） |
| 护栏2：wiki按需检索 | 3次搜索API | +100-300ms | 高（概念表述正确性） |
| 护栏3：快速验证 | 0 | +1-5ms | 高（抓信息遗漏和标注缺失） |
| 护栏3：重试机制 | ×1.5-2次生成 | +500-1000ms | 中高（仅首次失败时触发） |
| 前端追溯 | 0 | 0 | 高（用户可自行对照） |

**推荐初始配置**：
- Layer 1（卡叙事）：护栏1（必选）+ 护栏3快速验证（必选）。成本≈0，延迟可忽略
- Layer 2（仲裁推演）：护栏1（必选）+ 护栏2（必选）+ 护栏3验证（必选）+ 重试（可选）。这是高风险层，需要多层约束
- 降级兜底：无论哪层，验证2次失败→fallback拼接。**宁可语言生硬，不冒幻觉风险**

这套体系在**表达灵活性**上远胜固定模板，在**信息准确性**上通过三层护栏+降级兜底可接近模板方案的确定性。核心原则：结构化数据是唯一事实源，DeepSeek只是把事实从JSON翻译成自然语言的翻译器。
