## 一、页面定位与名称变更

### 1.1 变更对照

| 项目 | 原设计（165号） | 新设计 |
|:-----|:---------------|:-------|
| **页面名称** | 预警管理（alert） | **监控通知**（stock-monitor） |
| **心智模型** | "系统运行得怎么样？" | "我想监控什么，系统帮我看" |
| **核心用户** | 系统管理员 | A股普通投资者（非量化背景） |
| **核心功能** | 预警规则 + 通知列表 | 条件选配 → 规则组合 → 监控触发 → 通知管理 |
| **条件来源** | 手动创建规则 | 三通道：手动/自动转换(从策略)/自然语言推荐 |
| **条件复杂度** | 简单价格+指标阈值 | 全量100+条件，从基础到策略信号逐级覆盖 |
| **归类归属** | 管理组（GROUP 4） | 管理组（GROUP 4），保持不变 |

### 1.2 页面一句话定位

> 为用户提供**从条件选择→规则组合→监控触发→通知管理**的完整闭环，让用户不需要懂量化术语，也能对自己的自选股/全市场设置智能化的条件监控。系统定位为**"盯盘代理"**：系统按用户设定的条件持续扫描市场，当某只股票状态匹配时记录并通知用户"请你去看看"，用户收到通知后自主看盘评估决策。**系统不发出"买卖建议"，只报告"条件达成"。**

### 1.3 与其他类似功能的区隔

| 类似功能 | 所属页面 | 区别 |
|:---------|:---------|:-----|
| 实时行情查看 | 自选监控（watchlist） | 用户主动查看表格数据 → 监控通知是系统被动推送 |
| 策略信号展示 | 个股策略分析（indicator-ide） | 聚焦单股票现状分析 → 监控是持续追踪条件变化 |
| 选股筛选 | 选股系统（screener） | 一次性筛选符合条件的股票 → 监控是长期持续的条件等待 |
| 回测验证 | 回测系统（backtest） | 验证策略历史表现 → 监控是当前和未来的实时触发 |

## 二、使用者背景与设计原则

### 2.1 使用者画像

本页面面向**非量化分析师的普通A股投资者**，典型特征：
- 了解基本股票概念（涨跌、成交量、买入卖出）
- 不了解量化术语（夏普率、回撤、IC/IR、动量因子）
- 但知道自己的交易意图（"等它跌到位了反弹"、"等它放量突破"）
- 有自选股列表，希望系统帮自己盯盘

### 2.2 设计原则

| 原则 | 说明 |
|:-----|:------|
| **① 人话优先** | 所有条件名称、说明、参数描述使用自然语言，禁止量化术语 |
| **② 渐进复杂度** | 基础条件（价格、涨幅）无需学习；高级条件（策略信号）配说明? |
| **③ 推荐优先于手动** | 用户不用从零开始——模板推荐、自然语言推荐、策略自动转 |
| **④ 所见即所得** | 设置条件时就能看到参数含义的实时说明 |
| **⑤ 非量化友好** | 用"买入时机提醒"替代"BUY信号触发"，用"价格跌到多少提醒我"替代"价格阈值下破" |
| **⑥ 右侧交易为底层逻辑** | 所有条件规则以趋势确认后操作为设计前提，非抄底逃顶。系统定位为"盯盘代理"——只报告条件达成状态，不发出买卖建议，用户自主决策 |
| **⑦ 用户自配置优先** | 通知渠道/优先级/确认行为等全部由用户自主配置，系统不做强控，避免用户感知受到干扰 |
| **⑧ 跨平台兼容** | 170号方案已确定系统将在 macOS 和 Windows 上独立运行（裸跑 + 嵌入式 Python），通知触达需支持双平台原生通道 |

## 三、条件体系全量整理

### 3.1 条件来源总览

本系统的条件来源于三大通道：

| 来源 | 覆盖数量 | 说明 |
|:-----|:---------|:-----|
| ① 本系统策略引擎产出 | **~100+条件** | 量价策略(50+)/缠论策略(25+)/筹码策略(20+)/情绪周期(8+)/风控(8+) |
| ② 开源项目参考提炼 | **~15条件** | 价格阈值/指标阈值/MA交叉/通道进出/周线级别等 |
| ③ LLM Wiki知识库提取 | **~98条件(含重叠)** | 11大类：买卖信号/多形态共振/风险剔除/情绪周期/技术指标/实战战法/资金风控/因子评估/异常监控/缠论量化/知识图谱 |

去重合并后，**实际可用的唯一条件约100-120条**。

| 3 | 风险剔除与硬性否决 | ⛔ 风险排除 | ~10 |
| 4 | 情绪周期与市场状态 | 🌊 市场情绪 | ~10 |
| 5 | 技术指标条件 | 📊 技术指标 | ~9 |
| 6 | 实战战法 | 🎯 实战技巧 | ~14 |
| 7 | 资金管理与风控 | 💰 资金风控 | ~8 |
| 8 | 因子与量化评估 | 📐 综合评分 | ~6 |
| 9 | 异常监控与预警 | ⚡ 异动提醒 | ~7 |
| 10 | 缠论量化条件 | 🧩 缠论信号 | ~6 |

## 四、条件说明体系与独立生成模块

### 4.1 说明体系结构

每条条件配以下结构化说明字段（由独立生成模块产出，**不嵌入主系统代码**）：

| 字段 | 说明 | 示例 |
|:-----|:-----|:------|
| `user_description` | 一句话解释（不能出现量化术语） | "短期平均价格从下往上穿过长期平均价格，说明最近买的人多了" |
| `how_it_works` | 用普通人能懂的方式解释逻辑 | "计算最近几天的平均价格和更长时间的平均价格，当短期平均从下方穿过长期平均时触发" |
| `parameter_descriptions` | 各参数含义 | short_period: "多少天的价格计算短期平均，数字越小越敏感" |
| `when_to_use` | 适合什么情况 | "适合股票在底部盘整了一段时间，准备启动上涨时" |
| `when_not_to_use` | 什么情况下容易失效 | "股票剧烈波动或连续涨停/跌停时，均线信号会滞后" |
| `example` | 举个实际的股票例子 | "比如西藏矿业在2026年4月，5日均线上穿20日均线，之后上涨了15%" |
| `related_conditions` | 关联条件推荐 | 格兰维尔多头排列 / 格兰维尔死叉卖出 / 均线面积法 |

### 4.2 独立生成模块设计

在不与主系统代码交互的前提下，建立独立文件夹，通过调用 LLM 一次性批量生成说明。

#### 4.2.1 核心选择：Qwen3.5 9B（本地 LM Studio）→ DeepSeek V4（云端备用）

经过评估，**优先使用本地 LM Studio 的 Qwen3.5-9B 模型**，DeepSeek V4 作为备用方案。

| 评估维度 | Qwen3.5 9B (Q6_K) — 本地 | DeepSeek V4 — 云端 |
|:---------|:--------------------------|:-------------------|
| 中文能力 | ✅ **顶级**，阿里系模型中文第一梯队 | ✅ 很好，但偏英文语料 |
| 文本生成质量 | ✅ 生成通俗说明文字绰绰有余 | ✅ |
| Token 成本 | **¥0**（本地运行） | 约30-40万token → 数元 |
| 速度 | ⚠️ 9B 模型约 5-15 tok/s，100条约 15-30分钟 | 快 |
| 独立运行 | ✅ 不占用主系统资源，可后台异步运行 | 需网络 |

**判断：** 生成条件说明是 NLG 任务，不是复杂推理。Qwen3.5 9B 完全胜任，且零成本+独立异步运行。

#### 4.2.2 目录结构

```
_project-setup/
└── condition_descriptions/
    ├── generate_condition_descriptions.py   ← 主脚本（独立运行，支持LM Studio/DeepSeek双引擎）
    ├── conditions_input.json               ← 条件清单（约10条示例，可扩展至100-120条）
    ├── output/
    │   ├── condition_descriptions.json     ← 生成产物，前端直接消费
    │   ├── _checkpoint.json                ← 断点续传检查点（每5条自动保存）
    │   └── _generation_log.txt             ← 运行日志
    ├── scripts/
    │   └── import_to_db.py                 ← 导入 condition_registry 表
    ├── run_generation.sh                   ← 一键运行脚本
    ├── run_validation.sh                   ← 质量验证脚本
    └── README.md
```

**工作流程：**

```
Step 1 — 编写 conditions_input.json
  每条条件只需提供最少的"种子信息"：
  ├── condition_id（唯一标识）
  ├── name（标准名称）
  ├── category_path（归类路径）
  ├── default_params（默认参数及其范围）
  ├── wiki_sources（关联的知识库文件名）
  └── linked_code（关联的代码文件+函数，供人工审核参考）

Step 2 — 运行生成脚本
  A. 检查 LLM 提供者可用性（优先 LM Studio Qwen3.5-9B，备选 DeepSeek V4）
  B. 读取 LLM Wiki 知识库中关联文件的全文
  C. 逐条调用 LLM，组装 prompt：
     条件种子信息 + 知识库文件全文 → 要求输出 7 个说明字段
     → 约束禁止使用量化术语
  D. 每 5 条自动保存检查点（断点续传）
     → 输出 output/condition_descriptions.json
  E. 脚本命令：`bash run_generation.sh`

Step 3 — 质量验证
  A. 运行 `bash run_validation.sh` 自动检查：
     - 字段完整性（5个必填字段不缺）
     - 示例合理性（含股票和年份信息）
     - 量化术语污染检查（不应出现"夏普""因子暴露"等）
  B. 按需手动抽查 5-10 条修正细节，约 0.5 天工作量

建议先跑试验：挑 3 条差异大的条件（一条价格类、一条技术指标类、一条策略信号类）
跑一下，确认生成质量符合预期后全量跑 ~100 条。
```

**技术评估：**

| 方面 | 结论 |
|:-----|:------|
| LM Studio Qwen3.5 9B | ✅ 已完成评估，本地运行可用。`~/.lmstudio/models/lmstudio-community/Qwen3.5-9B-GGUF/Qwen3.5-9B-Q6_K.gguf` 已下载，Q6_K 量化质量可满足 NLG 任务 |
| DeepSeek V4（备用） | ✅ 备选方案。API key 已配置（sk-56efa3...4cfc） |
| LLM Wiki 知识库 | ✅ 知识库文件结构完整、内容详实，足以支撑有据可查的说明生成 |
| 批量成本 | LM Studio: ¥0 / DeepSeek: 约100-120条件 × 每条 prompt 2000-3000 token → 总 token 约30-40万 → 费用可忽略 |
| 质量风险 | 中等。需在 prompt 中明确约束"禁止使用量化术语"。修正约 0.5 天 |
| 断点续传 | ✅ 每5条自动保存 checkpoint，中断后重跑跳过已生成的 |

## 五、条件库管理系统

### 5.1 设计定位

条件库作为底层基础设施，独立于规则引擎运行。全量条件（163条，覆盖13个类别）存储在**后端 SQLite 数据库中**，每条条件标注 data_readiness 状态（ready / design / qmt_needed），前端通过 API 按需查询。

| 查看端口 | 入口位置 | 展示内容 |
|:---------|:---------|:---------|
| 📊 **条件库概览面板** | 监控通知页右上角 ⚙️ → 条件库概览 | 仅统计聚合（总数/就绪分布/Top使用/低频条件） |
| 📋 **条件库管理后台** | 系统管理页（system-management）→ Tab「条件库」 | 全量列表 + CRUD + 使用分析 |

### 5.2 SQLite 存储方案

#### 5.2.1 技术评估

| 评估维度 | 结论 |
|:---------|:------|
| 对现有系统的影响 | **无影响。** 系统已使用 `DATABASE_URL=sqlite:///test.db`（Flask-SQLAlchemy），`condition_registry` 表共 **163条**记录（13个类别），不改变现有 daily_data(610万行)/signals/holdings 等表的查询逻辑 |
| 查询性能 | 163条级别的表，任意条件查询 < 1ms，对现有业务零影响 |
| 存储空间 | < 1MB（含7字段说明JSON） |
| 迁移管理 | 沿用现有 Flask-Migrate，`flask db upgrade` 自动创建新表 |
| 可维护性 | SQLAlchemy ORM 管理，现有数据管理流程不变 |
| **实际数据状态** | ✅ **2026-06-27 全量导入完成。** 163条记录：ready 155 / design 4（知识图谱规则类）/ qmt_needed 4 |

#### 5.2.2 数据模型

| 字段 | 类型 | 说明 |
|:-----|:-----|:------|
| id | INTEGER PK | 自增主键 |
| condition_id | VARCHAR(80) UNIQUE INDEX | 唯一标识，如 `gelanwei_jincha` |
| name | VARCHAR(100) | 标准名称 |
| display_name | VARCHAR(200) | 用户端显示名 |
| category | VARCHAR(50) INDEX | 分类 |
| category_path | JSON | 归类路径 |
| difficulty_level | VARCHAR(20) | 入门\|进阶\|高级\|专业 |
| data_source | VARCHAR(50) | 数据源标识 |
| data_readiness | VARCHAR(20) INDEX | ready\|qmt_needed\|insufficient |
| default_params | JSON | 默认参数及范围 |
| linked_strategies | JSON | 关联策略列表 |
| related_conditions | JSON | 关联条件ID列表 |
| description | JSON | 说明体系（7字段） |
| usage_count | INTEGER default 0 | 被多少条规则引用 |

| notes | TEXT | 内部备注 |
| created_at | DATETIME | 创建时间 |
| updated_at | DATETIME | 更新时间 |

索引设计：
- `condition_id` — 唯一索引（主查询键）
- `category` — 普通索引（分类筛选）
- `data_readiness` — 普通索引（状态筛选）

## 七、监控范围三层级设计

### 7.1 三层范围定义

| 层级 | 名称 | 适用场景 | 示例规则 | 阈值设置方式 |
|:----|:-----|:---------|:---------|:------------|
| 📍 **单股级** | 精确到某只股票 | 基于个股策略分析页设为监控、对某只股票有明确预期 | "西藏矿业价格<11.80时预警" | 可设**具体价格/具体涨跌幅** |
| 📋 **列表级** | 监控自选股/选股结果中的全部股票 | 不想逐只设置，统一条件监控一个股票池 | "自选股中任意一只出现缠论三类买点时预警" | 设**百分比/相对阈值** |
| 🔭 **全市场级** | 全市场5000+异动扫描 | 捕捉市场热点、异动机会 | "全市场涨幅>5%且成交量>2倍均量时预警" | 设**百分比/相对阈值** |

### 7.2 三层联动关系

```
用户创建规则时选择监控范围：
                    │
├── 📍 单股级 ──────┤  直接从自选股中选择 1 只
├── 📋 列表级 ──────┤  选择自选股列表 / 选股结果列表（动态跟随列表变化）
└── 🔭 全市场级 ────┤  无需选择，监控全市场所有股票
```

**重要设计：** 列表级的监控股票来源应支持两种模式：
- **动态跟随**（默认）：规则创建时绑定某个列表（如"自选股列表"），列表增减股票时规则自动生效/取消
- **静态锁定**：规则创建时锁定具体几只股票（如从自选股中勾选了3只），列表变化不影响

### 7.3 单股级特殊设计

单股级规则支持**具体的价格阈值**（区别于其他两级只能用百分比）：

| 参数类型 | 示例 | 交互方式 |
|:---------|:-----|:---------|
| 固定价格 | 价格<11.80 | 输入框输入价格，自动取当前价格±10%作为默认值 |
| 浮动百分比 | 跌幅>3% | 输入百分比 |
| MA相对值 | 价格<MA20 | 选择均线周期 |
| 波幅相对值 | 涨幅>5日均幅×2 | 自动计算 |

### 10.2 交互方式设计

| 操作 | 交互方式 | 说明 |
|:-----|:---------|:------|
| ✅ 选择条件 | Checkbox | 从推荐列表或条件树中选择 |
| ⬆⬇ 排序 | 拖拽手柄 | 调整条件在组合中的顺序 |
| 📦 分AND组 | 将条件拖入AND容器 | AND组内条件全部满足才触发 |
| 📦 分OR组 | 将条件拖入OR容器 | OR组内任一条件满足即触发 |
| ⚙️ 参数设置 | 点击弹出弹窗 | 滑块+输入框适配不同参数类型 |
| ❌ 删除条件 | 点击✕ | 从组合中移除 |
| 🫸 拖到新容器 | 跨容器拖拽 | 从AND组拖到OR组 |

### 10.3 参数便捷设置

参数设置弹窗按参数类型展示不同交互组件：

| 参数类型 | 示例 | 交互组件 |
|:---------|:-----|:---------|
| 数值 | 均线周期5日 | 滑条 + 数字输入框同步 |
| 阈值 | 成交量倍数1.5倍 | 滑条(0.5-5.0步进0.1) |

| 开关 | 启用/禁用 | Toggle开关 |

**滑条默认值设计：** 以策略引擎经验值为基础，给用户留出±50%调整余量，超出范围时文字提示"该参数可能导致信号不准确"。

### 10.4 定时开关设计

每个规则支持组合式时段控制（A+B 双维度），精确到**分**：

#### 10.4.1 双维度时段模型

| 维度 | 名称 | 说明 | 示例 |
|:-----|:-----|:------|:------|
| **A — 每日时段** | 每日重复的开启/关闭窗口 | 每个交易日自动执行，无需人工干预 | "每天 09:30~15:00 运行" |
| **B — 有效期** | 规则的生效日历区间 | 到期自动停用，适合临时规则 | "2026-07-01 09:30 ~ 2026-09-30 15:00" |

两者可叠加使用。典型场景：
> "这条规则在 2026-07-01 ~ 2026-09-30 期间，每个交易日 9:30~15:00 运行"

#### 10.4.2 UI 控件

在规则创建/编辑表单中，"⏰ 定时开关"区块：

```

  ┌─────────────────────────────────────┐
  │ 起始: [2026-07-01 09:30]            │
  │ 截止: [2026-09-30 15:00]            │

  └─────────────────────────────────────┘

当前状态指示（自动计算）:
  ● 运行中（距关闭还有 4h 30min）

  ○ 已过期（有效期已过）
```

### 16.1 三区布局架构

页面采用 **汇总统计条 + 折叠表格 + 卡片列表** 三段递进式布局，信息详细度由简到繁逐步深入，满足不同粒度的监控需求。

```
┌─ ① 汇总统计条（始终可见）──────────────────────────┐
│                             │
├─ ② 折叠表格（默认折叠，按监控范围分组总览）────────────┤

**三者关系：** 表格和卡片是**不同切片视角**，不是同一视角的不同详细度：
- **表格** = 按「监控范围」（单股/多股/市场级）分组的总览，用户快速看三个维度各有哪些规则、整体健康度
- **卡片** = 按「最近触发时间」排序的日常管理视图，用户在这里执行操作
- **弹窗** = 查看完整配置和触发历史

### 16.6 条件重复判定机制

用户在新建规则时，系统自动检测是否已存在相同规则，避免规则冗余。

#### 16.6.1 重复判定维度

**以下三项全部匹配 → 视为重复：**

| 维度 | 匹配规则 | 说明 |
|:-----|:---------|:------|
| ① 条件 ID | `condition_id` 精确匹配 | `macd_jincha` ≠ `kdj_jincha`，不同条件不视为重复 |
| ② 参数配置 | `default_params` 逐字段精确匹配 | 阈值 1.5 倍 vs 2.0 倍为不同规则 |
| ③ 监控范围 | `scope_type` + `scope_value` 精确匹配 | 范围类型（单股/多股/全市场）+ 具体值（股票代码列表/板块ID）完全一致 |

**不纳入判定的维度：**

| 维度 | 理由 |
|:-----|:------|
| 时段配置 A/B | 同一条规则跑不同时段可共存（如日盘+夜盘） |
| 规则名称 | 用户可自定义命名，不影响功能实质 |
| 通知方式 | 同一条规则多路通知是正常需求 |

## 十七、通知触达体系设计

> **核心认知：** 监控通知模块有两大核心——① 监控找到符合条件的股票进行判定触发；② 触发后及时且多维地触达到用户。本节聚焦后者。通知记录管理（收件箱式）作为监控通知页面的子Tab，是用户集中查阅通知存档的板块。

### 17.1 三场景分析与渠道映射

| 场景 | 用户状态 | 可用渠道 | 延迟 | 适用优先级 |
|:----|:---------|:---------|:----|:----------|
| **1. 在系统** | 同一浏览器，但在其他页面 | ①侧栏🔔Badge + 数字；②页面内Toast浮层；③浏览器Tab标题闪烁 | <1s | 全部 |
| **2. 在电脑但未看系统** | 浏览器可能后台，或操作其他软件 | ①浏览器系统通知(Notification API)；②macOS/Win原生通知；③音频提示 | 1-5s | 中/高优先级 |
| **3. 人不在电脑前** | 手机/外出 | ①微信推送（Server酱/PushPlus）；②Bark(iOS推送)；③Telegram Bot | 2-30s | 仅高优先 |

## 十八、规则引擎评估机制设计

### 18.1 设计前提：右侧交易为底层逻辑

**右侧交易**：等趋势确认后动手，不试图抄底逃顶。

| 对比维度 | 左侧交易 | 右侧交易（本系统采用） |
|:---------|:---------|:---------------------|
| 思路 | 预判转折，在最低/最高点操作 | 等趋势成型后再操作 |
| 时效要求 | 秒级（必须在转折点附近） | 分钟到小时级（多根K线确认） |
| 判断依据 | 预判性指标 | 确认性信号（金叉/突破/量价配合） |
| 系统定位 | 高频量化交易 | 分析辅助 + 盯盘代理 |
| 风险 | 容易抄在半山腰 | 确认后成本略高，但趋势真实 |

**结论：** 本系统是A股分析辅助工具，非高频量化交易系统，因此：
- ❌ 不需要秒级实时评估
- ✅ 分钟级到小时级完全够用（右侧信号确认本身需多根K线）
- ⚠️ 实时模式应废弃，改为"与K线周期对齐的周期性扫描"

### 18.2 用户需求图谱 → 五类场景

用户的监控需求按意图目标可分为五类，各类对时效要求不同：

#### 第一类：抓买卖点（时效性高）

### 18.3 系统数据能力评估（5-60分钟扫描可行吗？）

#### 18.3.1 AkShare 数据频率限制核实

经调研和实测确认，`akshare` 目前提供两个实时行情接口，性能差异显著：

| 函数 | 调用方式 | 单次获取量 | 限流实测 | 本系统适用性 |
|:-----|:---------|:----------|:---------|:-----------|
| `stock_bid_ask_em(symbol)` | **逐只调用** | 1只股票盘口数据 | ~60次/分钟/IP；行情高峰期易触发断开 | ❌ 不适合批量扫描——5000只=5000次调用 |
| `stock_zh_a_spot_em()` | **一次调用全市场** | **全部A股**(~5000行×23列) | 每30-60秒1次稳定；更频则触发限流 | ✅ **完美匹配**——5/15/30/60分钟扫描一次，远低于限流阈值 |

**当前代码的问题：** `akshare_provider.py` 使用 `stock_bid_ask_em` 逐只调用，5000只需要5000次API请求，必然触限。

**优化方案：** 改用 `stock_zh_a_spot_em()` 全市场快照 + 内存缓存架构：

```
旧方案（逐只逐API调用）:
  5分钟扫描周期 → 50条规则×50只股票 → 50次API调用 → 很快触限

新方案（全市场快照 + 内存缓存）:
  1次 API 调用 → stock_zh_a_spot_em() 获取全部A股 → 写入内存DataFrame
  → 规则引擎从内存读取 → 零API消耗评估
  → 5分钟周期 = 12次/小时 << 限流阈值

  限流防护层:
  ├── 指数退避重试（失败后1s→2s→4s等待）
  ├── 请求间随机抖动（±20%，避免规律性请求）
  └── 降级到本地最近一次有效快照 + 标注"数据延迟"
```

**延迟说明：** AkShare 是爬虫方案，数据源自东方财富前端页面，存在约 **1-5秒** 准实时延迟。对右侧交易的分钟级扫描完全不是问题——确认一个 MACD 金叉本就需多根 K 线收盘。

**同类项目的常见做法：** 调研同类开源量化项目，对 AkShare 都采用**低频全市场快照+本地缓存**策略，没有逐只高频调用的做法。部分专业项目用 tushare pro 付费版做补充。我们的 5-60 分钟扫描频次是安全区间。

#### 18.3.2 综合能力评估表

| 数据需求 | 方案 | 限频/瓶颈 | 结论 |
|:---------|:-----|:---------|:------|
| **全市场实时行情** | `akshare.stock_zh_a_spot_em()` 全量快照 → 内存缓存 | 实测每30-60秒1次稳定；5分钟扫描=12次/小时，远低阈值 | ✅ **完全可支撑** |
| **分钟K线(5/15/30/60)** | `tushare.pro_bar()` | 200次/分钟配额，一次扫描约10次调用 | ✅ **完全可支撑** |
| **日线数据** | PostgreSQL 610万行 | 本地查询，零网络开销 | ✅ **无需额外数据源** |

**关键结论：**
- ✅ **可以实现分钟级扫描，且不需要 QMT。** 全市场快照 + 本地缓存的架构使 API 调用量极低
- QMT 是增强选项（Tick级推送、Level2），不是运行监控的基础依赖
- 性能瓶颈不在数据源，而在**本地规则评估计算量**——这也支持"每条规则独立选择扫描频次"
- 风险提示：东方财富偶尔调整接口（上次2025年5月），升级 AkShare 版本即可解决，无需架构改造

### 18.4 评估模式设计

基于右侧交易和数据能力，**暂定 2 种评估模式**（后续如有需要再做添加）：

#### 模式A：📋 盘中定时扫描（核心模式）

| 选项 | 间隔 | 数据源 | 适合条件类型 |
|:----|:----|:-------|:-----------|
| 快速 | 5分钟 | AkShare（全市场实时价）+ Tushare 分钟K | 价格突破+成交量等需较快响应的条件 |
| 标准 | 15分钟 | 同上 | 大多数趋势确认型条件 |
| 中速 | 30分钟 | 同上 | K线形态、多K线确认类 |
| 慢速 | 60分钟 | 同上 | 依赖60分K线级别信号 |

**工作机制：**
```
APScheduler 注册独立 Job（用户设定的扫描周期）
  ↓ 定时触发
条件评估引擎 evaluate()
  ↓ 只读内存/DB数据，零API消耗
条件成立？
  ↓ 是
右侧确认期检查（连续多个周期都满足？）
  ↓ 通过
冷静期检查（是否在N分钟内重复触发？）
  ↓ 通过
通知引擎 → 多渠道推送（Badge/桌面/微信）
```

**数据获取流程（与规则扫描分离）：**
1. 盘中：后台线程每5分钟拉 AkShare 实时全市场快照 → 更新至内存 DataFrame
2. 规则扫描时直接读内存数据 → **零 API 调用、毫秒级评估**
3. 日终：收盘后从 PostgreSQL 读取已缓存日线数据评估

#### 模式B：🌙 日终评估（收盘后批量）

| 时机 | 数据源 | 适合条件类型 |
|:----|:-------|:-----------|
| 收盘后 15:00-15:30 | PostgreSQL 日线数据 | 所有日线级别条件、多日形态确认 |

**可配置项：**
- 推送时间：15:30 / 17:00 / 20:00
- 合并推送：同只股票的多条通知合并为一条摘要

## 二十一、微信推送方案设计（WxPusher + 企业微信机器人）

### §21.1 选型结论

| 角色 | 方案 | 理由 |
|:-----|:-----|:------|
| **主渠道** | **WxPusher** | 完全免费 / 2000条/天 / 微信原生弹窗直达 / HTTP POST一键接入 / 无需用户额外装APP |
| **备渠道** | **企业微信机器人** | 腾讯官方产品无跑路风险 / 完全免费无条数限制 / 用户自愿选择是否下载企业微信APP启用 |
| **已排除** | Server酱 | 免费版5条/天严重不足（监控场景轻松破百条） |
| **已排除** | PushPlus | 实名强制认证 + 超额处罚严厉（超1万条永久禁用微信渠道），风险高于收益 |

# backend/app/services/wechat_notifier.py

import requests
import logging
from flask import current_app

logger = logging.getLogger(__name__)

class WxPusherNotifier:
    """WxPusher 微信推送主渠道"""
    
    API_URL = "https://wxpusher.zjiecode.com/api/send/message"
    
    def send(self, title: str, content: str, summary: str = "") -> bool:
        token = current_app.config.get("WXPUSHER_TOKEN", "")
        uids = current_app.config.get("WXPUSHER_UIDS", [])
        if not token or not uids:
            logger.warning("WxPusher 未配置，跳过推送")
            return False
        
        try:
            resp = requests.post(self.API_URL, json={
                "appToken": token,
                "content": content,
                "summary": summary or title,
                "contentType": 2,  # 2=Markdown
                "uids": uids,
            }, timeout=10)
            return resp.status_code == 200
        except Exception as e:
            logger.error(f"WxPusher 推送失败: {e}")
            return False
```

### §21.4 企业微信机器人接入方案（备选）

**① 用户侧流程**
1. 下载企业微信APP → 手机号注册（无需认证）
2. 创建个人内部群 → 添加群机器人 → 复制 Webhook URL
3. 粘贴到系统全局设置页 → 点击"测试推送"验证

**② 后端实现**

```python
class QyWeixinNotifier:
    """企业微信机器人备用渠道"""
    
    def send(self, content: str) -> bool:
        webhook = current_app.config.get("QYWX_WEBHOOK_URL", "")
        if not webhook:
            return False
        
        # 企业微信 Markdown 最长 4096 字节，超过则截断
        if len(content.encode("utf-8")) > 4096:
            content = content[:1000] + "\n\n> ⚠️ 内容过长已截断，请登录系统查看完整信息"
        
        try:
            resp = requests.post(webhook, json={
                "msgtype": "markdown",
                "markdown": {"content": content}
            }, timeout=10)
            return resp.status_code == 200
        except Exception as e:
            logger.error(f"企业微信推送失败: {e}")
            return False
```

**③ 限流管理**
- 限流阈值：20条/分钟/群（企业微信硬限制）
- 监控场景对照：
  - 盘中扫描（5分钟周期）：最多 1条/5分钟 = 0.2条/分钟，远低于限制
  - 日终评估（批处理 2-50条）：50条可能在分钟限制边缘，需要做队列缓冲

```python
class QyWeixinNotifier:
    """带滑动窗口限流的企业微信机器人"""
    
    def __init__(self):
        self._send_times = []  # 记录最近60秒的发送时间戳
    
    def _check_rate_limit(self) -> bool:
        now = time.time()
        # 清除60秒前的记录
        self._send_times = [t for t in self._send_times if now - t < 60]
        if len(self._send_times) >= 20:
            return False
        self._send_times.append(now)
        return True
    
    def send(self, content: str) -> bool:
        if not self._check_rate_limit():
            logger.warning("企业微信限流，排队等待")
            time.sleep(3)
            return self._try_send(content)  # 重试一次
        return self._try_send(content)
```

### §21.6 推送策略规则

| 策略 | 行为 | 适用场景 |
|:-----|:-----|:---------|
| 仅主渠道 | 只走 WxPusher，企业微信完全不用 | 不想装企业微信的用户 |
| 双渠道同时推 | 两个渠道各自发一次，用户可能收两份但确认到达率高 | 高频交易日的关键规则 |
| 主失败切备 | WxPusher 返回非200时自动走企业微信 | 最省心的默认模式 |

### §21.7 消息去重与冷静期

与 §17.3 冷静期设计保持一致，微信推送同样遵守：

```
同一规则同一股票，冷静期内不重复推送
冷静期：30分钟（用户可调，0=每次都推）
```

实现方式：推送前检查 Redis/内存缓存 → 有冷静期内记录则跳过微信推送，仅记录到系统收件箱。

### §21.8 离线场景边界

**桌面应用固有限制：** 系统关闭后，所有推送渠道（含微信）均无法工作。这不是某个方案的缺陷，是桌面应用架构边界。

**可选增强（未来）：** 如果后续需要"关机也能推"，可考虑：
- 部署轻量云函数（阿里云函数计算 / 腾讯云函数），定时轮询一个状态文件
- 由桌面应用在运行时定期将"需要推送的条件"写入一个公共状态
- 但这超出了当前监控通知模块的范围，列为远期增强

## 二十二、通知记录保存策略与周期报告方案

### §22.1 设计概览

采用 **方案B（时间分层）+ 方案C（用户自配置）** 的组合策略，以方案B的四层结构为默认模板，所有参数开放给用户调整。在此基础上增加周期性 MD 报告推送和历史报告归档管理。

### §22.2 分层保留策略

| 层级 | 默认范围 | 默认内容 | 用户可调范围 |
|:----:|:---------|:---------|:------------|
| 🌟 **L1 全量** | 近 30 天 | 完整记录（条件明细、渠道状态、股票详情、扫描批次、冷静期标记） | 1~90 天 |
| 📋 **L2 摘要** | 31~90 天 | 仅存规则名/股票代码/触发时间/判定结果（✅/❌），删除条件明细 | 30~365 天 |
| 📊 **L3 聚合** | 90 天+ | 每日 1 条聚合（规则ID + 当日触发次数 + 当日达标率） | 开/关 |
| 🔒 **L4 收藏** | 永久 | 用户手动标星的通知，不受清理规则影响 | 手动标记 |

**自动降级流程：**
- **每日 03:00**（系统在运行时）执行降级扫描
- 超 L1 期限 → `UPDATE SET record_level='L2'`（清空条件明细字段）
- 超 L2 期限 → 合并为每日聚合记录（`INSERT INTO daily_aggregation`），删除原摘要记录
- 聚合表结构：`rule_id, date, trigger_count, pass_count, created_at`

### §22.3 清理触发时机

| 触发点 | 行为 |
|:-------|:------|
| 每日 03:00 | 执行自动降级扫描（L1→L2、L2→L3） |
| 每次通知写入后 | 检查是否有超期记录，标记待清理（不阻塞写入） |
| 用户手动触发 | 点击"立即清理"按钮 |

### §22.4 搜索与筛选

搜索能力与右侧通知面板联动，统一实现：

| 筛选维度 | 实现方式 |
|:---------|:---------|
| 🔍 全文搜索 | SQLite LIKE / FTS5（按规则名/股票名称/股票代码） |
| 📅 时间范围 | 日期选择器，预设快捷：今天/本周/本月/自定义 |
| 🏷 规则筛选 | 下拉选择当前用户的所有规则 |
| 📋 股票筛选 | 输入股票代码/名称 |
| 🔴 状态筛选 | 未读/已读/已确认/全部 |
| 📊 结果筛选 | 全部/条件达成 ✅/条件不满足 ❌ |

### §22.6 周期 MD 报告

#### 22.6.1 推送时机

| 报告类型 | 触发时机 | 同步方式 |
|:---------|:---------|:---------|
| 📋 **周报** | 每周最后一个交易日闭市后（15:00-15:30，随日终评估一起执行） | 系统收件箱 + 微信推送（可选） |
| 📊 **月报** | 每月最后一个交易日闭市后（15:00-15:30，随日终评估一起执行） | 系统收件箱 + 微信推送（可选） |

**设计理由：** 考虑到本系统为桌面应用，用户在周日晚20:00或每月1日20:00很可能未开机。绑定在日终评估任务中执行，确保用户在任何一个交易日运行系统，报告都能在闭市后自动生成推送。

**如果用户整个周/月未开机：** 报告不会生成。机制上依耐于日终评估任务，系统未运行即无触发。

## 二十三、通知交互设计（规则分类→操作反馈→规则管理闭环）

### §23.1 从用户价值出发的规则分类

条件库（§3）是按**技术领域**（买卖信号/缠论/筹码等）组织的，而规则需要从**用户价值**出发分类。规则创建时用户需选择规则类型，系统据此自动匹配推送策略、冷静期、操作选项和反馈逻辑。

#### 23.1.1 五种价值类型总览

| 类型 | 用户目标 | 典型条件组合 | 用户心理 | 误报成本 |
|:-----|:---------|:------------|:---------|:--------|
| 📈 **发现机会** | "我想抓住上涨" | MA60突破+量能确认、格兰维尔买点、KDJ金叉 | 期待但需验证 | 高（浪费用户看盘时间）|
| 🛡️ **规避风险** | "帮我盯着别出事" | MACD死叉、跌幅>N%、ST检测 | 必须立刻处理 | 极高（忽略=真亏损）|
| ⚡ **异动追踪** | "这只股票不对劲" | 成交量放大>N倍、价格创新高、集合竞价异动 | 好奇但不确定 | 低（大部分异动是噪音）|

### §23.4 隐式确认与规则热度算法

#### 23.4.1 隐式确认原则

> **删除独立的"信号有效"按钮。用户只要进行有意义的操作即视为对该触发事件的认可。**

| 操作 | 算认可 | 权重 | 说明 |
|:-----|:------:|:----:|:-----|
| 📈 看行情 | ✅ | ×1 | 用户花时间去看股票了 |
| 🧠 策略分析 | ✅ | ×2 | 更深入的策略级分析 |
| 🤖 多角色投研 | ✅ | ×3 | 最高级别深入研判 |
| ⭐ 加自选 | ✅ | ×1 | 明确表示要关注这只股票 |
| ✅ 已处理（风险） | ✅ | ×1 | 用户确认并处理了风险 |
| ✅ 已处理（纪律） | ✅ | ×1 | 用户执行了纪律 |
| ❌ 误报 | ❌ | ×0 | 明确负面反馈 |

#### 23.4.2 规则热度算法

替代原"规则点赞（☆）"方案，用户无需主动点赞，系统根据操作行为自动计算：

```
hot_score = Σ(操作权重) / 总触发次数 × 操作率
其中：操作率 = 有操作的触发数 / 总触发数

示例：
  规则A 触发了100次，用户操作了60次（其中看行情40次、策略分析10次、加自选10次）
  hot_score = (40×1 + 10×2 + 10×1) / 100 × (60/100)
            = 70 / 100 × 0.6
            = 0.42
```

| 分数区间 | 展示 | 含义 |
|:---------|:-----|:------|
| > 0.6 | 🔥 热门 | 用户高频互动，规则非常有效 |
| 0.3~0.6 | 普通 | 用户偶尔关注 |
| < 0.3 | ⚪ 低关注 | 用户不太看这个规则触发 |

**数据用途：**
- 规则管理列表：按热度排序，用户知道哪些规则用得最多
- 规则模板市场：展示"高热度规则"（参考排序）
- 周期报告：标注"🔥 本周最热门规则"

### §23.5 通知→规则的反向影响闭环

| 用户行为类型 | 规则R1 发现机会 | 规则R2 规避风险 | 规则R3 异动追踪 | 规则R4 状态监控 | 规则R5 纪律执行 |
|:------------|:--------------:|:--------------:|:--------------:|:--------------:|:--------------:|
| **用户操作/确认** | 有效触发率↑ / 自信度↑ | 确认收悉 / 停止重推 | 标注关注度 | 无需反馈 | 纪律检查完成 |
| **用户忽略（持续未读/未操作）** | 累计忽略率→建议收窄参数或暂停 | 持续强推（微信2次后停止）+桌面提醒 | 自动降低推送频次 | 建议暂停规则 | 建议调整规则参数 |
| **用户标记误报** | 误报率↑→建议修改条件组合 | 不适用（风控无误报概念） | 不适用（异动是事实）| 不适用 | 不适用（纪律是用户自定）|

**规避风险的特殊性：** 无论用户做何操作，**规则参数不因用户行为自动调整**。风控条件是被动触发的（跌幅>5%就是5%），不能因为用户不看就把阈值放宽到8%。

### §23.6 页面跳转与操作进度保留

#### 23.6.1 新标签页跳转

通知卡片上的 "📈 看行情" 和 "🤖 多角色" 按钮使用 `window.open()` 在新标签页打开，**不干扰用户当前正在进行的操作**。用户看完行情后关闭标签页即可回到原页面。

```javascript
// 通知卡片点击处理

    window.open(url, '_blank');
}
// 示例
openInNewTab(`indicator-ide.html?stock=600519`);
openInNewTab(`ai-analysis.html?stock=600519`);
```

**适用场景：**
- 用户正在编辑规则 → 点"看行情"新标签打开 → 看完关标签 → 规则编辑页完整保留
- 用户正在调整回测参数 → 点"看行情"新标签打开 → 看完关标签 → 回测参数不动
- 用户等待回测结果 → 点"看行情"新标签打开 → 边看行情边等 → 结果出来了再切回去

### §23.7 复盘中心联动

#### 23.7.1 判断逻辑

通知生成时，系统根据当前股票在复盘中心是否有关联的跟进中策略，决定是否显示"📋 查看复盘"按钮：

```python
def has_active_playback(stock_code: str) -> bool:
    """检查该股票在复盘池中是否有跟进中的策略"""
    pool = get_review_pool()  # 从 localStorage / DB 读取
    for entry in pool:
        if entry.stock_code == stock_code and entry.status in ("pending", "holding"):
            return True
    return False
```

- 股票在复盘池中且状态为 `pending` 或 `holding` → 显示"📋 查看复盘"
- 股票不复盘池中，或状态为 `completed` → 不显示

#### 23.7.2 适用范围

### §23.8 统一"已处理"语义说明

将原本分散的"已收悉""已执行""已处理""已知悉"统一为 **"✅ 已处理"**，但其在不同规则类型中的语义略有差异：

| 规则类型 | "已处理"的语义 | 触发后的系统行为 |
|:---------|:--------------|:-----------------|
| 🛡️ 规避风险 | 用户确认风险信号，已完成或即将处理相关操作 | 停止所有重推；规则记录"已处理" |
| 📋 状态监控 | 用户已知悉当前状态 | 等同于已读，无额外操作 |
| 🎯 纪律执行 | 用户已按纪律执行了操作 | 停止该纪律规则的提醒，标记为"已执行" |
