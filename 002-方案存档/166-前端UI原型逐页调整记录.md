## 1. 登录页（login）

### 1.1 调整背景

对 165 号方案中登录页的设计进行合理性评估，用户提出以下疑问：
- 登录页为何存在？
- 访问令牌与 Tushare Token 的区别？
- Tushare Token 是否需要每次输入？
- 不输入 Token 是否影响使用？

经审查实际源码（`backend/app/auth.py` + `frontend/vue-project/src/views/login/index.vue`），明确了两类 Token 的真实作用，并指出原型设计中存在将 Tushare Token 误放登录页的问题。

### 1.2 调整确认内容

#### 1.2.1 访问令牌（AUTH_TOKEN）

| 属性 | 值 |
|:----|:----|
| 默认初始值 | `123456`（纯数字字符串）|
| 配置位置 | `backend/.env` 中 `AUTH_TOKEN=123456` |
| 作用 | 保护后端 API 入口，防止未授权访问 |
| 使用场景 | 部署到公网时启用；单机自用时保留默认值即可 |
| 可自定义 | ✅ 可在登录页通过「修改令牌」功能修改，持久化到 `data/auth_token.txt` |
| 免鉴权兜底 | `.env` 中不设置 `AUTH_TOKEN` 则完全免登录，直接进入系统 |

#### 1.2.2 Tushare Token

| 属性 | 值 |
|:----|:----|
| 性质 | 数据源 API 密钥（Tushare Pro 认证用）|
| 配置位置 | `backend/.env` 中 `TUSHARE_TOKEN=***` |
| 输入方式 | **后端静态配置，不需要用户在前端输入** |
| 缺失影响 | Tushare Pro 不可用，但 AKShare（已安装）作为备用源可正常工作 |
| 结论 | **从登录页移除** — 原型中之前放置 Tushare Token 输入框是错误设计 |

#### 1.2.3 修改令牌功能

新增「修改令牌」功能，交互流程如下：

```
[登录页] → 点击「修改令牌」按钮
    ↓
┌─ 弹窗 Step 1 ───────────────────────────┐
│  请输入当前令牌（验证身份）               │
│  [输入框]  placeholder: 输入当前使用的令牌 │
│  提示：若为默认令牌请输入 123456          │
└──────────────────────────────────────────┘
    ↓ 验证通过
┌─ 弹窗 Step 2 ───────────────────────────┐
│  请输入新令牌                             │
│  [输入框]  placeholder: 6位以上任意字符    │
│  提示：建议使用 8 位以上复杂字符串         │
└──────────────────────────────────────────┘
    ↓
┌─ 弹窗 Step 3 ───────────────────────────┐
│  请再次输入新令牌（二次确认）              │
│  [输入框]  placeholder: 与上一步一致      │
└──────────────────────────────────────────┘
    ↓ 两次一致 → 提交后端
┌─ 成功弹窗 ───────────────────────────────┐
│  ✅ 令牌修改成功                          │
│  请使用新令牌重新登录系统                  │
│  [我知道了]                              │
└──────────────────────────────────────────┘
    ↓ 关闭弹窗 → 登录框自动填入新令牌
```

**修改验证规则：**
- Step 1 → Step 2：旧令牌非空即通过（模拟验证，实际调用后端 `/api/auth/change-token` 校验）
- Step 2 → Step 3：新令牌 ≥ 6 位
- Step 3 → 提交：两次新令牌完全一致

**持久化方案：**
- 后端新增 `backend/data/auth_token.txt` 文件
- 容器启动时读取顺序：`.env` 的 `AUTH_TOKEN` → `data/auth_token.txt` → 默认 `123456`
- 修改令牌时写入 `data/auth_token.txt`（不修改 `.env`，避免 git 冲突）
- 重启容器后令牌保留

### 3.4 策略信号汇总的数据源问题与修订

#### 3.4.1 发现问题

经排查，仪表盘"策略信号汇总"区域存在严重数据源问题：

| 检查项 | 结果 |
|:-------|:-----|
| 前端 API 调用 | `dataService.getStrategySignals()` → `/api/v3/signals/summary` |
| 后端路由 | ❌ **该路由不存在** |
| AI 信号 API | `getAIAnalysisSignals()` → `/api/v3/ai-analysis/signals` |
| 后端路由 | ❌ **该路由不存在** |
| 当前渲染来源 | 前端 mock 数据的兜底逻辑（硬编码 12/8/15/7/5）|
| 风险 | 仪表盘信号汇总长期无真实数据支撑 |

#### 3.4.2 修订方案

策略信号汇总的数据源应统一为 **Screener Pipeline 的 L3 场景A 输出**。

**场景A 输出格式（每只股票）：**
```json
{
  "ts_code": "000762.SZ",
  "name": "西藏矿业",
  "suggestion": "BUY",
  "direction": "bullish",
  "entry_window": "¥12.00-¥13.00",
  "position_pct": 25,
  "holding": "T+20",
  "score": 92,
  "strategy_detail": {
    "volume_price": {"direction": "bullish", "score": 0.85},
    "chanlun": {"direction": "bearish", "score": 0.30},
    "chip": {"direction": "bullish", "score": 0.78}
  }
}
```

**信号汇总区域由场景A输出聚合计算：**

| 当前（硬编码 mock） | 改为（从场景A聚合） |
|:-------------------|:-----------------|
| 量价看多: 12 | `L3 结果中 strategy_detail.volume_price.direction=='bullish' 数量` |
| 缠论信号: 15 | `L3 结果中 strategy_detail.chanlun.score > 0 的数量` |
| 筹码信号: 7 | `L3 结果中 strategy_detail.chip.score > 0 的数量` |
| AI 研判: 5 | `L3 结果中 direction=='bullish' 且 score > 80 的数量` |
| 今日共 47 个信号 | `L3 结果中所有信号汇总` |

**数据流修订：**
```
Screener Pipeline (L1→L2→L3)
    ↓ 场景A输出格式
┌───────────────────────────────┐
│  /api/v3/screener/results     │  ← 新建或扩展此端点
│  返回含场景A格式的排序列表     │
└───────────┬───────────────────┘
            ↓
    ┌───────┴───────┐
    ↓               ↓
AI交易雷达      策略信号汇总
(逐只卡片)      (聚合统计)
```

#### 3.4.3 后端需补充

| 改动 | 说明 |
|:----|:------|
| 扩展 `screener.py` 路由 | `/api/v3/screener/results` 增加场景A格式输出字段 |
| 或新建 `/api/v3/dashboard/summary` | 新端点聚合 L3 输出+统计，一次返回雷达+汇总所需数据 |

### 7.10 筹码分布面板技术方案（重构记录）

#### 7.10.1 当前方案（已废弃）

- 独立 div + 绝对定位 DOM 元素绘制筹码条
- 通过 `chart.getModel().getComponent('yAxis',0).axis.scale.getExtent()` 获
取 K 线 Y 轴范围
- 手动计算 `gridTop+gridH*(1-(basePrice-minP)/priceRange)` 对齐
- 问题：缩放/平移后对齐不稳定，误差随窗口变化

#### 7.10.2 最终方案（像素覆盖法，已实施 → 后续经 DOM 坐标修正）

> **⚠️ 重要更正：** 此处"0 误差"的提法不准确。纯比例映射 `y = 325 × (1 - (price - minP) / priceRange)` 存在**坐标系不统一**的问题：chip-chart 使用独立 60 日价格范围 `[lo, hi]`，ECharts grid Y 轴使用可见 K 线极值范围 — 两个范围不同时，相同价格映射的 Y 像素存在偏差（实测最高 **153.48px**，见 176 号方案 §④）。
>
> **后续修正：** 2026-06-17 在 `renderChipChart()` 中新增 `crosshairY` 参数，通过 `getBoundingClientRect()` 将十字光标 Y 像素换算到 chip-chart DOM 坐标系中，**偏差降至 0.0003px**。详见 §6.7.7。

- chip-chart div 以 **块级 flow 元素** 置于 chip-panel 侧栏内
- **标题** `margin-top:2px; height:18px` → 与"数据:Tushare"同行
- **图谱** `margin-top:34px; height:325px`（= K 线 grid 绘图区 350-10-15）
- **数据面板** `margin-top:31px` → 顶部对齐第一副图(MACD)上沿
- 筹码条 Y 坐标通过比例映射：`y = 325 × (1 - (price - minP) / priceRange)`，与 `getExtent()` 价格范围一致
- 缩放/平移时 `renderChipData` 重新渲染，始终同步

#### 7.10.3 三种方案的对比

| 方案 | 对齐方式 | 精度 | 复杂度 | 最终选择 |
|:-----|:---------|:-----|:-------|:---------|
| DOM 绝对定位 (grid计算) | `gridTop + 325*(1-(P-min)/range)` | ±5px (grid 偏移) | 低 | ❌ 废弃 |
| ECharts custom series | 共享 yAxis | 0 误差 | 高（renderItem） | ❌ 废弃 |
| **像素覆盖 + 侧栏 flow** | `325*(1-(P-min)/range)` + 精确 margin | ⚠️ 独立坐标系偏差可达 153px（后被 DOM 坐标对齐修正至 0.0003px，见 §6.7.7） | 低 | ✅ 基础布局方案 |
| **DOM 坐标对齐（最终修正）** | `getBoundingClientRect()` + crosshairY 换算 | **0.0003px** | 低 | ✅ **最终修正方案**（见 176号方案） |

### 8.6 十字光标标准（修订版 2026-06-17 v3 — 单实例架构）

> **架构变更：** 2026-06-17 由多ECharts实例 + `echarts.connect` 同步方案，重构为 **单ECharts实例 + 多grid布局 + graphic元素 + DOM标签** 方案（见175号方案）。

#### 8.6.1 显示目标

十字光标由一条纵贯所有grid的**白色虚线（竖线）**和一条仅主图区域的**白色虚线（横线）**组成。

- **竖线**贯穿整个页面——从主图grid[0]顶部一直延伸到最底部副图grid的底部
- **横线**只有一根，位于主图K线区域内，锁定在当日收盘价位置
- 副图**不显示横线**，仅竖线穿过
- 完整的视觉呈现：图表区域被一根竖线纵贯，主图内有一根横线与竖线交叉形成"十"字

#### 8.6.2 横线锁定收盘价

不论通过何种方式触发十字光标，横线始终保持统一行为：

| 操作 | 横线行为 |
|:-----|:---------|
| 鼠标在主图上移动 | 横线始终停在竖线所指日期的**收盘价**位置，不跟随鼠标Y轴移动 |
| 鼠标在副图上移动 | 横线仍停在主图对应日期的收盘价位置（不变），不随副图移动 |
| 键盘←→ | 横线锁定在竖线所指日期的收盘价位置 |
| 左键拖拽平移 | 横线不响应 |

#### 8.6.3 副图数据展示方式

所有副图**取消横线**，十字光标穿过副图时仅保留竖线。

副图当前光标位置的数据值，以**文字形式**展示在副图标题栏中。例如：

```
[ 成交额 ▼ ] — 12px — ¥36,900
[ VOL ▼ ]    — 12px — 量:38  MA5:9.36
```

数据文字随十字光标移动实时更新。副图左侧不再显示Y轴数值标签。

#### 8.6.4 渲染方式

**架构：** 单ECharts实例 + 多grid布局（grid[0]=主图，grid[1..N]=副图），所有series并入同一实例。

**十字光标渲染：**

| 元素 | 渲染方式 | 样式 |
|:-----|:---------|:-----|
| 纵贯竖线 | ECharts graphic `{type:'line'}` | `stroke:#E2E8F0, lineDash:[4,4], z:9998` |
| 主图横线 | ECharts graphic `{type:'line'}` | `stroke:#E2E8F0, lineDash:[4,4], z:9998` |
| 价格标签 | DOM div `#cpl-dom`（绝对定位） | `color:#FFFFFF, background:#1e3a5f, z-index:10000` |
| 日期标签 | DOM div `#cdl-dom`（绝对定位） | `color:#FFFFFF, background:#1e3a5f, z-index:10000` |
| 副图标题数值 | DOM `#sub-chart-area .sub-val` | 标题右侧12px间隔，随光标更新 |

**渲染流程：**

```
鼠标移动/键盘方向
    ↓
确定 dataIndex (0~749)
    ↓
第①步：竖线X坐标 → chart.convertToPixel({xAxisIndex:0}, idx)
第②步：横线Y坐标 → chart.convertToPixel({xAxisIndex:0, yAxisIndex:0}, [idx, closePrice])
第③步：chart.setOption({graphic: {elements: [cv, ch]}}) 渲染线条
第④步：DOM标签定位 — cplEl.style.left/top + cdlEl.style.left/top
第⑤步：updateSubHeaders(idx) 更新副图标题栏数值
```

#### 8.6.5 坐标定位

- 横线Y坐标：`yPx = chart.convertToPixel({xAxisIndex:0, yAxisIndex:0}, [idx, closePrice])` — 使用主图yAxis

#### 8.11.1 analysis_report 字段定义（Phase 1）

| `sources` | array | 数据来源追溯（策略文件名 + 数据项名）| `[{ "type": "strategy", "source": "chanlun_strategy.py", "data_item": "上升笔+中枢" }]` |

## §10 回测系统（backtest.html）

> **设计依据：** [197号](002-方案存档/197-量化策略回测知识库调研与UI现状对比分析报告.md)（LLM wiki知识库调研）定义了回测方法论体系；[198号](002-方案存档/198-回测系统底盘设计：标的层级与结果运用策略分析.md)（底盘设计分析）定义了回测标的层级、结果运用策略，以及"实操参考"板块作为回测信号源。
>
> **核心前提：** 本系统最终用户为非专业人士，回测系统设计为"自动贩卖机"而非"实验室"。

### 10.1 页面定位

**回测系统**，用于验证个股策略分析页面输出的操作信号（通过"实操参考"板块）在历史数据上的表现。核心回答用户关心的三个问题：
1. **"按这套分析逻辑操作，历史表现如何？"** — 组合策略回测
2. **"哪个策略在拖后腿？"** — 策略健康度诊断
3. **"回测结果可信吗？"** — 置信度评估

**与indicator-ide页面的关系：**
- indicator-ide（个股策略分析）：输出策略现状 + 实操参考（结构化规则）
- backtest（回测系统）：基于实操参考规则进行历史回测 → 回测报告
