```
function switchStock(tsCode):
  // Step 1: 更新输入框
  stockCodeInput.value = tsCode

  // Step 2: 并行拉取全页数据
  Promise.all([
    fetch(`/api/v3/chart/kline/${tsCode}`),   // K线 + 副图
    fetch(`/api/v3/quote/${tsCode}`),          // 右侧28字段 (待建设)

    fetch(`/api/v3/stk-limit/${tsCode}`),      // 笼子价格 (待建设)
  ]).then(all => {
    // Step 3: 更新各面板
    updateChart(klineData)         // K线 + 副图 + 筹码
    updateSidePanel(quoteData)     // 右侧28字段
    updateMoneyFlow(mfData)        // 资金流向
    updateOrderBook(obData)        // 盘口
    updateChipChart(klineData)     // 筹码
    updateStrategyCards(tsCode)    // 策略解读
  })
```

#### 2.5.1 POST /api/v3/ai/status-interpret — #65 V2输出格式说明

**2026-06-21 #64/#65/#66 升级后，status-interpret 输出格式变更为：**

```json
{
  "success": true,
  "data": {
    "aggregated_status": {
      "version": "2.0",
      "status": {"state": "状态名", "confidence": 0.X, "market_state": "MOMENTUM"},
      "verification_chains": [
        {"chain_id": "chip_volume", "chain_name": "筹码×成交量",
         "passed": true/false, "confidence_multiplier": 1.X,
         "conflict_detail": "冲突描述", "evidence": ["证据1", "证据2"]}
      ],
      "chain_passed_ratio": 0.X,
      "conflicts_for_ai": [{"chain_id": "...", "chain_name": "...", "conflict_detail": "..."}],
      "ai_mode": "summary_only" | "arbitrate_conflicts" | "full_analysis",
      "state_consensus": {...},
      "risk_aggregation": {...},
      "momentum_consensus": {...},
      "dimensions": [...]
    },
    "ai_interpretation": {...}
  }
}
```

**三级AI仲裁模式（依赖chain_passed_ratio）：**
| chain_passed_ratio | ai_mode | 决策逻辑 |
|:-------------------|:---------|:---------|
| ≥ 75% | summary_only | 直接使用验证链共识，AI仅做摘要，置信度×avg_mult |
| 25%~75% | arbitrate_conflicts | 待解决冲突发送给AI仲裁，仲裁加权作为最终决策 |
| ≤ 25% | full_analysis | 全部信息发送给AI做综合分析，固定置信度0.2 |

**四条验证链（#65 新增，均通过 status_intercept:verification_chains 输出）：**
| 链ID | 链名称 | 验证规则 |
|:-----|:-------|:---------|
| chip_volume | 筹码×成交量 | ACCUMULATING+放量→通过(×1.15)；ACCUMULATING+缩量→未通过(×0.85) |
| chanlun_factor | 缠论×因子 | 因子BULLISH+缠论有支撑→通过(×1.1)；因子BEARISH→未通过 |
| emotion_fundamental | 情绪×筹码 | BOCIASI看涨+筹码ACCUMULATING→通过(×1.15)；相反→未通过(×0.8) |
| technical_fundamental | 技术全维度 | 看涨占比≥66%→强通过(×1.2)；≥50%→通过(×1.0)；否则未通过(×0.8) |

## 四、关键差距分析

### 4.1 回测系统 — 全面重构（P0，2026-06-23设计确认）

> **2026-06-23 修订：** 基于197号(知识库调研)和198号(底盘设计)方案，回测系统已完成全新设计，包含"实操参考"信号源机制。原型已全面重写，详见[166号§10](002-方案存档/166-前端UI原型逐页调整记录.md)。

**旧问题：** `views/backtest/index.vue` 的 `finishBacktest()`（第 418 行）返回完全硬编码的假数据，不调用任何后端 API。

**影响：**
- 回测结果不可信
- 因子组合选择无实际效果（UI 上切换组合改变的只是 TotalReturn 从 15.68 变为 25.68）
- 无法复现/无法验证
- 无回测结果中的诊断能力（策略健康度/冲突成本/可信度）
- 无回测信号源——不知道"用什么信号来回测"

**新设计解决方案：**
1. **实操参考信号源**：回测信号来自 indicator-ide 的结构化开仓/平仓条件，用户一键发送到回测
2. **三层级回测**：组合为主（用户视角）+ 消融为辅（策略诊断）+ 单策略后台分析
3. **非专业用户兜底**：三档判定(红/黄/绿) + 情景回溯 + 置信度评估 + 预设模式切换

**设计依据文档：**
- [197号](002-方案存档/197-量化策略回测知识库调研与UI现状对比分析报告.md) — 知识库方法论与UI对比

- [166号§10](002-方案存档/166-前端UI原型逐页调整记录.md) — 原型规格
