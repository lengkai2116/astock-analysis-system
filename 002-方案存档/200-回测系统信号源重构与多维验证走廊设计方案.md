# 200 — 回测系统信号源重构与多维验证走廊设计方案

## 1. 问题背景

### 1.1 当前设计中的架构错误

回测系统的信号源定义存在根本性偏差。198号方案将 indicator-ide 页面的"实操参考"板块设计为回测信号源，但这个设计在架构上是错误的：

| 概念 | 性质 | 能否作为回测信号源 |
|:----|:----|:-----------------|
| **实操参考** | 基于今日数据的AI推理结论，是时点静态快照 | ❌ 不可——无法在历史日期上评估 |
| **AI叙事情景推演** | 对未来的概率性描述（"放量突破概率40%"） | ❌ 不可——非机械条件 |
| **组合策略引擎（UPF/UIDF）** | 可在任何历史日期上独立运行的机械规则 | ✅ 可——有固定入参和输出 |

核心矛盾在于：**回测引擎接收的是静态信号列表（`signals: [{date, ts_code, signal: 1/-1/0}]`），而实操参考输出的是不可跨时间复用的AI结论。**

### 1.2 正确架构：回测应验证组合策略机制本身

回测回答的根本问题应该是：

> **"如果我选择这几种策略、按这个权重融合、用这套冲突仲裁逻辑，它们在历史上会表现得怎么样？"**

而不是：

> **"如果把今天的AI建议拿到历史上跑一遍会怎样？"**

前者验证的是**策略组合机制的有效性**，后者验证的是**一个静态规则的巧合吻合**——后者没有意义。

### 1.3 实操参考的新定位

实操参考应**与回测解耦**，仅作为面向用户的参考信息展示：

- ✅ 保留实操参考板块（已有开仓/平仓条件展示）
- ✅ 保留"仅供回测验证"标签
- ❌ **移除**"发送到回测"按钮关联——实操参考内容不送入回测引擎
- ❌ **移除**条件③中引用"情景A/B/C"的不可机械执行条件
- ✅ **新增**"仅供参考"提示，明确声明不构成操作建议

## 2. 回测信号源的精确定义

### 2.1 新链路

```
策略组合来源：默认「个股策略分析」配置（可选独立覆盖）
                ↓
标的范围来源：⾃选池（来⾃选股系统筛选结果）
                ↓
后端逐日期运行组合策略引擎：
  Date D：
    1.  取截至D日的数据
    2.  并行运行：缠论(lookback=120) / 量价(lookback=120) / 筹码(lookback=60) / 多因子(20) / BOCIASI
```

### 2.4 信号生成时序细节

组合策略的回测信号不是一次性生成的，而是在每个历史日期上**逐期执行**（每只股票独立）：

```python
def run_single_stock(ts_code, price_data, ...):
    signals = []
    for date in historical_dates:
        # 取截至该日期的数据（防止前视偏差）
        data_slice = price_data[:date]

        # 并发运行各策略（各策略按各自的 STRATEGY_WINDOWS 取数）
        chanlun_result = chanlun_strategy.analyze(data_slice, param)
        vp_result = volume_price_strategy.analyze(data_slice, param)
        chip_result = chip_strategy.analyze(data_slice, param)
        factor_result = signal_service.compute_factor_signal(data_slice)
        bociasi_signal = get_bociasi_signal(data_slice)

        # UPF融合
        context = {
            'chanlun_result': chanlun_result,
            'volume_price_result': vp_result,
            'factor_result': factor_result,
            'bociasi_signal': bociasi_signal,
        }
        upf_result = upf_engine.evaluate_all(context)

        # 冲突仲裁
        conflict_result = conflict_arbiter.arbitrate(upf_result)

        # UIDF决策
        uidf_input = AIInterpretInput(upf_result=upf_result, ...)
        decision = uidf_engine.consolidate(upf_result, ai_output=None)  # AI层可选
    
    # 生成交易信号
    signal = 1 if decision.signal == 'BULLISH' else -1 if decision.signal == 'BEARISH' else 0
    signals.append({'date': date, 'ts_code': ts_code, 'signal': signal})
    return signals

# 横截面执行
def run_cross_sectional(ts_codes, ...):
    with concurrent.futures.ThreadPoolExecutor(max_workers=16) as executor:
        futures = {executor.submit(run_single_stock, code, ...): code for code in ts_codes}
        all_results = {}
        for future in concurrent.futures.as_completed(futures):
            code = futures[future]
            all_results[code] = future.result()

    # 横截面聚合
    sharpe_all = [r.sharpe for r in all_results.values()]
    return {
        'median_sharpe': np.median(sharpe_all),
        'iqr_sharpe': np.percentile(sharpe_all, [25, 75]),
        'positive_ratio': sum(s > 0 for s in sharpe_all) / len(sharpe_all),
        'stability': np.std(sharpe_all),
        # 各判据通过率汇总 → 2D矩阵
        'matrix': build_validation_matrix(all_results),
        # 修订建议自动生成
        'recommendations': generate_recommendations(matrix),
    }
```

## 2.5 策略回测适用性评估

### 2.5.1 各策略类型回测可行性总表

| 策略类型 | 示例 | 可回测 | 条件 | 当前状态 |
|:--------|:----|:------|:-----|:---------|
| **系统内置5策略** | 缠论/量价/筹码/多因子/BOCIASI | ✅ **可** | 已有 `analyze(data_slice)` 确定性输出 | ✅ 已实现，零额外适配 |
| **系统策略模板（196号）** | S1情绪周期/S2主力行为/S3趋势通道/S4涨停板/S5风控/S6波浪/S7斐波那契 | ✅ **可** | 需实现与现有策略相同的 `analyze(data_slice, param)` 接口；S6波浪需额外标准化规则 | ⏳ Phase 1-3 |
| **用户自组合因子** | 从FactorRegistry选N个因子×权重 | ✅ **可** | 因子必须为FactorRegistry中已注册的**确定性计算**因子（当前500+基本满足） | ⏳ 已有`/api/factors/combinations` CRUD接口 |
| **Vibe Coding自建策略** | 用户编写的自定义规则组合 | ⚠️ **有条件可** | 需额外实现 `generate_signal(data_slice)` → `{signal: 1/-1/0, confidence}`；静态校验确保确定性；参考§2.5.2 | ⏳ Phase 待定义 |

### 2.5.2 Vibe Coding 策略的回测约束

Vibe Coding 策略当前设计输出 `describe_status()`（叙事文本），但回测链路需要的是**机械可执行的交易信号**：

```python
class CustomVibeStrategy:
    """用户自建策略"""

    # ✅ 已有（供展示用）
    def describe_status(self, ts_code) -> Dict:
        return {"组合指向": "短期动量向上", ...}

    # 🔔 新增（供回测用）
    def generate_signal(self, data_slice: pd.DataFrame) -> dict:
        """
        纯确定性计算，回测引擎逐期调用
        输出：{signal: 1/-1/0, confidence: float}
        """
        # 可用工具：均值/交叉/比值/突破判断/注册因子值/条件组合(AND/OR/XOR)
        ...
```

**核心约束三原则**：

| 约束 | 说明 | 违规示例 |
|:----|:-----|:---------|
| ❌ 无外部调用 | 不得请求外部API（如LLM判断） | `requests.post('...', json={"走势": data})` |
| ❌ 无随机性 | 必须确定性计算 | `random.choice(['BUY','SELL'])` |
| ❌ 无用"未来数据" | 只能用 `data_slice`（截至当前日期的数据） | `data.iloc[-1]['close']/data.iloc[0]['close']` 即访问了整段 → 系统截断强制 |

**静态校验方式**：策略编辑器在保存时对代码做AST扫描，识别非法调用模式。不通过的策略在回测模块中**标记为"不可回测"**，仅保留描述性展示。

### 2.5.3 两套策略组合配置入口的决策

系统存在两套策略组合配置入口，职责分离如下：

```
┌─ 选股系统（screener.html） ─────────────────────────────┐
│  目的：全市场 → 候选池筛选                                │
│  策略含义：「什么特征的股票是好的」（打分权重）             │
│  输出：股票列表+评分                                      │
│  包含：经典动量组合/多因子评分/Vibe评分                    │
│  回测角色：标的来源（通过自选池间接提供） ✅ 200号§4.2    │
└──────────────────────────────────────────────────────────┘

┌─ 个股策略分析（indicator-ide.html） ────────────────────┐
│  目的：单只股票 → 现状识别+AI仲裁                          │
│  策略含义：「用什么维度分析这只股票」（信号权重）           │
│  输出：五维现状描述+AI仲裁+实操参考                        │
│  包含：缠论/量价/筹码/多因子/BOCIASI + 可选扩展策略         │
│  回测角色：策略组合默认来源 ← NEW                         │
└──────────────────────────────────────────────────────────┘
```

**决策结论：回测信号源默认遵循「个股策略分析」的策略配置，标的来源默认遵循选股系统的自选池。**

| 回测要素 | 来源 | 说明 |
|:--------|:-----|:-----|
| **策略组合（+权重）** | 默认同步自 `indicator-ide` 的策略配置 | 回测验证的就是"这个分析框架本身"是否有效，与现状分析保持一致 |
| **标的范围** | 自选池（来自选股系统筛选结果） | 已按§4.2规则自动/手动选取 |
| **用户可覆盖** | 在回测面板中独立修改策略配置 | 展开后临时覆盖，不影响 indicator-ide 的配置 |

### 2.5.4 策略组合来源版本管理

每次用户修改 indicator-ide 策略配置时：
- 回测面板顶部显示"当前策略配置已更新，建议同步"
- 用户可选择：「🔄 同步」或「保持当前回测配置」
- 后端记录版本号：`backtest_config_versions` 表（strategy_json_hash → timestamp）

### 3.2 四个稳健性判据的判定规则

#### 3.2.1 走步前推一致性（Walk-Forward Consistency）

**后端状态**：❌ 未实现

拆分时间窗口：S行窗口（如60天训练+20天验证，向前滚动）

核心指标：

```
一致性 = 1 − |训练期夏普 − 验证期夏普| / max(|训练期夏普|, |验证期夏普|)
```

| 范围 | 判定 |
|:----|:----|
| ≥ 0.95 | 🟢 优秀（几乎无过拟合） |
| 0.85 ~ 0.95 | 🟢 合格 |
| 0.75 ~ 0.85 | 🟡 需谨慎 |
| < 0.75 | 🔴 不可信 |

**最低验证标准**（知识库定义）：
- 夏普衰减 < 0.3
- 收益衰减比例 < 30%
- 回撤恶化 < 5%

#### 3.2.2 CSCV / PBO（过拟合概率）

**后端状态**：✅ `cscv_validator.py` 已实现（#56）

| PBO | 判定 |
|:----|:----|
| < 20% | 🟢 稳健，可考虑部署 |
| 20% ~ 30% | 🟢 较好 |
| 30% ~ 60% | 🟡 中等风险 |
| > 60% | 🔴 不可信 |

#### 3.2.3 参数平原分析（Parameter Plateau）

**后端状态**：✅ `optimizer.py` 已实现（#45）

| 判据 | 要求 |
|:----|:----|
| 宽度 | 参数跨度 ≥ 30%（如窗口20→40） |
| 深度 | 绩效波动 < 15% |
| 连通性 | 单连通区域，无断裂 |

满足全部三条 → 🟢 捕捉真实市场机制
满足两条 → 🟡 部分可信
满足少于两条 → 🔴 参数山峰，过拟合高风险

#### 3.2.4 蒙特卡洛分布（Monte Carlo）

**后端状态**：❌ 未实现

生成N条随机价格路径（或打乱交易顺序），观察收益分布：

- 如果95%置信区间不包含零 → 🟢 统计显著
- 如果收益分布与随机策略无显著差异 → 🔴 不可信
- 破产概率 > 5% → 🔴 风险过大

#### 3.2.5 综合判定规则

```
四项中通过 ≥ 3 项 → 🟢 结果可信
                   回测可作决策参考，且多维度间无矛盾
通过 2 项         → 🟡 需谨慎
                   回测结果仅作参考，需等待更多验证
通过 ≤ 1 项       → 🔴 不可信
                   回测结果不能作为任何决策依据
```

### 3.3 策略贡献度诊断

| 指标 | 计算方式 | 含义 |
|:----|:--------|:-----|
| 边际贡献率 | 去除该策略后的收益变化百分比 | 该策略对总收益的正/负贡献 |
| 信号一致率 | 该策略与最终决策方向一致的信号占比 | 该策略的"正确率" |
| 冲突频率 | 该策略与其他策略信号方向相反的次数占比 | 该策略的"争议程度" |
| 错误方向占比 | 该策略看多但亏钱 / 看空但少亏的占比 | 该策略的实际可靠性 |

### 3.4 情境分析设计

按市场状态（TRENDING_BULL / TRENDING_BEAR / RANGING / HIGH_VOL）切片统计：

```
┌────────────────────────────────────────────────────────────────┐
```

### 4.1 关键交互行为

- **策略多选**：勾选策略后自动显示权重滑块，未勾选的策略权重自动归零
- **权重联动**：拖动任一滑块，其他权重按比例缩放保持100%
- **预设权重**：提供三种预设（均衡/保守/进取），一键切换
- **模式切换**：切换回测模式时，底部的"开始回测"按钮文案变化（基础→一致性→PBO→平原）
- **标的范围联动**：默认读取自选监控列表；自动计算并显示已选数量与目标数量（30只）；当自选≤30只时，灰色显示选择方式选项，标明"全量回测"；当自选>30只时，暴露随机/手动二选一
- **策略来源默认同步**：默认直接从 indicator-ide 读取策略组合配置，首次进入回测页面时自动同步一次
- **同步状态提示**：显示✅已同步或🔄待同步（当用户修改indicator-ide配置后），点击同步按钮更新
- **独立覆盖入口**：展开「✏️ 在此处独立修改」后，策略配置直接面板编辑，不影响indicator-ide；再次同步会覆盖独立修改
- **可回测策略自动识别**：未实现 `generate_signal()` 的Vibe策略、数据源不可用的策略模板 → 灰色禁用，hover提示"暂不可回测"

## 8. 时长评估与优化策略

### 8.1 单标的各环节耗时

| 环节 | 耗时 | 说明 |
|:----|:----|:-----|
| **基础回测**（5策略+UPF+UIDF+交易模拟） | ~3-5s | 1500天数据，逐日运行 |
| **走步前推**（10个滚动窗口） | ~30s | 每次窗口=一次基础回测 |
| **CSCV/PBO**（30个样本组合） | ~15s | 组合对称交叉计算 |
| **参数平原**（4策略×5网格点） | ~80s | 每网格点=一次基础回测 |
| **蒙特卡洛**（200次随机路径） | ~10s | 随机打乱交易顺序 |
| **单只全维度总计** | **~130-140s** | |

### 8.2 多标的组合耗时

| 模式 | 标数 | 验证维度 | 原始耗时 | 16核并行 | 可行性 |
|:----|:----|:--------|:--------|:---------|:------|
| 🚀 **快速试算** | 1 | 基础（无4判据） | ~5s | ~5s | ✅ 即时出结果，无需异步 |
| ⚡ **轻量验证** | 10 | 基础+走步前推+CSCV | ~8min | ~50s | ✅ 接受，设进度反馈 |
| ⚡ **标准验证（推荐）** | 20 | 全部4判据 | ~25min | ~5min | ⚠️ 需异步+进度通知 |
| 🔬 **深度验证** | 30 | 全部4判据 | ~45min | ~8min | ⚠️ 需异步+并行优化 |

### 8.3 优化策略

| 优化手段 | 效果 | 实现难度 |
|:--------|:----|:--------|
| 股票间并行（16核K=30→2批） | 45min→15min（进一步8min） | ✅ Python `concurrent.futures` |
| 缓存策略中间结果（各股票每日UPF输出） | 二次运行→秒级 | ⚠️ 需DB表+版本号 |
| **预判筛选**（先跑基础指标，剔除明显无效股票后全维度验证） | 有效股数少50-60% | ✅ 规则简单 |
| 异步队列 + 进度WebSocket | 用户可边做其他事边等待 | ⚠️ 需队列服务 |

**标准配置推荐**：
1. 用户选30只股票 → 先跑基础回测（~50s，设进度条）
2. 自动剔除夏普<0 或信号频次<10的股票（~20只保留）
3. 仅对保留的股票跑全维度验证（~5min，异步后台）
4. 完成后通知用户查看结果

### 8.4 回退与降级方案

如果多股票+多维验证性能不可接受：

| 方案 | 性能 | 精度 |
|:----|:----|:----|
| **逐期运行+16核并行**（推荐） | 慢但准确 | ✅ 最高 |
| 按月取样（每月末运行一次） | 中（~0.5s/标的） | ⚠️ 信号可能遗漏 |
| 缓存策略中间结果 | 快（首次慢，后续复用） | ✅ 最高 |
| 简化策略（部分策略固定输出） | 快（~0.2s） | ❌ 无法反映参数变化 |
