### 4.4 备份 Token 方案（防丢失）

在 `global.less` 中使用 `var(--xxx, fallback)` 语法为每个 Token 提供 fallback 值：

```less
--text-primary: #F1F5F9;
/* 使用处 */ color: var(--text-primary, #F1F5F9);
```

该方案已在部分组件中使用（如 `DisclaimerFooter.vue`），需系统化推广。

### 8.3 从 163 方案调整的内容

| 163 项 | 调整 | 原因 |

### 8.4 风险与控制

| 风险 | 概率 | 影响 | 缓解措施 |
|:----|:----:|:----:|---------|

## 总结

本方案在 163 号规划的基础上，通过全量后端 API 映射发现了前端 **70% 的后端能力缺口**。核心交付包括：

| 维度 | 163 方案 | 本方案（164） |
|:----|:--------:|:------------:|
| 阶段数 | 4 期 | 5 期（新增 1.5 + 2.5） |
| 后端 API 覆盖率 | 未考量 | ~31% → ~88% |
| 前端 Route 页面 | 14 | 17 |
| 新 Service 文件 | 0 | 11 新建 + 4 扩展 |
| 新组件 | ~15 | ~18 |
| 总工作量 | ~106h | ~160h |
| 实施周期 | ~13 工作日 | ~26 工作日 |

**核心原则**：不重构现有框架，在既定技术栈内做增量扩展。**先服务层基建（Phase 2 前 2 天）→ 再页面填补（Phase 2 余下 + 2.5）→ 最终体验打磨（Phase 3）**。每期结束后执行 Playwright 回归验证。
