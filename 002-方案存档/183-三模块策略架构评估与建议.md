# 三模块策略架构评估与建议

**依据：用户沟通（2026-06-19） + 后端代码完整扫描 + 前端页面逐文件分析 + 历史文档交叉验证**

---

## 一、对用户架构的理解（确认）

系统整体的策略逻辑拆分为三个**互相独立**的板块，各有明确分工：

| 板块 | 页面 | 定位 | 用户角色视角 |
|:-----|:-----|:-----|:------------|
| **① 选股系统** | `/screener` | **快速筛选** — 全市场范围内剔除风险、识别主力、多策略评分，输出"值得关注的股票清单" | 狩猎者 — 从5000只中锁定目标 |
| **② 个股策略分析** | `/indicator-ide` | **深度现状解读** — 对指定股票运行全策略进行**现状判定**（不是建议输出），然后 AI + LLM Wiki 联动，对现状进行分析解读，按输出模板给出策略建议 | 分析师 — 对目标做功课 |
| **③ AI多角色评判** | `/ai-analysis` | **综合评判** — 多角色从不同视角对相同股票进行独立分析、讨论、综合评定，输出专业评判 | 投委会 — 各方专家会诊投票 |

**三者的关系不是流程上下游，而是并列的三条线索：**
- 用户先用**①选股系统**锁定目标
- 然后进入**②个股策略分析**做深度功课（AI解读现状 + 策略建议）
- 同时可发起**③AI多角色评判**从多视角获得综合意见
- 最终用户结合①+②+③的信息，自主做出投资决策

**你强调的核心理念确认：**
> "系统策略不是建议输出而是现状识别，建议由 AI 根据现状来进行确认"

这条原则我**完全认同**，且是目前系统中最需要修正的设计偏差。当前系统的策略（缠论/量价/筹码）直接输出 BUY/SELL/HOLD 建议，跳过了"现状识别→AI解读"的关键环节。

## 二、当前状态逐模块评估

### 模块①：选股系统 — ✅ 基本成型，需局部优化

**当前状态：**
- 后端：L1（DarwinRiskFilter）+ L2（ChipScorer）+ L3（多策略验证），流水线完整
- 前端：PipelineFlow 组件展示三层漏斗 + ScreeningResults 榜单 + SignalFusionConfig 权重配置
- API：`POST /api/v3/screener/run` + 各层级独立端点（`/layer1`、`/layer2`、`/layer3`）

**问题：**
1. **L3 多策略验证较弱** — 当前仅用 ChipScorer.score() + 简单指标提取（ASR/集中度/量比/RSI），并未跑缠论/量价/因子三套策略做交叉验证。真正的 L3 应该让三个策略各输出现状判定，再融合评分
2. **结果与模块②联动缺失** — 选股结果双击后路由到 `/indicator-ide`，但该页面是代码编辑器而非策略分析页，用户无法直接看到该股票的完整现状解读
3. **筛选参数缺少保存机制** — industry/market 等筛选条件每次重新填

**建议：**
- L3 增强为真正的三策略交叉验证（当前约 80 行代码的 extension）
- 选股结果项增加"深度分析"按钮，路由到模块②的个股策略分析页
- 筛选参数持久化到 localStorage

### 模块②：个股策略分析 — ❌ 最核心的缺口

**当前状态：**
- 前端路由到 `/indicator-ide`，页面本质是 **K线图表 + 自定义指标代码编辑器**（CodeEditor组件）
- 右侧信号面板已显示 L2/L3 策略信号，**但展示的是原始的 BUY/SELL/HOLD 和证据列表**，没有 AI 解读，没有规范化的现状描述，没有执行建议模板
- 后端 `SignalComputationService` 已具备全策略实时计算能力（筹码/缠论/量价/因子），但输出的是统一的 signal 字典格式，**缺少"现状描述"层**

**核心差距分析：**

| 维度 | 当前系统做了什么 | 用户想要什么 |
|:-----|:---------------|:-----------|
| 系统策略输出 | 直接输出 BUY/SELL/HOLD + 置信度 + 证据列表 | 输出**现状判定**（如"日线震荡筑底中，周线上升趋势，量价背离需关注"） |
| AI 的角色 | 仅在 `/ai-analysis` 做多角色分析 | 在个股层面用 AI（LLM Wiki）对现状做解读、解释、给出执行建议 |
| 输出格式 | JSON 结构化的信号字典，前端直接显示 | 有**输出模板**，让用户能看懂、能据此制定执行方案 |
| LLM Wiki 联动 | 仅在 `explain_signal()` 中简单使用，且是 Mock | 深度联动 — 用知识库中的分析方法来解释当前符合什么形态、处于什么状态 |

**建议 —— 全新「个股策略分析页」的蓝图：**

这个页面不应是 `/indicator-ide`（那个应改为一个独立的开发者工具页），而是新建页面，结构为：


**技术路线：**
1. **新建前端页面** `/stock-analysis`（`views/stock-analysis/index.vue`），复用现有 KLineChart 组件
2. **新建后端路由** `/api/v3/stock/:ts_code/status` — 调用 SignalComputationService + StrategyOutputService，输出所有策略的**现状判定**（非建议）
3. **新建 AI 解读端点** `/api/v3/ai/interpret-status` — 接收策略现状数据 + 调用 LLM Wiki 联动，按输出模板生成解读和建议
4. **输出模板**定义为可配置的 JSON Schema，前端按模板渲染
5. **现状判定数据模型**（新概念）：

```python
# 策略层输出不再是 signal（BUY/SELL），而是 status
{
    "strategy": "筹码主力分析",
    "status": {
        "phase": "主力建仓期",           # 建仓/洗盘/拉升/出货/不明
        "phase_confidence": 0.65,
        "key_observations": ["筹码集中度上升", "量能温和放大", "股价在60日低位区"],
        "signal": "NEUTRAL"              # 保留原始信号供系统内部使用
    }
}
```

### 模块③：AI多角色评判 — ⚠️ 骨架存在，需深化

**当前状态：**
- 5 个角色（技术/基本面/宏观/风控/基金经理），串行分析
- 全部 Mock 兜底（DeepSeek 未实际接入）
- 有进度条展示、分析日志、最终报告
- `ai_context_builder` 已提供上下文注入框架
- `ai_structured_parser` 已支持 JSON 结构化解析

**问题：**
1. **分析流程过于简单** — 5 个角色独立分析后基金经理合成，缺乏角色之间的讨论和辩论环节
2. **Mock 数据的质量太低** — 固定模板，每个股票都是一样的泛泛内容
3. **报告结构单一** — 只有文本摘要，缺少可视化、对比分析、共识/分歧标记
4. **无历史记录** — 每次分析结果只在内存中，刷新页面就丢失
5. **未接入 LLM Wiki** — 角色 prompt 中未引用知识库，分析师无法基于系统知识回答

**深化建议：**

**第1层：现有的并行分析（P0）→ 优化数据质量和接入真实 LLM**
- 配置 DeepSeek/LM Studio 接入，替换 Mock
- 每个角色分析时注入 `ai_context_builder.build_context(ts_code)` 获取完整的多维度数据
- 角色 prompt 中加入引用 LLM Wiki 的引导（"请参考缠论背驰判定标准分析当前走势"）

**第2层：增加辩论环节（P1）**
```
Phase 1 (并行分析): 5角色各自独立分析 → 输出初步意见
Phase 2 (辩论讨论): 技术 vs 宏观 观点冲突时，触发第二轮辩论
Phase 3 (最终合成): 基金经理综合所有意见 + 辩论记录 → 输出最终报告
```
- 辩论规则：当两个角色的 bullishScore 差距 > 30 分时，自动触发一轮辩论（各角色回应对方观点）
- 辩论格式：角色 A 陈述 → 角色 B 反驳 → 角色 A 回应 → 双方各自更新评分

**第3层：结果结构化+可视化（P2）**
- 输出共识矩阵：显示所有角色在哪些维度上达成共识，在哪些维度上存在分歧
- 观点分歧可视化：雷达图展示各角色评分差异
- 历史对比：同一股票多次分析的评分变化趋势

**第4层：持久化（P2）**
- 分析结果写入数据库（`AiAnalysis` 模型），按 ts_code 可查询历史
- 记录每个角色的 full report text，支持回溯

## 三、跨模块关键问题汇总

### 问题1：策略层的"现状识别"与"建议输出"混淆（优先级最高）

**当前设计缺陷：**
所有策略（SignalComputationService）的输出格式都是面向建议的：
```python
{
    "signal": "BULLISH",          # 直接给建议
    "confidence": 0.75,
    "entry_zone": [xx, xx],
    "risk_line": xx,
    "target_zone": [xx, xx],
    "position_suggestion": "15%",
    "holding_period": "1-3个月",
}
```

**你的理念要求策略层做现状识别，由 AI 做建议：**
```python
# 策略层输出（现状）
{
    "strategy": "筹码主力分析",
    "status_description": "当前处于主力建仓阶段，筹码集中度持续提升中，成交量温和放大",
    "status_details": {
        "phase": "建仓期",
        "phase_score": 7.2,
        "key_findings": ["筹码集中度上升15%", "量比1.8", "股价在60日低位区"],
        "divergence": None,
        "ma_alignment": "缠绕",
    },
    "raw_signal": "BULLISH",    # 保留但仅供系统/后端消费
}

# AI 层输出（建议）
{
    "operations": [
        {"title": "操作方向", "value": "逢低布局"},
        {"title": "入场区间", "value": "188.50 - 195.00"},
        {"title": "仓位建议", "value": "首档5%，确认后10%"},
        # ... 模板填充
    ],
    "ai_reasoning": "...",
    "wiki_references": ["...", "..."],
}
```

**改动范围：**
- 新建 `StatusOutputService`（或修改 `StrategyOutputService`）—— 在现有信号输出之上增加"现状描述"层
- 策略层代码基本不变，只在其输出外面包裹一层 status_description
- AI 解读端点 `/api/v3/ai/interpret-status` 为核心新工作

### 问题2：各模块间的导航与联动缺失

**当前导航结构问题：**
```
Sidebar:
  📊 仪表盘
  🎯 选股系统         → /screener (模块①)
  📈 个股策略分析       → /indicator-ide (名义是模块②，实际是代码IDE)
  👁️ 自选监控
  🤖 AI分析            → /ai-analysis (模块③)
  🔙 回测系统
  ...
```

**问题：**
- "个股策略分析"应该是一个独立于"策略IDE"的新页面
- 选股结果 → 双击应跳转到正确的个股策略分析页，而非代码编辑器
- AI分析页应能直接通过股票选择触发（目前只能从自选股列表选）

**建议导航调整：**
- `/stock-analysis` → 个股策略分析（模块②，完整现状解读+AI建议）
- `/indicator-ide` → 策略IDE（开发者工具，给高级用户写自定义指标代码）
- 三模块之间通过股票代码关联：选股结果 → 个股分析 → AI多角色评判（一键发起）

### 问题3：LLM Wiki 在模块②和模块③中的应用

**当前现状：**
- `ai_context_builder` 负责构建数据上下文字段（股价/指标/技术面），但不引用 LLM Wiki 知识库
- 模块③的角色 prompt 纯文本描述，没有引用知识库中的分析方法论
- 模块②完全没有 AI 解读

**建议的 Wiki 联动方式：**

模块②（个股策略分析）：
```
系统策略输出现状"日线MACD底背离" →
AI 查询 LLM Wiki 中关于"MACD面积比较背驰"的判定标准和解释 →
生成解读"当前符合标准顶底背驰形态，背驰确认度XX%，详见Wiki的chanlun/背驰/章节"
```

模块③（AI多角色评判）：
```
角色prompt中嵌入知识库引用：
"当分析趋势时，请参考系统知识库中关于趋势判定(均线系统/通道位置/缠论笔段)的方法论"
"当分析量价关系时，请参考系统知识库中的量价八准则三步法"
```

### 问题4：达尔文风险筛选在模块②中的角色

**你的疑问："达尔文风险筛选策略需要单独评估是否使用"**

我的评估：
- **不应用于模块②的现状分析流程中**。风险筛选是"是否值得关注"的门槛判断，不是"现状是什么"的描述维度。
- 理由：风险筛选是二值判断（通过/不通过），不适合作为维度的特征描述。
- 更好的定位：在模块①的 L1 中已经使用，模块②可以在 AI 解读的"风险提示"部分引用其结论。

## 五、核心结论

1. **三模块架构设计本身是清晰的** — 三个独立板块各司其职，和当前系统的代码结构没有根本冲突
2. **最核心的架构修正：策略层做现状识别，AI 层做建议输出** — 这是你最重要的设计决策，当前系统完全跳过了这个分层
3. **模块②（个股策略分析）是最大的建设缺口** — 目前 `/indicator-ide` 页面的定位是错误的（那是代码IDE，不是策略分析页），需要一个全新的页面和配套后端
4. **模块③（AI多角色评判）骨架已有但内容需深化** — 接入真实 LLM + 增加辩论环节 + 结构化输出
5. **LLM Wiki 的联动是模块②和③的核心差异竞争力** — 没有 Wiki 联动，AI 解读就和任何通用 LLM 没有区别
6. **所有 P0 项都可在不涉及前端改动的前提下完成**（后端逻辑重构 + 新 endpoint），P1 项待前后端打通时做
