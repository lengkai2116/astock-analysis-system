## 一、当前数据源架构总览

```
Tushare Pro (主, priority=0)
  ├─ 积分: **≥5000分（已确认就绪）**
  ├─ Token: 已配置已验证
  └─ 状态: NORMAL (健康)

AKShare/Sina (备, priority=1)
  ├─ 安装: 已安装 pip 包, 按需导入
  ├─ 延迟: 3-5秒
  └─ 范围: 仅10只写死股票名映射

QMT/招商证券 (预留, 未激活)
  ├─ 代码: 6个端点已写, 全部标注休眠
  └─ 依赖: xtquant 库(需安装QMT客户端)
```

**三层数据管道：**
```
Tushare API
    ↓
Redis (L1热缓存, TTL 1h, ~250交易日/股)
    ↓
DuckDB (L2全量缓存, 307MB, 8张表)
    ↓
PostgreSQL (L3持久化, 15表, 608MB日线数据)
    ↓
128个后端路由 → 前端展现
```

## 六、核心结论

1. **基础数据管道是通的** — Tushare→DuckDB→PostgreSQL→后端的全链路已验证，日线OHLCV 5年覆盖600万条
2. **✅ P0（2026-06-19已实施）** — 换手率、市值过滤、指数环境过滤、资金流向/大单流向已全部从 Provider/Cache 接入到策略层。`SignalComputationService` 中获取 daily_basic/moneyflow/index + 传入 market_context；`ChanlunScorer.score()` 基于 turnover/moneyflow/index_condition 做 ±5~3 分调整
3. **✅ P1+P2（2026-06-19已实施）** — Provider层11项空白已补齐：财务指标表/利润表/资产负债表/现金流量表/前十大股东/股东人数/融资融券/业绩预告/行业分类/概念分类/指数成分股。DataManager 封装 + DataSourceManager `_route_provider` 路由已全部注册
4. **仍缺失** — 大宗交易、IPO日历、研报（Tushare 权限不明确）；L2 五档盘口、Tick 全推、全市场实时行情（依赖 QMT 激活）
5. **前端Mock问题** — 仪表盘~70%硬编码数据，43%后端端点前端未对接（此问题待前后端打通时处理）
6. **相较于180号"最优"标准** — 当前实现了约45-55%的数据端能力（基础管道P0完成+Provider层补齐），增强方向完全对应[181号方案](181-个股现状识别五维度增强方案.md)的五维度规划
