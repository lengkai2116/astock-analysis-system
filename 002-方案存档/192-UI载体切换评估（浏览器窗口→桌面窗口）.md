# UI 载体切换评估：浏览器窗口 → 桌面软件窗口

> 对"当前浏览器窗口能否切换为普通软件展示窗口类型"的重新评估。
> 前置依据：170号《Windows软件化迁移完整方案》(v2.0) — 无Docker裸跑 + Inno Setup + 嵌入式Python + VBScript静默启动。

---

## 一、背景与目标

### 1.1 用户需求

原本：

> "我们现在使用的是浏览器窗口，该UI载体后续是否可以切换成普通软件的展示窗口类型，这样浏览器上端的部分也能省出。评估需要同步考虑后续系统迁移到windows系统时的兼容性。"

核心诉求：
1. **省出浏览器上端空间** — 约 30~86px 垂直区域（取决于是否显示书签栏）
2. **更"软件化"的体验** — 桌面图标双击 → 直接看到系统界面，不先看到浏览器壳
3. **与 Windows 迁移兼容** — 符合 170 号方案的"无 Docker 裸跑 + Inno Setup 安装程序"路线

### 1.2 评估前提

| 前提 | 来源 | 说明 |
|------|------|------|
| 架构已确定为**无 Docker 裸跑** | 170号§1.2 | Flask 直接运行，SQLite + DuckDB |
| 部署目标为**Windows 10/11** | 170号§2.1 | 安装包 ~80MB，双击安装 |
| 前端为**预构建 SPA**（Flask send_from_directory 托管） | 170号§2.2 | 已实现，无需 Nginx |
| 启动方式为**VBScript 静默启动** | 170号§4.2 | 无黑框框，双击即达 |
| Python 分发方式为**嵌入式 Python 3.11** | 170号§1.2 | ~30MB，免安装免管理员 |
| 打包工具为**Inno Setup** | 170号§4.1 | `.exe` 安装程序 |

## 二、可行性方案对比

### 2.1 候选方案一览

| 方案 | 原理 | 额外大小 | 对170号的变更量 | 桌面化程度 |
|------|------|---------|----------------|-----------|
| **A. PWA Standalone** | 浏览器 manifest `display: standalone` | 0 MB | 极小（增1个 manifest.json） | 低 - 仍有浏览器底栏 |
| **B. PyWebView** | Python 原生 webview 窗口，指向 `localhost:5001` | ~0 MB（pip包） | 小（替换 VBScript 启动逻辑） | 高 - 无浏览器 Chrome |
| **C. Tauri** | Rust WebView2 wrapper + sidecar 进程 | ~5-10 MB | 大（新增 Rust 编译链） | 高 |
| **D. Electron** | Chromium + Node.js 包装 | ~150+ MB | 大（颠覆 80MB 目标） | 高 |

### 2.2 方案 A：PWA Standalone

**原理**：在 `frontend/vue-project/public/` 下添加 `manifest.json`，设置 `display: standalone`。浏览器识别为 PWA 后可"安装"到桌面，安装后打开无 URL 栏。

**与170号方案的兼容性**：
- ✅ 无任何构建链变更
- ✅ 与 Inno Setup 安装程序完全兼容
- ✅ 与嵌入式 Python + VBScript 完全兼容
- ✅ 不影响 80MB 安装包大小

**实际省出的空间**：
- 移除 URL 栏（~34px）+ 标签栏（~30px）→ 总计约 **64px**
- 书签栏（~22px）不由 PWA 控制，取决于用户
- 浏览器底栏（下载/扩展栏）仍在

**缺点**：
- 首次仍需通过浏览器访问 → 用户先看到浏览器壳，然后"安装"
- 浏览器底栏无法隐藏（Chrome 要求安全指示器可见）
- 各浏览器行为不一致（Edge/Chrome/360 等表现不同）
- 用户容易混淆"PWA 安装"和"浏览器收藏"

**结论**：可作为零成本过渡方案，但不能彻底解决"软件感"问题。

### 2.3 方案 B：PyWebView（推荐）

**原理**：`pywebview` 是一个轻量 Python 库，调用操作系统原生 webview（Windows 上为 WebView2 / Edge WebView，macOS 上为 WKWebView），创建一个无浏览器 chrome 的桌面窗口，指向 `http://localhost:5001`。

**与170号方案的兼容性分析**：

| 维度 | 评估 | 说明 |
|------|------|------|
| 安装方式 | ✅ 兼容 | `pip install pywebview`，可预装到嵌入式 Python 的 site-packages |
| 打包 | ✅ 兼容 | 嵌入到 embedded Python 后，Inno Setup 原封不动带走 |
| 启动方式 | ✅ 小改 | 替代 `start-backend.vbs` 的 `WshShell.Run "http://localhost:5001"`，改为启动 Python webview 窗口 |
| 前端代码 | ✅ 零改 | 前端 SPA 不变，WebView 模拟浏览器渲染 |
| 后端代码 | ✅ 零改 | Flask 服务不变，WebSocket/SSE 均通过 HTTP 端口正常通信 |
| macOS 兼容 | ✅ 可用 | pywebview 在 macOS 上使用 WkWebView，行为一致 |
| 安装包大小 | ✅ 不受影响 | pywebview 本体 ~500KB，无额外运行时 |
| VBScript 启动 | ⚠️ 需替换 | VBS 改为调用 `python.exe -m webview_app` 而非打开浏览器 |

**实际省出的空间**：
- 浏览器全部 chrome 消失 → **约 64~86px 垂直空间**
- 全屏模式下系统本身的菜单栏/任务栏仍在，但 Python webview 可设置 frameless 窗口进一步压缩

**缺点**：
- WebView2 在 Windows 7 上需要手动安装（Windows 10+ 已内置）
- 需处理窗口生命周期（关闭窗口→停止后端进程）
- WebSocket 重连逻辑仍需保留（后端重启场景）
- C 扩展在嵌入式 Python 中的兼容性需验证（pywebview 核心为 pure Python，但 Windows 上需 `pythonnet` 或 `clr` 包）

**实施工作量估算**（基于170号方案继续）：

| 工作项 | 工时 | 涉及文件 |
|--------|------|---------|
| 1. 创建 `webview_app.py` 入口文件 | 2h | `installer/webview_app.py`（新建） |
| 2. 修改 `start-backend.vbs` 调用 webview | 0.5h | `installer/start-backend.vbs` |
| 3. 修改 `stop-backend.vbs` 关闭窗口 + 后端 | 0.5h | `installer/stop-backend.vbs` |
| 4. CI 流水线安装 pywebview 依赖 | 0.5h | `.github/workflows/build-windows.yml` |
| 5. Windows 真实环境验证 | 2h | — |
| **合计** | **5.5h** | — |

**示例代码框架**：

```python
# installer/webview_app.py — 桌面窗口入口
import webview
import subprocess
import sys
import os

def start_backend():
    """启动 Flask 后端（隐藏控制台窗口）"""
    backend_dir = os.path.join(os.path.dirname(__file__), 'backend')
    python_exe = sys.executable  # 嵌入式 Python
    startupinfo = subprocess.STARTUPINFO()
    startupinfo.dwFlags |= subprocess.STARTF_USESHOWWINDOW
    return subprocess.Popen(
        [python_exe, 'run.py', '--port', '5001'],
        cwd=backend_dir,
        startupinfo=startupinfo,
        creationflags=subprocess.CREATE_NO_WINDOW
    )

if __name__ == '__main__':
    proc = start_backend()
    # 等待后端就绪后打开窗口（不带浏览器 chrome）
    webview.create_window(
        title='A股分析系统',
        url='http://localhost:5001',
        width=1400,
        height=900,
        resizable=True,
        min_size=(1024, 768)
    )
    webview.start()
    proc.terminate()  # 窗口关闭时停止后端
```

### 2.4 方案 C：Tauri

**原理**：使用 Rust 编写的小型桌面壳（~5MB），调用系统 WebView2 渲染前端界面。Flask 后端以 "sidecar" 进程方式启动。

**与170号方案的兼容性分析**：

| 维度 | 评估 | 说明 |
|------|------|------|
| 构建链 | ❌ **严重冲突** | Tauri 需要 Rust 工具链（`rustc`, `cargo`, 700MB+），与170号 "无额外工具链" 路线矛盾 |
| CI 流水线 | ❌ **大幅改造** | GitHub Actions 需安装 Rust、编译 Rust 项目，构建时间从 ~5min 增至 ~15-25min |
| 嵌入式 Python | ⚠️ 兼容但复杂 | Tauri sidecar 启动 Flask，进程管理逻辑需 Rust 编写 |
| 安装包大小 | ⚠️ 增加 5-10MB | 虽然远小于 Electron，但超出170号的纯 Python 打包策略 |
| 前端代码 | ✅ 零改 | 同 PyWebView |
| macOS 兼容 | ✅ 原生支持 | Tauri 在 macOS 上使用 WKWebView |

**结论**：技术上有可行性，但不匹配 170 号方案的"最小依赖、纯 Python"路线。引入 Rust 编译链与运维复杂化的收益不足以覆盖成本。

### 2.5 方案 D：Electron

**原理**：Chromium + Node.js 包裹前端，Flask 作为子进程。

**与170号方案的兼容性**：

| 维度 | 评估 |
|------|------|
| 安装包大小 | ❌ 从 ~80MB 膨胀至 ~250MB+ |
| 构建链 | ❌ 需安装 Electron 依赖（node_modules 100MB+） |
| 嵌入式 Python | ❌ 与170号矛盾（Electron 不需要 Python 运行时） |
| 启动体验 | ⚠️ 启动慢（加载 Chromium 约 2-5s） |

**结论**：否决。与 170 号方案的设计目标完全背道而驰。

## 三、垂直空间收益评估

### 3.1 浏览器 Chrome 垂直消耗

| 元素 | 高度 | 说明 |
|------|------|------|
| 标签栏 | ~30px | Chrome 标签页 |
| URL 栏 + 扩展栏 | ~34px | 地址栏 + 扩展图标 |
| 书签栏（可选） | ~22px | 用户可选开启 |
| 底栏（下载进度等） | ~24px | 可能临时出现 |
| **合计（无书签）** | ~64px | 浏览器打开即占 |
| **合计（含书签）** | ~86px | |

### 3.2 对 K 线布局的影响

已知的**K线 + 策略滚动区域的空间冲突**（2026-06-18 §009 锁定参数）：

```
浏览器窗口高度: 900px
  ├── 浏览器 Chrome: 64px  ← 切换桌面窗口可省出
  ├── 顶部栏: 48px
  ├── K线主图区: 340px
  ├── 副图区: 152px (筹码)
  ├── 策略结果区: 296px  ← 这里与K线争空间
  └── 总计: ~900px
```

切换桌面窗口后：
```
桌面窗口高度: 900px（全归应用）
  ├── 顶部栏: 48px
  ├── K线主图区: ~384px  (+44px, 从340→384)
  ├── 副图区: 152px
  ├── 策略结果区: ~316px  (+20px)
  └── 总计: 900px（全应用）
```

**判定：64px 垂直空间的释放有一定帮助，但不足以根本解决 K线+策略的布局冲突。**

- K线区 +44px 可多显示约 1.5 根 K 线柱
- 策略区 +20px 改善不大
- 核心方案（K线 sticky/foldable + 策略区独立滚动）仍是必须的

### 3.3 结论

> **桌面窗口切换能提供约 64~86px 的额外垂直空间，是"锦上添花"而非"雪中送炭"。** 它与 K线+策略的布局问题本质上是正交的：
> - 桌面化解决的是"浏览器壳占空间"的问题
> - K线+策略布局解决的是"图表内容与策略结果的交互关系"问题
>
> **两者可以并行推进，互不依赖。**

## 五、推荐路线

### 推荐方案：PyWebView（方案 B）

**理由**：
1. **与170号方案高度兼容** —纯 Python，pip 安装，嵌入嵌入式 Python，Inno Setup 原封不动
2. **不影响 80MB 安装包大小** — pywebview 无额外运行时
3. **实施成本低** — 仅增 1 个文件、改 2 个文件，约 5.5h
4. **macOS/Windows 跨平台** — 同一代码两平台可用
5. **垂直空间回收 64~86px** — 浏览器 chrome 全部消失

**不推荐 Tauri 的理由**：引入 Rust 编译链与 170 号的"最小依赖、纯 Python"路线矛盾，构建时间增长 3~5 倍，运维复杂度显著增加。

### 建议的实施时机

**立即可作为170号方案的 §8 追加实施**，安排在 P0 安装脚本与 CI 流水线之后：

| 阶段 | 工作项 | 工时 | 前置 |
|------|--------|------|------|
| P0-1 | 完成 170 号 §7 全部 P0 项（安装脚本+CI 流水线） | 6.5h | — |
| P0-2 | 创建 `webview_app.py` + 修改 VBS 启动脚本 | 2.5h | P0-1 |
| P1 | Windows 真实环境验证 + 边缘情况修复 | 2h | P0-2 |
| P2 | README 更新、Release Note 说明 | 0.5h | P1 |

**在 P0-1 未完成之前，桌面化无法启动** — 必须先有基础的安装程序和构建流水线。
