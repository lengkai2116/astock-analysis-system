### 015 — 监控通知页面沟通修订：触达体系+右侧交易逻辑+评估引擎

**沟通主题：** 页面改名、通知触达体系设计、右侧交易底层逻辑确认、规则引擎评估机制设计、系统数据能力评估、170号跨平台设计考量

**关键内容：**

**决策确认：**

| 决策项 | 结论 |
|:-------|:------|
| 页面名称 | ✅ 系统内中文名称改为**"监控通知"**，原名"预警管理"废弃 |
| 通知记录展示 | ✅ **方案A — 收件箱式**，作为监控通知页面的子Tab |
| 触达渠道 | ✅ 微信推送 + 桌面系统通知（macOS/Windows）为必要渠道 |
| 用户自配置原则 | ✅ 通知优先级/确认/渠道等全部由用户控制，系统不做强控 |
| 跨平台设计 | ✅ 留意170号方案（macOS和Windows各自独立运行），通知API需区分两平台 |
| 右侧交易底层逻辑 | ✅ 本系统为右侧交易分析工具，非高频量化，不设实时模式 |
| 评估模式 | ✅ **2种模式：盘中定时扫描（5/15/30/60分钟）+ 日终评估** |
| 数据能力 | ✅ 现有Tushare+AkShare可支撑分钟级扫描，无需QMT |
| 通知定位 | ✅ 系统为"盯盘代理"——只报告条件达成状态，不发出买卖建议 |

**右侧交易定位说明：**
- 本系统是A股分析辅助工具，非高频量化交易系统
- 右侧交易：等趋势确认后操作，不试图抄底逃顶
- 因此❌不需要秒级实时，✅分钟级到小时级完全够用
- 系统定位为"盯盘代理"：系统找到满足条件的股票状态，通知用户"请你去看看"，用户自主评估决策

**系统数据能力评估结论：**
| 数据需求 | 能力 | 说明 |
|:---------|:-----|:------|
| 全市场实时报价 | ✅ AkShare（新浪财经） | 单次调用获取全部A股实时价，无硬限频 |
| 分钟K线(5/15/30/60) | ✅ Tushare pro_bar() | 200次/分钟配额，远满足需求 |
| 日线数据 | ✅ PostgreSQL 610万行 | 本地查询，零网络开销 |
| 全市场扫描频次极限 | 每3秒(AkShare)+每分钟(Tushare) | 远超需要的5-60分钟频次 |

**通知内容定位（右侧交易理念）：**
- 不写"买入/卖出"，只写"条件达成"
- 展示各条件当前值及判定结果（哪些满足、哪些还差一点）
- 明确标注"请自行评估"——**这不是交易信号，是状态报告**
- 附带扫描时间和数据源
