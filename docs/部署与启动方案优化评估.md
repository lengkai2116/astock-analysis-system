# 部署与启动方案优化评估

评估日期：2026-05-25

## 结论

当前系统可以启动，但部署和启动方案还有明显优化空间。建议从单一 `docker-compose.yml` 改为“开发 compose + 生产 compose + 一键启动脚本/Makefile”的双模式方案，并把前端、后端、数据库、Redis、定时任务、初始化迁移统一纳入启动链路。

优先级最高的问题不是技术栈选择，而是端口、环境变量、初始化流程、前端服务方式和生产运行命令不统一。

## 当前现状

已有能力：

- `docker-compose.yml` 已包含 PostgreSQL、Redis、Backend。
- 后端已有 Dockerfile，依赖中已包含 `gunicorn`、`eventlet`。
- 后端支持 Flask-SocketIO。
- 前端 Vite 项目可独立 `npm run dev` 启动。
- 本地已有 DuckDB 缓存目录，适合通过 `/data` 挂载复用。
- 已有 `scheduler.py`，具备定时同步和缓存预热能力。

主要问题：

- Compose 没有纳入前端服务，用户仍需手动启动前端。
- Compose 没有纳入定时任务服务，数据同步和缓存预热无法随系统运行。
- 后端容器使用 `flask run`，适合开发，不适合生产。
- 端口文档不统一：README 写 API 5000，Compose 对外映射 5001，部分测试文档又建议本地 5000。
- 前端 API 默认值存在不一致：Vite 注入默认 `localhost:5001`，request fallback 仍有 `localhost:8080`，Socket.IO 默认硬编码 `localhost:5001`。
- 数据库初始化只挂载 `init.sql`，后续迁移和系统模板初始化没有统一入口。
- 默认 Compose 暴露 PostgreSQL 和 Redis 到宿主机，生产环境不应默认公开。
- `docker compose config` 会展开 `.env` 中的敏感变量，应避免在日志、文档和排错输出中直接展示完整配置。

## 推荐部署模式

### 1. 开发模式

目标：本机开发体验优先，支持热更新、前后端独立调试。

建议服务：

- `postgres`
- `redis`
- `backend-dev`
- `frontend-dev`

建议入口：

```bash
make dev
```

等价命令：

```bash
docker compose -f docker-compose.dev.yml up --build
```

建议端口：

- 前端：`http://localhost:9000`
- 后端 API：`http://localhost:5001`
- PostgreSQL：仅开发模式可映射 `5432`
- Redis：仅开发模式可映射 `6379`

后端开发命令建议：

```bash
python run.py --port 5000
```

容器映射继续保持 `5001:5000`，前端统一访问 `http://localhost:5001`。

### 2. 生产/本机稳定运行模式

目标：一条命令启动完整系统，用户只访问一个 Web 地址。

建议服务：

- `postgres`
- `redis`
- `backend`
- `frontend-nginx`
- `scheduler`

建议入口：

```bash
make up
```

等价命令：

```bash
docker compose -f docker-compose.prod.yml up -d --build
```

建议端口：

- Web 入口：`http://localhost:8080`
- 后端 API 不直接暴露，由 Nginx 反代 `/api` 和 `/socket.io`
- PostgreSQL、Redis 不暴露到宿主机，除非显式开启调试 profile

后端生产命令建议：

```bash
gunicorn --worker-class eventlet -w 1 --bind 0.0.0.0:5000 run:app
```

说明：当前系统使用 Flask-SocketIO，生产模式不能简单使用多 worker Gunicorn，除非同时配置 Socket.IO message queue。第一版用 eventlet 单 worker 更稳。

前端生产建议：

- 用 Node 镜像构建 `dist`
- 用 Nginx 静态服务前端
- Nginx 反向代理：
  - `/api/` -> `backend:5000`
  - `/socket.io/` -> `backend:5000`

### 3. 数据任务模式

定时任务应作为独立容器运行，不应和 Web 后端混在同一进程里。

建议服务命令：

```bash
python /app/scheduler.py --scheduler
```

建议补充一次性命令：

```bash
make sync
make warmup
make full-sync
```

## 推荐文件结构

建议新增或调整：

```text
docker-compose.dev.yml
docker-compose.prod.yml
.env.example
Makefile
backend/docker/entrypoint.sh
frontend/vue-project/Dockerfile
frontend/vue-project/nginx.conf
docs/启动部署指南.md
```

## 初始化流程建议

启动前应统一执行：

1. 等待 PostgreSQL 和 Redis 健康。
2. 执行数据库迁移或 SQL 初始化。
3. 执行系统模板初始化。
4. 检查 DuckDB 数据目录权限。
5. 启动后端服务。

建议把这些封装进 `backend/docker/entrypoint.sh`，避免用户手动执行多个 Python 命令。

## 端口统一建议

建议固定如下：

| 场景 | 前端 | 后端宿主端口 | 后端容器端口 | Web 统一入口 |
| --- | --- | --- | --- | --- |
| 开发 | 9000 | 5001 | 5000 | 无 |
| 生产 | Nginx 容器 | 不公开 | 5000 | 8080 |

README、测试文档、前端环境变量、Socket.IO 默认值都应按这套端口更新。

## 前端配置建议

开发模式：

```env
VITE_API_BASE_URL=http://localhost:5001
VITE_WS_URL=http://localhost:5001
```

生产模式：

```env
VITE_API_BASE_URL=/api
VITE_WS_URL=
```

同时应把 `frontend/vue-project/src/utils/request.js` 的 fallback 从 `localhost:8080` 改为统一配置来源，把 `socketService.js` 的默认地址改为环境变量或当前 origin。

## 安全建议

- 提供 `.env.example`，真实 `.env` 不进入版本管理。
- 不在排错输出、文档和日志中展示完整 `docker compose config`。
- 生产模式不暴露 PostgreSQL、Redis 端口。
- 后端生产模式不要使用 `FLASK_ENV=development`。
- 补充 `SECRET_KEY`，不要使用默认 `dev`。
- 如后续部署到服务器，Nginx 层负责 TLS、压缩、静态缓存和访问日志。

## 改造优先级

P0：

- 统一端口和文档。
- 新增 `.env.example`。
- 前端 API/WebSocket 配置去硬编码。
- 生产后端改用 Gunicorn eventlet。
- Compose 增加前端服务。

P1：

- 拆分 `docker-compose.dev.yml` 和 `docker-compose.prod.yml`。
- 增加 Makefile 一键命令。
- 增加 backend entrypoint，统一迁移和模板初始化。
- 增加 scheduler 独立服务。

P2：

- Nginx 单入口访问。
- 生产模式隐藏数据库和 Redis 端口。
- 增加日志轮转、健康检查和备份任务。
- 增加 CI 构建验证。

## 总体建议

最佳方案是保留 Docker Compose，但从“后端 + 基础设施”升级为“完整应用编排”。开发时使用 Vite + Flask 热更新，生产或日常稳定使用时只运行 Nginx + Gunicorn + Postgres + Redis + Scheduler。

这样可以把启动方式收敛为两条命令：

```bash
make dev
make up
```

对当前系统来说，这是投入较小、收益最高的部署优化路径。
