## 一、系统架构

```
┌──────────────┐     ┌──────────────┐     ┌──────────────┐
│   Nginx      │────▶│   Gunicorn   │────▶│  PostgreSQL  │
│   (前端静态)  │     │  (Flask API) │     │  (主数据库)   │
│   local:9000 │     │  local:5001  │     │  local:5432  │
└──────────────┘     └──────┬───────┘     └──────────────┘
                            │                     │
                            ▼                     ▼
                     ┌──────────────┐     ┌──────────────┐
                     │   Redis 7    │     │   DuckDB     │
                     │  (缓存/会话) │     │  (OLAP缓存)   │
                     └──────────────┘     └──────────────┘
```

## 二、启动与停止

```bash
# 启动
./start.sh

# 停止 (优雅关闭, 等待请求完成)
./stop.sh

# 查看日志
docker compose logs -f          # 全部
docker compose logs -f backend  # 仅后端
docker compose logs -f frontend # 仅前端
```

## 三、健康检查

| 端点 | 用途 | 预期响应 |
|------|------|---------|
| `/api/v3/health` | 综合健康检查 | `{"success":true,"status":"healthy"}` |
| `/api/v3/health/live` | 进程存活 (liveness) | `{"success":true,"status":"alive"}` |
| `/api/v3/health/ready` | 依赖就绪 (readiness) | `{"success":true,"status":"ready","dependencies":{...}}` |

## 四、数据备份

```bash
# 手动备份到默认目录
./scripts/backup.sh

# 手动备份到指定目录
./scripts/backup.sh /path/to/backup

# 定时备份 (添加 crontab)
crontab scripts/crontab.txt
```

备份内容:
- PostgreSQL: `pg_dump` 完整导出
- DuckDB: 缓存文件快照
- .env: 配置文件副本

## 五、常见故障排查

### 5.1 前端页面白屏

```bash
# 检查前端容器
docker compose logs frontend | tail -20
# 检查 Nginx 是否正常
curl -sI http://localhost:9000 | head -5
# 检查 API 代理
curl -s http://localhost:9000/api/v3/health
```

### 5.2 后端 API 超时

```bash
# 检查后端日志
docker compose logs backend | grep ERROR | tail -20
# 检查数据库连接
curl -s http://localhost:5001/api/v3/health/database
# 检查 Redis
docker compose exec redis redis-cli ping
```

### 5.3 数据不更新

```bash
# 检查数据源状态
curl -s http://localhost:5001/api/v3/data-source/status
# 强制同步
curl -s -X POST http://localhost:5001/api/v3/data/sync
# 检查缓存状态
curl -s http://localhost:5001/api/v3/cache/stats
```

### 5.4 数据库问题

```bash
# 检查 PostgreSQL
docker compose exec postgres pg_isready -U stock_user -d stock_analysis
# 检查表结构
docker compose exec postgres psql -U stock_user -d stock_analysis -c "\dt"
# 修复迁移
docker compose exec backend flask db upgrade
```

## 八、日志管理

```
日志位置 (Docker 内):
  /app/logs/                     # 后端日志
  /var/log/nginx/                # Nginx 日志 (Docker 外映射)
  
日志轮转:
  后端: 每日轮转, 保留 30 天
  Nginx: Docker 日志驱动自动管理

关键关键词:
  ERROR     - 系统错误, 需关注
  WARNING   - 资源不足或降级
  TRIGGERED - 策略触发事件
  SWITCH    - 数据源切换
```

## 附录一：原型UI修改执行规则

> **适用范围：** 所有涉及 `_ui-prototype/` 目录下 HTML 文件的修改、评估和沟通。
> **不适用范围：** 后端代码、数据库、策略引擎、API 等非前端原型的工作项。

### A.1 修改前检查清单

开始任何原型 UI 修改前，必须依次完成以下检查：

| 步骤 | 操作 | 验证标准 |
|:----:|:-----|:---------|
| ① | 读取 `_项目运行手册.md` 全文 | 确认手册内容已加载到上下文 |
| ② | 读取 `_工作上下文.md` | 了解项目当前状态和待办 |
| ③ | 搜索 `002-方案存档/` 中最近相关评估文档 | 是否存在已有评估方案或冲突记录 |
| ④ | 搜索 `001-沟通记录/` 中最近会话的对应主题 | 确认用户之前提出的需求和决策 |
| ⑤ | 浏览器打开目标原型 HTML 文件 | 确认当前实际展示状态 |

### A.2 修改流程

```
用户提出修改需求
    ↓
① 理解需求本质 —— 问清楚"要什么效果"，不假设
    ↓
② 浏览器实测 —— 确认当前实际状态，记录测量数据（像素、位置、颜色等）
    ↓
③ 根因分析 —— 不直接打补丁，找到问题的根本原因
    ↓
④ 评估方案（如有必要） —— 复杂修改先出评估方案存档到 002-方案存档/
    ↓ 用户确认后
⑤ 执行修改 —— 手术刀式修改，只改必要的代码，不重构无关部分
    ↓
⑥ 浏览器验证 —— 逐项验证，输出实测数据
    ↓
⑦ 同步更新所有相关文档
```

### A.3 浏览器验证标准

每次修改后必须进行浏览器验证，并记录以下数据：

| 维度 | 要求 |
|:-----|:------|
| 视觉对齐 | 精确到像素级的坐标/位置测量（使用 `getBoundingClientRect()` 或 Elements panel）|
| 异常值覆盖 | 边界情况测试（最大值/最小值/0值/空值）|
| 交互覆盖 | 所有触发路径测试（鼠标/键盘/弹窗/切换）|
| 回归检查 | 验证修改未破坏现有功能 |

### A.4 文档同步要求

每次完成原型 UI 修改后，必须同步更新以下文档：

| 文档 | 更新内容 |
|:-----|:---------|
| `001-沟通记录/` 当日纪要 | 按 §XXX 编号记录，含关键结论、决策、验证数据 |
| `001-沟通记录/沟通索引.md` | 索引当天条目 |
| `002-方案存档/`（如适用） | 评估方案更新状态→"已实施" |
| `_工作上下文.md` | 记录完成状态、涉及文件及行号 |
| `002-方案存档/166-前端UI原型逐页调整记录.md` | 更新对应页面 § 节的调整记录 |
| `002-方案存档/171-功能视图API映射对照表.md` | 如有 API 关联变更，更新对应 § |

### A.5 禁止事项

| 禁止行为 | 原因 |
|:---------|:------|
| 不经浏览器实测直接改代码 | 无法确认真实现状，可能修复了不存在的问题 |
| 不更新 166 号文档对应 § 节 | 166 号方案是本项目唯一的前端调整记录总账 |
| 不更新 171 号文档 API 映射 | 171 号是后端 API 与前端视图的映射基准，不同步会导致后续开发方向错误 |
| 评估后不修改，但也不标注状态 | 导致无法追溯"已评估/已确认/已拒绝"的状态 |
| 修改后不验证就告知完成 | 验证不是可选项，是必经步骤 |

### A.6 坐标系校验规则（筹码面板相关）

筹码面板涉及 **两套独立的坐标系统**，修改时必须特别注意：

| 坐标系 | 来源 | 用途 |
|:-------|:-----|:------|
| **ECharts grid Y 轴** | `chart.convertToPixel()` | 十字光标横线、价格标签定位 |
| **chip-chart 自身坐标** | `(1-(price-lo)/range)*containerH` | 筹码条 Y 位置 |

**原则：** 凡是需要与十字光标对齐的视觉元素（白色线、红蓝分界），必须使用 DOM `getBoundingClientRect()` 进行坐标换算，**不得使用 chip-chart 自身的比例映射公式**。
